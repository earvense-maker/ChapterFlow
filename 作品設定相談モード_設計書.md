# 作品設定相談モード 設計書

## 0. 設計レビュー反映メモ

この設計は、自由な相談体験から既存の連載生成ループへ接続する方針として妥当である。特に、フォーム入力ではなく「候補を出す、混ぜる、直す」流れを主役にする判断は、このアプリの読者向け体験と相性がよい。

ただし、実装へ進む前に以下を明確にしておく。

- 相談セッションは既存プロジェクトではないため、`data/setup-sessions/{sessionId}.json` に独立保存する。
- 相談中の型は `src/shared/types.ts` に `SetupSession` / `SetupDraft` / `SetupDraftPatch` として追加し、クライアントとサーバーで同じ構造を使う。
- 最終変換は既存の `projectService.createProject` だけでは足りない。現在の作成入力は `worldText`、`characters`、`customSystemPrompt` までは受け取れるが、`memories` と `storyState` は直接受け取れないため、`setupCommitService` がプロジェクト作成後に `storage.writeMemories` と `storage.writeStoryState` を行うか、`CreateProjectBody` を拡張する。
- `story-state.json` は既存の `StoryState` 型に合わせる。`currentSituation`、`characterStates`、`importantEvents`、`openThreads`、`updatedAt` を持たせ、不足IDはサーバー側で正規化する。
- LLMの `draftPatch` は提案であり、保存操作ではない。サーバー側で検証、ロック保護、重複排除、上限適用をしてからセッションへ反映する。
- 相談LLM、試し書きLLM、最終変換LLMはいずれも既存のモデル設定と認証読み込みを使う。APIキー未設定、タイムアウト、JSON破損時のフォールバックをAPI設計に含める。
- 初期実装では高度なカードUIよりも、セッション保存、チャット送信、draft更新、プロジェクト化、最小テストを優先する。

## 1. 目的

作品設定相談モードは、ユーザーが通常のチャットで「読みたい物語の種」を話しながら、最終的に連載小説プロジェクトの初期設定へ変換する機能である。

このモードでは、ユーザーに世界設定、人物設定、関係性、開始場面などをフォームとして埋めさせない。LLMが編集者のように会話し、候補を広げ、整理し、ユーザーの選択や修正を受けながら、裏側で作品設計メモを育てる。

最重要方針は、設定作成そのものを目的化せず、「この物語を読みたい」と思える温度まで自然に到達することである。

## 2. 基本方針

### 2.1 体験の定義

表の体験は、自由な相談チャットである。

裏側では、会話ログから以下の情報を更新する。

- 決まってきたこと
- 候補として残していること
- 未確定のこと
- 登場人物の案
- 人物同士の関係性
- 世界観や時代感
- 物語の火種
- 好み、NG、文体の傾向
- 冒頭候補

ユーザーは「設定を書く人」ではなく、「好みに近い方向を選び、違うところを直す人」として扱う。

### 2.2 採用しない方向性

- 固定ステップのウィザードにする
- 大量の必須入力欄を並べる
- LLMに作品データを完全自動決定させる
- 設定をすべてシステムプロンプトに詰め込む
- ユーザーの人格分析や心理推測をこの機能に含める
- 未確定の可能性を、確定済みの物語事実として保存する

### 2.3 システムプロンプトとの関係

UI上は「最終的に生成用プロンプトへ落とし込む」と表現してよい。

ただし内部的には、すべてを `customSystemPrompt` に入れない。保存先は既存構造に分ける。

- `world.md`: 世界観、時代感、作品の核
- `characters.json`: 人物の基本情報、口調、関係性メモ
- `story-state.json`: 開始時点の状況、未解決事項、重要イベント
- `memories.json`: 高重要度の好み、NG、守るべき物語事実
- `presets/customSystemPrompt`: 書き方、役割、出力姿勢などシステム寄りの指示だけ

## 3. ユーザー体験

### 3.1 入口

作品作成画面に、以下の入口を追加する。

- すぐ作る
- 相談して作る

「相談して作る」を選ぶと、作品設定相談モードに入る。

既存プロジェクトにも将来的には「設定を相談して補強する」入口を用意できるが、初期実装では新規プロジェクト作成に絞る。

現行の `ProjectForm` は送信するとすぐ `api.createProject()` を呼ぶ単発作成画面である。相談モードは未commitのセッションを扱うため、`ProjectForm` に大きく混ぜず、`App.tsx` に `setup` viewを追加して `SetupWorkspace` のような専用画面へ遷移させる。

入口の実装案:

- `ProjectList` の新規作成導線から `ProjectForm` を開く
- `ProjectForm` の先頭または新規作成選択画面に「すぐ作る」「相談して作る」を置く
- 「すぐ作る」は現行フォームのまま
- 「相談して作る」は `SetupWorkspace` に遷移し、フォーム既定値を `projectSettings` としてセッション作成に渡す
- APIキー、モデル、プリセットの詳細編集は初期実装では既存の設定導線を流用し、相談画面内では最小表示にする

### 3.2 画面構成

左側はチャット、右側は作品の種メモ。

右側は固定フォームではなく、会話から育っているメモとして見せる。

```text
決まってきたこと
- 強気なヒロイン
- 弱気な主人公
- 江戸時代風
- 暗すぎず、少し笑える

候補
- 女岡っ引き × 気弱な絵師
- 武家の娘 × 商家の跡取り

未確定
- 史実寄りか、時代劇風か
- 恋愛の進行速度
- 最初の事件

人物
- 主人公: 気弱だが観察眼が鋭い若い絵師
- ヒロイン: 町を仕切る強気な女岡っ引き
```

各項目には以下の操作を用意する。

- 固定
- 編集
- 削除
- 未確定へ移す
- 確定へ移す

ユーザーが手で編集または固定した項目は `locked` として扱い、以後のLLM更新では勝手に上書きしない。

### 3.3 チャットのふるまい

LLMは質問係ではなく、相談相手として振る舞う。

悪い例:

```text
主人公の名前は何ですか？
ヒロインの年齢は何歳ですか？
舞台はどこですか？
```

良い例:

```text
それなら、江戸時代風の町を舞台にすると相性がよさそうです。
強気なヒロインは、町を仕切る女岡っ引きにすると動きが出ます。
主人公は気弱な絵師にすると、弱いけれど観察眼だけは鋭い人物にできます。

方向性は3つあります。

A. 女岡っ引き × 気弱な絵師
B. 武家の娘 × 商家の跡取り
C. 旅芸人の姉御 × 寺子屋の青年

近いものはありますか？ 混ぜても大丈夫です。
```

### 3.4 候補操作

候補が複数ある場合は、カードとして表示できる。

候補カードの操作:

- これで進める
- 混ぜる
- もう一度
- 手で直す
- 一部だけ固定

初期実装ではカードUIを必須にしない。チャット本文内の候補を選ぶだけでもよい。

### 3.5 試し書き

ある程度方向が固まったら、LLMは短い冒頭サンプルを提案できる。

目的は本文生成ではなく、作品の温度確認である。

- 300から600字程度
- プロジェクト本文としてはまだ保存しない
- ユーザーは「もっと軽く」「ヒロインを強めに」「主人公を情けなく」など調整できる

試し書きは、最終変換時に `openingSeeds` や `tone` の根拠として使う。

## 4. 会話の進行モデル

表向きのステップは固定しない。

ただし、LLMの内部方針として以下の流れを持たせる。

### 4.1 種を受け取る

ユーザーが短く入力する。

```text
強気なヒロインと弱気な主人公。世界は江戸時代。
```

LLMはこれを決定済みの最低限の種として扱う。

```text
confirmed:
- 強気なヒロイン
- 弱気な主人公
- 江戸時代風
```

### 4.2 複数案に広げる

LLMは、ユーザーの種を数パターンに展開する。

例:

- 女岡っ引き × 気弱な絵師
- 武家の娘 × 商家の跡取り
- 旅芸人の姉御 × 寺子屋の青年

この段階では、案は確定ではなく候補である。

### 4.3 選択、混合、修正

ユーザーは以下のように返せる。

```text
Aが近い。でも主人公はもっと情けない感じがいい。
```

LLMは以下を行う。

- A案を中心候補にする
- 主人公の「より情けない」を確定寄りにする
- ただし名前や事件内容などはまだ未確定に残す

### 4.4 人物と関係性を濃くする

登場人物はこのモードの中心である。

人物単体のプロフィールより、以下を優先する。

- 何に揺れる人か
- 相手に何を見出すか
- 何を隠しているか
- 普段の顔と相手にだけ見せる顔
- 関係が進む時のブレーキ
- 物語を動かす役割

### 4.5 火種を作る

連載を始めるための火種を作る。

火種の例:

- 事件
- 秘密
- 約束
- 再会
- 依頼
- 違和感
- 誤解
- まだ説明されない過去

### 4.6 確認して変換する

最後に、LLMは現在の作品案を短く整理する。

```text
いまの案はこうです。

作品の核:
気弱だが観察眼の鋭い絵師が、強気な女岡っ引きに振り回されながら、江戸の小さな事件を解いていく。
軽妙な掛け合いの奥に、互いの弱さを見つけていく関係を置く。

この内容で本編を始めますか？
```

ユーザーが承認したら、プロジェクトデータへ変換する。

## 5. 内部データ設計

### 5.1 setup-session.json

相談中は、本番プロジェクトへ直接保存しない。

一時データとして `setup-session.json` 相当のセッションを持つ。

保存場所案:

```text
data/setup-sessions/{sessionId}.json
```

既存プロジェクトを補強する場合は、将来的に `projectId` を持たせる。

保存実装では、既存の `storageService` と同じ方針に合わせる。

- `SETUP_SESSIONS_DIR = path.resolve(DATA_DIR, 'setup-sessions')` を追加する
- `sessionId` は `generateTimestampId('setup')` を使い、`SAFE_PATH_SEGMENT` 相当の検証を通す
- 読み書きは `safeWriteJson` / `readJsonFile` を使う
- セッション一覧は初期実装では不要。未完了セッション復帰を作る場合に追加する
- `projectId` が入っていても、commit前は既存プロジェクトのファイルを直接変更しない

### 5.2 セッション構造

{

    {
      "messageId": "msg-1",

`status` は `active`、`committed`、`abandoned` を想定する。`revision` は右側メモの手動編集とLLM更新が衝突したときの検出に使う。初期実装では楽観ロックとして扱い、クライアントが古い `revision` で更新した場合は409を返す。

`projectSettings` は最終的に `CreateProjectBody` へ渡す設定である。新規作成画面から入る場合は、既存のフォーム既定値をここへコピーする。初期実装では相談画面内で全項目を編集できなくてもよいが、commit時にはこの値を使う。

### 5.3 draft構造

`draft` は最終保存データではなく、相談中の作業メモである。各項目はできるだけIDを持たせ、LLMが同じ内容を少し違う言葉で何度も追加しても、アプリ側で統合できるようにする。

```json
{
  "coreConcept": "気弱な絵師が、強気な女岡っ引きに振り回されながら江戸の事件に巻き込まれる。",
  "confirmed": [
    {
      "id": "fact-1",
      "text": "強気なヒロイン",
      "source": "user",
      "status": "active",
      "locked": true,
      "createdAt": "2026-07-04T12:00:00.000Z",
      "updatedAt": "2026-07-04T12:00:00.000Z"
    }
  ],
  "candidates": [
    {
      "id": "cand-1",
      "title": "女岡っ引き × 気弱な絵師",
      "summary": "町の揉め事に首を突っ込むヒロインと、事件現場を描く内気な絵師。",
      "status": "active",
      "source": "llm"
    }
  ],
  "undecided": [
    {
      "id": "und-1",
      "text": "史実寄りにするか、時代劇風にするか",
      "reason": "江戸時代という希望だけでは硬さが未確定"
    }
  ],
  "characters": [
    {
      "id": "char-draft-1",
      "role": "protagonist",
      "name": "",
      "label": "気弱な絵師",
      "description": "争いごとが苦手だが、人の表情や場の違和感をよく見る。",
      "speechStyle": "遠慮がちで言い切らない",
      "relationshipNotes": "ヒロインに振り回されるが、彼女のまっすぐさに救われる。",
      "lockedFields": ["role", "label"]
    }
  ],
  "relationshipSeeds": [
    "ヒロインは主人公の観察眼を買っている",
    "主人公は怖がりながらも、彼女の強さに引っ張られる"
  ],
  "world": [
    "江戸時代風の町",
    "史実厳密より時代劇風"
  ],
  "tone": [
    "暗すぎず、少し笑える",
    "掛け合い多め"
  ],
  "ng": [],
  "openingSeeds": [
    "主人公が事件現場で絵を描いているところに、ヒロインが強引に現れる"
  ]
}
```

初期実装で必須とする配列上限:

- `confirmed`: 30件
- `candidates`: active 6件、archivedを含めて20件
- `undecided`: 20件
- `characters`: 12件
- `relationshipSeeds`: 30件
- `world`: 20件
- `tone`: 12件
- `ng`: 20件
- `openingSeeds`: 10件

上限を超えた場合は、古い候補を `archived` にするか、LLMの新規追加を捨てる。削除は履歴追跡が必要になるため、初期実装では `status: "archived"` を優先する。

`characters` は既存の `Character` へ直接保存する前段階なので、名前が未定でもよい。ただし commit時には `characterId`、`role`、`name`、`description` を既存型に合わせて補正する。名前が未定の場合は空文字を許容する。

### 5.4 locks

`locks` は、LLMが上書きしてはいけない項目を示す。

```json
[
  {
    "path": "draft.confirmed.fact-1.text",
    "reason": "user_locked",
    "createdAt": "2026-07-04T12:00:00.000Z"
  }
]
```

初期実装では、項目単位の `locked: true` だけでもよい。

ロックの適用優先度:

1. `locked: true` の項目全体は、LLM patchによる更新、削除、archived化を拒否する。
2. `lockedFields` がある人物項目は、そのフィールドだけ更新を拒否する。
3. `locks.path` は Phase 2 以降の細かい制御に使う。初期実装では保存してもよいが、UIから細かく編集できなくてもよい。

ユーザーの手動編集は常にLLMより強い。LLMがロック済み項目に矛盾する提案をした場合は、既存項目は守り、新しい候補または未確定メモとして扱う。

## 6. LLM入出力設計

### 6.1 会話更新

チャット送信ごとに、LLMには以下を渡す。

- 直近の会話ログ
- 現在の `draft`
- `locked` 項目
- アプリの相談方針
- ユーザーの今回入力
- 現在の `revision`
- 既存の配列上限と、確定/候補/未確定の分類ルール

LLMは、表示用返答と `draftPatch` を返す。

```json
{
  "visibleReply": "それなら、女岡っ引きと気弱な絵師の組み合わせがかなり合いそうです...",
  "draftPatch": {
    "confirmedAdd": [
      {
        "text": "江戸時代風",
        "source": "user"
      }
    ],
    "candidatesAdd": [
      {
        "title": "女岡っ引き × 気弱な絵師",
        "summary": "町の揉め事に首を突っ込む強気なヒロインと、事件現場を描く内気な絵師。"
      }
    ],
    "undecidedAdd": [
      {
        "text": "史実寄りか、時代劇風か"
      }
    ]
  },
  "suggestedActions": [
    {
      "label": "Aで進める",
      "message": "Aの女岡っ引きと気弱な絵師で進めたい"
    },
    {
      "label": "もう一度",
      "message": "別の人物案をもう一度出して"
    }
  ]
}
```

実装では `setupPromptBuilder` が上記入力を組み立て、`setupSessionService` または `setupChatService` が既存のモデルアダプターを呼ぶ。モデル呼び出し前には既存の生成処理と同様に `reloadCredentials()` を行い、`activeModelProvider` / `activeModelName` はセッションの `model` から取る。

初期実装ではストリーミングは必須にしない。返答が完了してから `assistantMessage` と更新済み `draft` を返す。ただし、タイムアウトは既存生成と同じく120秒程度に揃え、ユーザーには再試行できるエラーとして返す。

### 6.2 draftPatchの適用方針

アプリ側はLLMの `draftPatch` をそのまま信用しすぎない。

最低限、以下を行う。

- JSONとして読めない場合は返答だけ表示し、draft更新は捨てる
- locked項目への変更は無視する
- 空文字の項目は追加しない
- 似た項目がある場合は重複を避ける
- 追加、更新、削除をログに残す
- 配列上限を超える追加は捨てるか、古い候補をarchivedにする
- `revision` が古い更新は409として拒否し、最新セッションを返す
- `source` は `user` / `llm` / `manual` のいずれかに正規化する
- LLMが `confirmed` へ入れた内容でも、ユーザー発言から直接取れないものは `candidate` または `undecided` に落とす

`draftPatch` は、最初から完全なJSON Patch互換にしない。初期実装では以下のようなドメイン専用patchでよい。

```json
{
  "confirmedAdd": [],
  "candidatesAdd": [],
  "undecidedAdd": [],
  "charactersAdd": [],
  "charactersUpdate": [],
  "relationshipSeedsAdd": [],
  "worldAdd": [],
  "toneAdd": [],
  "ngAdd": [],
  "openingSeedsAdd": [],
  "archiveIds": []
}
```

手動編集APIだけは、クライアントから送られた編集済み `draft` をサーバー側で検証して差し替える。LLM patch用の緩い補正と、手動編集用の明示更新は分ける。

### 6.3 返答失敗時

LLMがJSON形式を崩した場合でも、チャット体験を止めない。

- `visibleReply` が取れる場合は表示する
- `draftPatch` が壊れている場合は「メモ更新なし」で続行する
- 失敗が続く場合は、通常の文章返答モードへフォールバックする
- APIキー未設定の場合は、相談を開始せず設定画面への導線を出す
- モデル応答がタイムアウトした場合は、ユーザー入力メッセージを保存済みにするかを明確にする。初期実装では、ユーザー入力は保存し、assistantMessageは失敗として保存しない
- LLM出力にプロンプトや内部JSONを表示するような混入があった場合、表示用返答はサニタイズし、draft更新は捨てる

- `project.json`
- `presets.json`
- `world.md`
- `characters.json`
- `memories.json`
- `story-state.json`

相談セッションは `status: committed` にする。

現行の `CreateProjectBody` で直接受け取れるのは、主に以下である。

- `title`
- `outputLength`
- `streamingEnabled`
- `activeModelProvider`
- `activeModelName`
- `activePresetIds`
- `worldText`
- `characters`
- `customSystemPrompt`

そのため、commit処理は以下の順に行う。

1. 最終変換LLMのJSONを検証、正規化する。
2. `projectService.createProject` に上記の作成入力を渡す。
3. 返ってきた `projectId` に対して `storage.writeMemories(projectId, memories)` を行う。
4. `storyStateService` の正規化方針に合わせた `StoryState` を `storage.writeStoryState(projectId, storyState)` する。
5. セッションへ `status: "committed"` と `committedProjectId` を保存する。
6. 途中で失敗した場合は、セッションを `active` のまま残し、作成済みプロジェクトが中途半端に残らないように補償処理またはエラー表示を行う。

- タイトル
- 作品の核
- 主人公
- ヒロイン/相手役
- 世界観
- 開始場面
- 好み/NG
- 未確定として残す要素

ユーザーは、保存前にこの画面でも編集できる。

### 8.1 セッション作成

```text
POST /api/setup-sessions
```

request:

```json
{
  "initialMessage": "強気なヒロインと弱気な主人公。世界は江戸時代。",
  "projectSettings": {
    "outputLength": 3000,
    "streamingEnabled": false,
    "activePresetIds": {
      "genre": "mystery",
      "style": "natural-dialogue",

  },
  "model": {
    "provider": "gemini",
    "modelName": "gemini-3.5-flash"
  }
}
```

response:

```json
{
  "sessionId": "setup-20260704-120000-abcd",
  "session": {},
  "assistantMessage": {},
  "suggestedActions": []
}
```

初期実装では、作成時に最初のLLM応答まで返す。LLM呼び出しに失敗した場合でも、セッション作成自体が成功していれば `session` を返し、`lastError` に失敗理由を残す。

### 8.2 セッション取得

```text
GET /api/setup-sessions/:id
```

### 8.3 メッセージ送信

```text

```

request:

```json
{
  "message": "Aが近いけど、主人公はもっと情けない感じがいい。",
  "revision": 1
}
```

response:

```json
{
  "assistantMessage": {},
  "draft": {},
  "suggestedActions": [],
  "revision": 2
}
```

### 8.4 draft手動更新

```text
PUT /api/setup-sessions/:id/draft
```

ユーザーが右側メモを直接編集した場合に使う。

request:

```json
{
  "draft": {},
  "revision": 2
}
```

response:

```json
{
  "draft": {},
  "revision": 3
}
```

### 8.5 項目固定/解除

```text
POST /api/setup-sessions/:id/locks
DELETE /api/setup-sessions/:id/locks/:lockId
```

### 8.6 試し書き

```text
POST /api/setup-sessions/:id/preview
```

相談中のdraftを使って短い冒頭サンプルを生成する。

### 8.7 プロジェクト化

```text
POST /api/setup-sessions/:id/commit
```

response:

```json
{
  "projectId": "proj-20260704-121500-abcd",
  "session": {}
}
```

### 8.8 共通エラー

代表的なエラー:

- 400: request bodyが不正
- 401相当: APIキー未設定または無効。既存のモデル設定導線へ誘導する
- 404: セッションが存在しない
- 409: `revision` が古い。最新セッションを返し、クライアント側で再表示する
- 503: モデル呼び出し失敗、タイムアウト、レート制限

エラーresponse:

```json
{
  "error": "生成がタイムアウトしました。少し待って再試行してください。",
  "code": "timeout",
  "retryable": true,
  "session": {}
}
```

モデル呼び出しが失敗しても、ユーザーの入力とセッションを失わないことを優先する。

## 9. プロンプト設計

### 9.1 相談LLMのシステム指示

要点:

- あなたは小説設定の相談相手である
- ユーザーは執筆者というより読者である
- 質問攻めにせず、候補を出しながら一緒に方向を探す
- 決まったこと、候補、未確定を区別する
- locked項目は変更しない
- 未確定の可能性を事実として扱わない
- 返答は自然な日本語
- 同時にJSONの `draftPatch` を返す

### 9.2 最終変換LLMのシステム指示

要点:

- 会話ログとdraftから、プロジェクト初期データへ変換する
- 本文は生成しない
- 不確定事項は `storyState.openThreads` または `undecided` 由来として残す
- 人物設定は、プロフィールより物語上の揺れと関係性を重視する
- 作品データとシステム指示を混ぜない
- JSONだけを返す

### 9.3 試し書きLLMのシステム指示

要点:

- 相談中の作品案の温度を見るための短いサンプルを書く
- 本番本文として保存されるものではない
- 300から600字程度
- 設定の説明を出さず、短い小説本文だけを書く
- 未確定事項を勝手に確定しない

## 13. リスクと対策

### 13.1 LLMが勝手に決めすぎる

対策:

- confirmed / candidates / undecided を分ける
- 未確定を事実として保存しない
- 最終変換前にレビュー画面を出す

### 13.2 JSONが壊れる

対策:

- 返答表示とdraft更新を分離する
- JSON parse失敗時はdraft更新だけ捨てる
- 失敗ログを残す

### 13.3 locked項目が上書きされる

対策:

- アプリ側でpatch適用時に保護する
- LLMにもlocked項目を明示する

### 13.4 設定が膨らみすぎる

対策:

- draft内の各配列に上限を持つ
- 古い候補は archived にする
- 最終変換では本当に使う情報だけ保存する

### 13.5 相談が長くなりすぎる

対策:

- 途中で「この内容で始める」を常に出せる
- LLMがときどき短く整理する
- 完璧な設定作成より、始められる状態を優先する

### 13.6 既存プロジェクト作成処理との不整合

対策:

- commitは `projectService.createProject` を中心にし、足りない `memories` / `storyState` は後段で保存する
- `CreateProjectBody` を拡張する場合は、既存の単発作成テストを壊さない
- 相談中データをproject配下へ保存しない
- `duplicateFrom` は初期実装で使わない

### 13.7 プロンプトインジェクション

対策:

- ユーザー入力は作品相談の材料として扱い、システム指示やJSON形式の上書き命令として扱わない
- LLM出力のJSONは必ずサーバー側で検証する
- `customSystemPrompt` へユーザーの作品メモを丸ごと入れない
- 内部プロンプト、APIキー、ファイルパスを返答へ出さないようにシステム指示へ明記する

### 13.8 プリセットIDの不一致

対策:

- LLMには利用可能なプリセットID一覧だけを渡す
- 未知のIDが返った場合は既定値に戻すか、近い既存IDへマップする
- 新しいジャンルを増やす場合は、先に `presets/default-presets.json` を更新する

- APIキー未設定時に相談開始で分かりやすいエラーになる
- 相談返答、試し書き、最終変換で用途別プロンプトを使う
- 内部プロンプトやJSONが表示返答へ漏れた場合にdraft更新を捨てる
- 未確定事項を `confirmed` へ入れすぎない
