# 世界設定 2 領域化（世界の土台 / 開始時点の状況）設計書

## 0. 背景

`設定レイヤー分離_設計書.md` の L4 で、world.md 内の「変わらない設定」と「開始時点の状況」を分ける**見出し規約**（オプトインの `## 開始時点の状況`）を導入した。ただし規約であって UI 上の構造ではないため、次の問題が残る。

1. 利用者が規約の存在自体を知らないと機能しない。ガイドの1段落に説明を書いたが、UI に構造がないため書き分けが根付かない。
2. 相談モードのセットアップAIには「`## 開始時点の状況` 見出しを使え」と自然文で指示しているだけで、書き分け精度が担保されない。AI が見出しを付け忘れると、後段の 【世界設定（開始時点の状況）】 セクション分離が働かない。
3. 変わりうる情報が「変わらない設定」側に紛れ込むと、本文が進行しても world.md が古い状況を主張し続ける。L5 の陳腐化ナッジは開始時点側に対して効くが、そもそも分類が正しく行われていることが前提。

本書は、L4 の意味論を維持したまま、書き分けを **UI と AI 出力スキーマの物理構造**として固定する。ファイル数は増やさず、world.md 1 ファイルに 2 見出しを明示保存する新フォーマットに標準化し、旧フォーマットは読み込み時に透過的に写像する。

前提となる現状仕様（変更しないもの）:

- `splitWorldByConvention` が返す segment の意味論（`normal` = 【世界設定】、`initial` = 【世界設定（開始時点の状況）】）と、`renderWorldSettings` が付ける L4 の時点注記は維持する。
- L1〜L3（`Character.currentState` の初期状態再定義、【作品設定】冒頭の時点注記、人物編集UI）は無変更。
- storyState / storyStateService / refineScanService / refineChatService / roleplayPromptBuilder の内部プロンプトは無変更。world を渡す形は「連結後の 1 文字列」を維持する（新フォーマットで書かれた world.md をそのまま渡すだけ）。
- world.md のファイル数は 1 のまま。ファイル分割はしない。
- L5 のナッジ判定に含める hash 対象は現行仕様どおり（world 全文を含む）。

互換性方針:

- **ディスク保存形式は新フォーマットに正規化する**。次のいずれかのタイミングで、world.md は `## 世界の土台` と `## 開始時点の状況` の 2 見出しを持つ形に書き換わる:
  - 利用者が世界タブで保存操作を行ったとき
  - 相談モードのコミットが新規プロジェクトを作成するとき
- **読み込みは常に透過写像**。旧フォーマット（見出しなし・`## 開始時点の状況` のみ）を書き換えずに保管してきたプロジェクトも、`readWorld` の中でパースして `WorldContent` に写像する。ディスクは書き込み操作が入るまで旧形式のまま残る。
- **API 型は差し替え**（外部クライアントは存在しない）。`GET/PUT /projects/:id/world` のペイロードを `{ text: string }` から `{ foundation: string; initialSituation: string }` に変更する。旧形式レスポンスの後方互換維持は行わない。
- **相談モードの出力 JSON スキーマは差し替え+フォールバック**。`worldText: string` を廃止し `world: { foundation: string; initialSituation: string }` を導入する。ただし AI がスキーマから外れて `worldText` を返した場合はパースして 2 領域に写像する（fail-safe）。

区切り規則（`parseWorldMd` と `splitWorldByConvention` の共通仕様）:

world.md は次の 2 モードのいずれかで解釈する。モード判定はコードフェンスの外にある H2〜H4 見出しの中に `## 世界の土台`（前後空白許容、レベル 2〜4）が存在するかどうかで決まる。

- **canonical モード**: `## 世界の土台` を含む入力。**カノニカル 2 見出しのみ**（`## 世界の土台` と `## 開始時点の状況`）を領域区切りとして扱う。それ以外の見出し（`## 現在の勢力図` などのサブ見出しを含む）は本文としてそのままセグメント内に残す。書き込み後の world.md は必ずこのモードになる。
- **legacy モード**: `## 世界の土台` を含まない入力。既存 L4 の挙動を維持する。すなわち `## 開始時点の状況` 見出しがあればそこから次の**同レベル以上の見出し**直前までを `initial` セグメントとし、それ以外は `normal`。見出し自体が無ければ全文 1 セグメント。

**canonical モードを分離した動機**: 開始時点の状況の本文中に「## 現在の勢力図」のようなサブ見出しが混ざるのは、利用者手書きでも setup AI 出力でも頻出パターンである。legacy モードの「同レベル以上で打ち切り」規則をそのまま新フォーマットに適用すると、「保存 → 再読込で領域境界がずれ、書いた内容が別タブへ勝手に移動する」という利用者可視のバグになる。canonical モードでは領域区切りをカノニカル 2 見出しに限定することで、`parseWorldMd(serializeWorldMd(x))` の round-trip 性と、UI とプロンプトの分類一致性を担保する。

`parseWorldMd` が返す `WorldContent` の写像規則:

1. **canonical モード（新フォーマット）**: `## 世界の土台` 直下から次のカノニカル見出しまでを `foundation`、`## 開始時点の状況` 直下から末尾までを `initialSituation` とする。**両見出しより前にあるプレアンブル本文は `foundation` に寄せる**（自己修復。将来の手動編集で見出し位置を崩されても復元できるように）。両見出しは 1 回ずつ出現する前提。2 回目以降の同名見出しは本文として扱う。
2. **legacy モード + `## 開始時点の状況` あり**: 現行 L4 と同じ segment 規則で分割し、`normal` セグメント群を出現順に `foundation`、`initial` セグメント群を出現順に `initialSituation` へ畳み込む。したがって開始見出しと同レベル以上の後続見出しで `normal` に戻った本文は `foundation` に入る（既存プロンプト分類との互換性を保つ）。
3. **legacy モード + 見出しなし（pre-L4 旧フォーマット）**: `foundation = ''`、`initialSituation = 全文`。利用者の指定どおり。
4. **コードフェンスが未閉じの legacy 入力**: 旧 `splitWorldByConvention` と同じ fail-open にし、全文を `foundation`（normal）へ入れる。未閉じフェンスより前に canonical の `世界の土台` マーカーが見えている壊れた canonical 入力だけは、本文欠落を避けて全文を `initialSituation` へ入れる。

**プロンプト挙動の意図した変更**（pre-L4 プロジェクトのみ、初版レビュー §重大1 を受けて明示化）:

pre-L4 プロジェクト（規則 3 に該当）は、保存操作までは `readWorldPromptText` から旧文書をそのままプロンプトへ渡し、従来どおり `【世界設定】{全文}` として扱う。世界設定を保存して canonical 化した後は、UI の写像どおり `【世界設定（開始時点の状況）】{全文}` + 時点注記へ一度だけ切り替わる。

この変更は次の理由で**意図的に受け入れる**:

- pre-L4 プロジェクトの world.md は不変設定と可変状況が未分類で混在しているため、UI では全文を「開始時点の状況」へ寄せる。ただし読み込みだけで生成挙動を変えないため、保存までは legacy 原文を使う。利用者が2領域UIを確認して保存した時点で UI 分類をプロンプトにも反映する。
- L4 済みプロジェクト（規則 2）にとっては行動変化なし。既に手動分類されているため。
- canonical プロジェクト（規則 1）は F1 以降にできるものなので比較対象なし。

したがって読み込みだけでは legacy の生成プロンプトを変えない。canonical 保存後は、`normal → initial → normal` の normal 群前方集約と、pre-L4 全体の initial 化を、利用者の保存操作に伴う意図的な移行差分として受け入れる。

実装順（同一リリース内）:

1. **Phase 1**: F1（型・パーサ・API・保存レイヤー） + F5（refine パッチ経路の整合）
2. **Phase 2**: F2（WorkSettingsTab の 2 タブ化） + F3-3（ProjectForm も 2 領域化）
3. **Phase 3**: F3（相談モードの AI 分類）
4. **Phase 4**: F4（生成プロンプト・ガイド文の整理）

`WorldContent` への型差し替えは `createProject` と setup パイプラインも同時にコンパイル不能にするため、F1〜F5 は同一リリースで実装する。段階リリースする場合は `projectService` と setup の両方に旧 `worldText` を `parseWorldMd` へ通す一時ブリッジが別途必要だが、本実装では採用しない。

## F1. 型・パーサ・API・保存レイヤー

**問題**: 現在は `world.md` を単一の `string` として扱い、L4 分離ロジック `splitWorldByConvention` はプロンプト組み立ての内部だけに閉じている。UI・API・setup コミットのどこにも「2 領域」という概念がない。

**方針**: 「1 領域の文字列」を扱う層を最小化し、公開境界（API・UI・setup 出力）はすべて `WorldContent = { foundation: string; initialSituation: string }` を扱う。ディスクとプロンプトビルダーの中だけが素の文字列を扱う。両者の変換は 1 対の純関数に集約する。

**実装**:

1. `src/shared/types.ts` に型を追加:
   ```ts
   export interface WorldContent {
     foundation: string;
     initialSituation: string;
   }
   ```
2. 新規ファイル `src/server/utils/worldMd.ts` を作り、次の 3 つのエクスポートを置く:
   ```ts
   export function parseWorldMd(text: string): WorldContent;
   export function serializeWorldMd(content: WorldContent): string;
   export function isCanonicalWorldMd(text: string): boolean;
   export function hasCompleteCanonicalWorldStructure(text: string): boolean;
   ```
   - `parseWorldMd` と `splitWorldByConvention` は共通の行スキャナを使う。canonical モードならカノニカル 2 見出しのみを区切りとし、legacy モードなら L4 と同じ分岐に入る。コードフェンス内の見出しは無視し、未閉じ legacy フェンスは全文 normal の fail-open を維持する。CRLF はパース時に正規化し、返す各文字列は前後の空白・空行を trim する。
   - `isCanonicalWorldMd` は「コードフェンス外で `^\s*(#{2,4})\s*世界の土台\s*$` にマッチする行が 1 つ以上ある」ときだけ true。この判定は `parseWorldMd` と `splitWorldByConvention` が同じ関数を共有すること（両者で mode 判定がずれないこと）で担保する。
   - `hasCompleteCanonicalWorldStructure` はコードフェンスが閉じ、コードフェンス外のカノニカル 2 見出しがそれぞれ厳密に 1 回ずつ存在するときだけ true。refine パッチ後の構造保護に使い、片方の削除や重複追加も拒否する。
   - `serializeWorldMd` は常に両見出しを明示する。片側が空でも見出しは省略しない。空側は見出しの直後に空行 1 行だけを置く（初版で指摘された「テンプレートと文言の食い違い」を修正した仕様）:
     ```ts
     export function serializeWorldMd(content: WorldContent): string {
       const foundationBody = content.foundation.trim();
       const initialBody = content.initialSituation.trim();
       const foundationBlock = foundationBody ? `\n${foundationBody}\n` : '\n';
       const initialBlock = initialBody ? `\n${initialBody}\n` : '\n';
       return `## 世界の土台\n${foundationBlock}\n## 開始時点の状況\n${initialBlock}`;
     }
     ```
     これで空側は `## 世界の土台\n\n\n## 開始時点の状況\n...`（見出し直後に空行 1 行）となり、両側非空でも同じ間隔になる。本文行がカノニカル見出しまたはコードフェンス行と一致する場合は構造文字の前に `\` を 1 個加えて保存し、canonical パース時に 1 個だけ戻す。既存の `\` にもさらに 1 個加えるため可逆であり、未閉じコードスニペットを貼っても後続領域見出しがフェンス内へ飲み込まれない。
3. `splitWorldByConvention`（既存、`promptBuilder.ts`）は signature 不変で、内部を mode 分岐に書き換える:
   - `isCanonicalWorldMd(input)` が true のとき: カノニカル 2 見出しのみを区切りとし、`## 世界の土台` セグメントは `normal`、`## 開始時点の状況` セグメントは `initial` として返す。**それ以外の見出し（サブ見出し含む）は本文としてセグメント内に残す**。両見出しの見出し行自体は出力に含めない（既存の `## 開始時点の状況` skip 動作を `## 世界の土台` にも広げるだけで、他のロジックは同一）。
   - `isCanonicalWorldMd(input)` が false のとき: 既存の L4 挙動を維持（コードフェンス処理・`^\s*(#{2,4})\s*開始時点の状況\s*$` 検出・同レベル以上の見出しで打ち切り・fail-open）。
   - legacy の `normal → initial → normal` は 2 フィールドへ畳む際に `normal` 群が `foundation` へ前方集約される。分類と本文は保持するが表示順は変わるため、これを意図した移行差分としてテストで固定する。
4. `storageService.ts`:
   - `readWorld(projectId): Promise<WorldContent>` に signature を変更する。読み込み後 `parseWorldMd` を通す。ファイルが存在しない場合は `{ foundation: '', initialSituation: '' }` を返す。
   - `writeWorld(projectId, content: WorldContent): Promise<void>` に変更する。`serializeWorldMd` を通してから `safeWriteFile` する。
   - 破壊的変更のため、TypeScript コンパイルエラーで全呼び出し元を検出する。呼び出し元の実測結果と対処:

     | ファイル | 経路数 | 対処 |
     |---|---|---|
     | `routes/settings.ts` | R×1, W×1 | 型を差し替え、API レスポンス/リクエストも新形状に（下記 5 参照）|
     | `generationService.ts` | R×4 | `readWorldPromptText` で旧文書順を維持し、意味的に空なら空文字列を渡す |
     | `refineScanService.ts` | R×2 | `readWorldPromptText` を使い、保存前の legacy hash を維持 |
     | `refineChatService.ts` | R×2, W×1 | パッチ適用は F5 で扱う。read → serialize → パッチ適用 → parse → write の順に組む |
     | `roleplaySessionService.ts` | R×1 | `buildWorldDigest` の入力を `serializeWorldMd` 経由に |
     | `projectService.ts` | R×2, W×2 | `createProject` の入力形状変更（F3 と連動）、`copySettings` は `WorldContent` を素通し |
     | `setupCommitService.ts` | 生 JSON パース | F3 で扱う（`readWorld/writeWorld` は直接呼ばない）|

   - 生文字列を要求する既存関数（プロンプト構築、hash 計算、digest 抽出）は `readWorldPromptText` を使う。旧文書は保存までバイト順を維持し、canonical の両領域が空なら空文字列を返して `なし` / `未設定` センチネルを有効にする。
5. `routes/settings.ts` の `GET /projects/:id/world` と `PUT /projects/:id/world` を新形状に:
   ```ts
   // GET → { foundation, initialSituation }
   // PUT body → { foundation, initialSituation }（両フィールド必須。string 型のみ受理。undefined は 400）
   ```
   バリデーションは `typeof body.foundation === 'string' && typeof body.initialSituation === 'string'`。空文字列は許容。
6. `src/client/clientApi.ts` の `updateWorld` と対応する `getWorld` を新形状に:
   ```ts
   getWorld: (id) => request<WorldContent>(`/projects/${id}/world`);
   updateWorld: (id, content: WorldContent) => request<WorldContent>(...);
   ```
   領域単位の UI 保存には `PATCH /projects/:id/world/:area` と `updateWorldArea` を使い、サーバーの project write lock 内で最新値を読み、指定領域だけを更新する。他領域を古いクライアントキャッシュで上書きしない。
7. `setupCommitService.ts:118` の `raw.worldText` パースおよび `buildFallbackWorldText` の出力先は F3 で置き換わる。F1 単独リリースはしない前提のためこの段階では触らない（F1 と F2 と F3 が段階リリースになる場合のみ、中間として `parseWorldMd(raw.worldText ?? '')` を挟む一時ブリッジを入れる）。
8. `promptBuilder` には `readWorldPromptText` の結果を渡す。保存前の legacy は既存出力と等価で、canonical 保存後だけ2領域分類へ切り替わる。canonical の両領域が空なら空文字列を渡し、空の `【作品設定】` ヘッダを出さない。

**受け入れ条件**:

- `parseWorldMd`: 0 節の 4 写像規則（canonical / L4 互換 / 見出しなし / 壊れ入力）で仕様どおりの `WorldContent` を返す。CRLF・見出しレベル 2〜4・見出し前後の空白差・コードフェンス内の同名行・未閉じフェンスをカバーする。canonical モードでは `## 現在の勢力図` のような同レベル・下位レベルの非カノニカル見出しがセグメント区切りにならないことを個別ケースで確認する（初版レビュー §重大2 の再発防止）。
- `serializeWorldMd`: 両見出しを常に出力する。片側が空でも見出しは残り、本文中の同名予約見出しと既存バックスラッシュを可逆エスケープし、trim 済み `WorldContent` について `parseWorldMd(serializeWorldMd(x))` が `x` と等価（round-trip）。両側非空時と片側空時の間隔が同じであること（1 行）。
- `isCanonicalWorldMd`: canonical マーカ 1 個で true、コードフェンス内マーカで false、legacy L4 パターンで false。
- 旧フォーマットの `world.md` を持つ既存プロジェクトが、保存操作なしでも生成・相談・エディタ表示のすべてで新形状 API を通じて 2 領域として扱える（ディスクは未変更）。
- `GET /projects/:id/world` が新形状 `{ foundation, initialSituation }` を返す。旧形状で書かれたクライアントコードはコンパイルエラーで検出できる。
- `writeWorld` を通した後、ディスク上の world.md が canonical モードの新フォーマットになっている（次回 `parseWorldMd` が canonical 分岐に入る）。
- `splitWorldByConvention` の既存 L4 テストが通り、以下を追加する:

## F5. refine パッチ経路の整合（world-replace / world-append）

**問題**: `refineChatService.ts:299〜317` は世界設定への LLM 生成パッチとして `world-replace`（anchor 文字列を検索して置換）と `world-append`（末尾追記）を、`readWorld` から得た生文字列上で直接動かし、結果を `writeWorld` で書き戻している。F1 で保存形式が canonical モードの新フォーマットに固定された後、この経路には以下のリスクがある:

- `world-append` の追記先は「ファイル末尾」だが、新フォーマットではそこが `## 開始時点の状況` セクションの本文末尾になる。この挙動は意図的に受け入れて明文化する（refine パッチは陳腐化解消の色が強く、開始時点側への追記が妥当）。
- `world-replace` の anchor がカノニカル 2 見出しのいずれかに一致すると、見出し行が置換で消える・書き換わる。次回 `parseWorldMd` が canonical 判定を落とし、legacy モード（規則 2 または 3）に落ちて領域境界がずれる。

**方針**: refine パッチ経路を引き続き文字列レベルで動かす。提案時と適用時はいずれも `readWorldText` の同じ原文を使い、legacy 文書の並びを跨ぐ anchor も一致させる。元文書が canonical の場合だけ完全構造 post-check を行い、適用後は `parseWorldMd → writeWorld` で canonical 化する。保存形式更新前の `RefineSession.schemaVersion: 1` に残る pending パッチは読み込み時に stale 化し、再提案を促す。

**実装**:

1. `refineChatService.applyPatch` の world 経路（292〜358 行）を書き換え:
   ```ts
   const [originalWorldText, characters] = await Promise.all([
     storage.readWorldText(projectId),
     storage.readCharacters(projectId),
   ]);
   let nextWorldText = originalWorldText;

   for (const op of patch.operations) {
     switch (op.kind) {
       case 'world-replace': { /* 既存 applyWorldReplace を nextWorldText に対して適用 */ }
       case 'world-append': { /* 末尾追記 = 開始時点セクション末尾への追記 */ }
       // ...
     }
   }

   let nextWorldContent: WorldContent | null = null;
   if (nextWorldText !== originalWorldText) {
     if (hasCompleteCanonicalWorldStructure(originalWorldText) &&
         !hasCompleteCanonicalWorldStructure(nextWorldText)) {
       return recordApplyError(session, patchIndex, 'world パッチ適用でカノニカル見出しが壊れました。anchor を見直してください。');
     }
     nextWorldContent = parseWorldMd(nextWorldText);
   }
   // ...
   if (nextWorldContent) {
     await storage.writeWorld(projectId, nextWorldContent);
   }
   ```
2. `refineChatService` の system prompt（535 行〜）に 1 行追加: 「world-replace の anchor に `## 世界の土台` / `## 開始時点の状況` の見出し行そのものは含めない。これらを削除・書き換えるパッチは失敗する。」LLM 側の生成品質を先に高める。
3. `applyWorldReplace` 自体は無変更（anchor 一致判定は生文字列で行うほうが LLM 出力と噛み合う）。
4. `world-append` の追記先が「開始時点セクション末尾」になる挙動を、`refineChatService` の system prompt 内のパッチ説明にも 1 行足す: 「world-append は開始時点の状況セクションの末尾に追記される。」
5. world / characters / refine session は別ファイルなので、全 operation 検証後の書き込みを `try` でまとめる。後段書き込みが失敗した場合は、読み込み時の world・characters・session スナップショットを再保存してロールバックし、パッチが pending のまま設定だけ部分反映される状態を防ぐ。

**受け入れ条件**:

- 正常系: 開始時点セクション本文中の一意文字列を anchor にした world-replace が適用され、write 後の world.md が canonical モードのまま保たれる。
- 正常系: world-append が開始時点セクション末尾に追記される。
- 異常系: anchor が `## 世界の土台` や `## 開始時点の状況` を含み、置換結果でどちらかの見出しが欠落・重複する場合、パッチ適用が失敗して既存内容が書き換わらない。
- 異常系: world 書き込み後の characters または session 書き込みが失敗した場合、全ファイルが適用前スナップショットへ戻り、パッチは pending のまま残る。
- 既存の refine 適用 UI 動作は変わらない（apply エラーメッセージだけ増える）。
- ユニットテスト: 上記 3 ケース + 既存のパッチテスト回帰。

## L5 ナッジの hash 互換

`refineScanService` の `staticInputHash` は world 全文を含む。読み取りだけで一斉再計算を起こさないよう `readWorldPromptText` は legacy 原文を保存操作まで維持する。canonical 保存が実際に行われたプロジェクトだけ hash が変わり、「設定が変更されています」ナッジが点灯する。

これは:

- L4 済みプロジェクト（規則 2）でも「先頭に `## 世界の土台` 見出しが加わる」ため hash は変わる。
- 旧プロジェクトを開いただけでは hash は変わらない。
- 世界設定を2領域UIから保存した時点で canonical 化されるため、その実変更に対するナッジは正常に点灯する。

## 将来課題（本書のスコープ外）

- **人物設定の可変/不変分割**: `Character.description`（≒ 不変プロフィール）と `Character.currentState`（開始時点の状態、L2 で再定義済み）は同構造の関係にある。今回の world 分割の UI パターン（1 領域内 2 サブタブ、領域ごとの独立編集、片空許容）を人物カードに流用する形で「人物の土台 / 開始時点の状態」の 2 サブタブ化ができる。ただし人物側は既に `speechStyle` / `want` / `fear` / `secrets` などのフィールド分割があり、単純な 2 領域分割ではなく「不変フィールド群 / 開始時点フィールド群」のグループ化になる。共通コンポーネント抽出は今回はしないが、命名（「土台」「開始時点」）だけは揃えておく。
- **AI 分類のヒント返却**: 相談モードで AI が foundation/initialSituation に分けた根拠を短く返させ、UI で「なぜこう分類したか」を表示するモード。書き分け精度の学習曲線を短縮できるが、プロンプトの複雑度と引き換え。書き分け精度の実運用フィードバックが集まってから検討する。
- **foundation 側の陳腐化検出**: 「世界法則が本文中で書き換わる」ケース（大掛かりな世界改変プロット等）は稀に起こる。L5 のナッジは initialSituation の陳腐化を主対象とする現行仕様を維持し、foundation は「基本変わらない」前提でしばらく運用する。
