# 配布準備 レビュー修正指示

コードレビュー(8観点並列 + 検証)の確定所見に基づく修正指示。本書単体で実装に渡せることを目的とする。§1 は必須(重大度順)、§2 は任意。各項目に受け入れ条件を付す。

前提: 対象は未コミットの配布準備実装(D1〜D7)。修正はすべて同じ作業ツリー上に重ねてよい。

---

## §1. 必須修正(重大度順)

### F1. 配布 zip の日本語ファイル名が CP932 で格納される【配布ブロッカー】

**場所**: `scripts/package.mjs:34`(`tar.exe -a -cf`)

**問題**: tar.exe はエントリ名を UTF-8 フラグなし・アクティブコードページ(日本語 Windows では CP932)で格納する。現物 `release/yumeweaving-v0.1.0.zip` で確認済み(全エントリ `flag_bits=0x0`、`Yumeweaving起動.bat` が Shift-JIS バイト列)。非日本語ロケールの 7-Zip 等で展開すると bat とガイドが文字化け名になり、受領者が起動手段を失う。

**修正**(設計書D7 の ASCII フォールバックを発動):

1. zip 内のエントリ名をすべて ASCII にする:
   - 生成 bat: `Yumeweaving起動.bat` → `Start-Yumeweaving.bat`、`Yumeweavingスマホ共有.bat` → `Start-Yumeweaving-LAN.bat`
   - ガイド: `docs/利用ガイド.md` → ステージング時に `docs/UserGuide.md` へリネームコピー(**内容は日本語のまま**。リポジトリ内のファイル名は変えない)
2. 参照の追随: 生成 bat 内の `docs\利用ガイド.md` 言及、`docs/利用ガイド.md` 本文中の bat 名(「インストールと起動」「スマホから使う」)、README の該当記述。
3. **再発防止**: package.mjs の zip 生成後に、エントリ名が ASCII のみであることを検証し、非 ASCII が混入したら exit 1 する自己チェックを追加する(`tar -tf` の出力を `/^[\x20-\x7e]+$/` で検査、または zip ヘッダを直接読む)。
4. 既存の `release/yumeweaving-v0.1.0.zip` は破損状態なので削除して作り直す。

**受け入れ条件**: 再生成した zip を 7-Zip と Explorer の両方で展開してファイル名が正しい。package.mjs に非 ASCII 名を意図的に混ぜると exit 1 する。

### F2. 明示 IP バインド時にトークン URL が表示されず全端末ロックアウト

**場所**: `src/server/index.ts:116` 付近(`if (HOST === '0.0.0.0')`)

**問題**: 認証の有効化条件は `isLanAuthRequiredForHost(HOST)`(非ループバック全般)なのに、トークン付き URL のログは `HOST === '0.0.0.0'` のときだけ。`YUMEWEAVING_HOST=192.168.1.5` 等で起動すると全リクエストが 401 になり、トークンを知る手段がない。

**修正**: ログ出力の分岐を `lanAuthRequired` に揃える。URL の列挙は「`0.0.0.0` / `::` なら `listLanUrls(PORT)`、それ以外は `[\`http://${HOST}:${PORT}\`]`」とし、`lanAuthToken` があれば `appendTokenToUrl` を通して表示する。

**受け入れ条件**: `YUMEWEAVING_HOST=<自機のLAN IP>` で起動するとトークン付き URL がログに出て、スマホから認証・利用できる。`0.0.0.0` の従来動作は不変。

### F3. 中断された npm ci から回復できない(生成 bat 両方)

**場所**: `scripts/package.mjs` の `:install_deps`(2テンプレート両方、~110行目と~163行目)

**問題**: `if not exist "node_modules\"` のフォルダ存在ガードのため、初回インストールを途中で閉じると不完全な node_modules が残り、以後インストールが永久にスキップされ「Cannot find module」で落ちる。:fail はポート衝突を示唆し、利用ガイドにも復旧手順がない。

**修正**:

1. 成功マーカー方式に変更: `npm ci --omit=dev` が exit 0 のときだけ `node_modules\.yumeweaving-install-ok` を作成し、ガードを `if not exist "node_modules\.yumeweaving-install-ok"` にする(npm ci は実行時に既存 node_modules を消すので、再実行自体が部分状態の修復になる)。npm ci 失敗時は `exit /b 1`。
2. `docs/利用ガイド.md` のトラブルシューティングに「起動時に Cannot find module と出る」→「`node_modules` フォルダを削除してもう一度 bat を起動」を追加。

**受け入れ条件**: 空の `node_modules` フォルダだけ置いた状態で bat を起動すると npm ci が走って正常起動する。npm ci を途中で強制終了 → 再起動、でも同様に回復する。

### F4. Cookie 値のエンコード不一致で env トークンが永久 401

**場所**: `src/server/services/lanAuthService.ts:90`(`res.cookie`)/ `:104-115`(`readCookie`)

**問題**: `res.cookie` は値を percent エンコードして保存するが `readCookie` はデコードしない。既定の base64url 生成トークンは無害だが、`YUMEWEAVING_LAN_TOKEN` に空白や `+` を含む値を設定すると、クエリ認証→Cookie 保存→以後すべて 401 のループになる。

**修正**(両方実施):

1. `readCookie` の戻り値を `decodeURIComponent` する(不正シーケンスに備え try/catch で失敗時は生値を返す)。
2. `ensureLanToken` の env 分岐で文字種を検証: `/^[A-Za-z0-9_-]+$/`(base64url)以外なら、起動時に「YUMEWEAVING_LAN_TOKEN は英数と -_ のみ使えます」と明示エラーで exit させる(黙って壊れるより早期失敗)。

**受け入れ条件**: ユニットテスト追加 — ①エンコードが必要な値の Cookie 往復が成立する ②不正文字の env トークンが起動時エラーになる ③既定トークンの既存テストが通る。

### F5. lan-token.json の内容不検証(起動不能 / 全リクエスト 500)

**場所**: `src/server/services/lanAuthService.ts:28-32`

**問題**: (a) 不正 JSON だと `readJsonFile` が SyntaxError を再スローし、`main().catch` が原因ファイル名なしの一般エラーで終了 → LAN モードが復旧手順不明のまま起動不能。(b) `{"token":123}` は truthy チェックを通り、リクエストごとに `createHash().update(数値)` が TypeError → 全 LAN トラフィックが 500。

**修正**: `ensureLanToken` 内で読み込みを try/catch し、「パース失敗」または「`typeof token !== 'string' || token === ''`」の場合は、`console.warn` で `lan-token.json を再生成します`(フルパス付き)と告知して新トークンを生成・上書き保存する。破損でアプリを止めない(トークンは再共有すればよい消耗品)。

**受け入れ条件**: ユニットテスト追加 — 不正 JSON / 数値 token / 空文字 token のそれぞれで、例外にならず新しい文字列トークンが返り、ファイルが修復される。

### F6. 空文字 YUMEWEAVING_DATA_DIR で開発サーバーが実データを見る

**場所**: `scripts/dev-server.mjs:5`

**問題**: `??` は空文字を素通しし、`config.ts` の truthy 判定は空文字を「未設定」と見て `Documents\Yumeweaving` に解決する。空エクスポート(PowerShell の `$env:X=''`、CI)で `npm run dev` が実データに書き込む。

**修正**: 空白のみも未設定扱いにする:
```js
if (!process.env.YUMEWEAVING_DATA_DIR?.trim()) {
  process.env.YUMEWEAVING_DATA_DIR = 'data';
}
```

**受け入れ条件**: `YUMEWEAVING_DATA_DIR=''`(空文字)で `npm run dev` 相当を起動すると DATA_DIR ログが リポジトリ内 `data` を指す。

### F7. 配布 zip 内で「npm run build を実行してください」と案内される

**場所**: `scripts/start-lan.mjs:8`(dist 欠落時のエラーメッセージ)

**問題**: このスクリプトは zip に同梱され、生成 LAN bat から呼ばれる。展開不全などで dist が欠けると、src も devDeps もない環境で実行不能なビルドコマンドを案内してしまう。

**修正**: メッセージを両対象向けに変更: `[start:lan] dist/server/index.js が見つかりません。開発環境では npm run build を実行してください。配布版では zip を展開し直してください。`

**受け入れ条件**: dist を退避して bat / `npm run start:lan` を実行し、新メッセージが表示される。

### F8. ?token= 付き非 GET リクエストが 302 で握り潰される

**場所**: `src/server/services/lanAuthService.ts:96`

**問題**: クエリトークン検証成功時の 302 リダイレクトがメソッド不問。現行クライアントは発火しないが、Cookie 失効後にトークン付き URL で POST /api/... を叩く利用(ガイドがトークン URL 共有を案内している)ではミューテーションが黙って消える。

**修正**: 302 リダイレクトは `req.method === 'GET' || req.method === 'HEAD'` のときのみ。それ以外のメソッドは Cookie を設定したうえで `next()` に流し、リクエストをそのまま実行する。

**受け入れ条件**: ユニットテスト追加 — 有効な `?token=` 付き POST がハンドラまで到達し、レスポンスに Set-Cookie が付く。GET は従来どおり 302。

### F9. 生成 bat テンプレートの約40行重複

**場所**: `scripts/package.mjs:74-177`(`createDesktopStartBat` / `createLanStartBat`)

**問題**: `:check_node` / `:node_version_fail` / `:install_deps` / `:fail` / 前置きが逐語重複(3つのレビュー観点が独立に同一指摘)。共通ロジックの修正漏れが受領者の初回起動で発現する。

**修正**: 共通テンプレート1本に統合する。例: `createStartBat({ introLines, launchLines, failTitle })` が共通サブルーチンを内包し、両者は差分(起動コマンドと echo 2行、fail タイトル)だけ渡す。F1・F3 の修正はこの統合後のテンプレートに対して行うと一度で済む(**実装順は F9 → F3 → F1 を推奨**)。

**受け入れ条件**: 統合後の生成 bat 2本の内容が、意図した差分(起動コマンド・echo・タイトル)以外は同一である。

### F10. 意図を記録した NOTE コメント2件の消失(コメント規約違反)

**場所**: `src/server/index.ts:44` / `scripts/start-lan.mjs:1`

**問題**: グローバル CLAUDE.md の規約(コメントは「なぜこの実装か」を保存する)に違反。①CORS ループの NOTE が「LAN モードではスマホが same-origin fetch でも Origin ヘッダを送るケースがあるため LAN IPv4 を許可リストへ自動追加する」という実際に踏んだバグの記録を失い、事実の記述だけになった。②start-lan.mjs 冒頭の「シェル差(cmd/PowerShell/bash)の env 書き分けを避けるため Node スクリプト経由にしている」という存在理由の NOTE が削除された。

**修正**:

1. 両 NOTE を復元する(原文の日本語のままでよい。翻訳したい場合も**理由を落とさない**こと)。
2. あわせて `src/server/routes/system.ts` のシャットダウン NOTE を実プロセス連鎖に合わせて更新する(dev 時の親は現在 `dev-server.mjs` 経由の多段: concurrently → npm → dev-server.mjs → tsx watch → server。動作自体は実機検証で正常確認済み)。

**受け入れ条件**: 3箇所のコメントが「何をするか」ではなく「なぜそうしているか」を含む。
