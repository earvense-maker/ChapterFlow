# 記憶系整合性改善 設計書

作成日: 2026-07-21
対象バージョン: 0.1.0-beta.7 予定
実装後の扱い: 実装完了後 `archive/` へ移動する。

## 0. 背景

連載を進めていくと、以下3つの症状が繰り返し観察される。

- **症状1**: 過去の発言がどの人物のものか混同される（会話の主体入れ替わり、約束の主客反転）。
- **症状2**: 視点人物が知らないはずの出来事を知っている前提で書かれる（POV違反、伏線の早漏れ）。
- **症状3**: NG表現に登録した言い回しが本文に出る。

各症状の根因は別途レビュー結果に整理済み。本設計書は、そのうち以下3本の改善を対象とする（優先度と実装コストのバランスで選抜）。

- **トラック1（症状1向け）** 1A: 重要イベントに発話主体（`actor`）・受け手（`recipient`）を持たせ、生成プロンプトに主客付きで表示する。
- **トラック2（症状2向け）** 2A: 抽出プロンプトの既定を反転し、「その場に居なかった人物は原則 `explicitlyUnknownBy`」で列挙する。上限も緩める。
- **トラック2（症状2向け）** 2B: `characterStates.knowledge` を抽出プロンプトの守備範囲に入れ、自動更新（追加・削除）できるようにする。
- **トラック3（症状3向け）** 3A: 本文生成プロンプトのNGセクションを【今回の希望】の直前へ移動する。頻出フレーズと分離する。

視点人物検出の拡張・参考資料の視点フィルタ・NG事後検証は本設計書の対象外（別トラック）。

## 前提となる現状仕様（変更しないもの）

- `StoryState` は `data/projects/{projectId}/story-state.json`。schemaVersion は 1 のまま変更しない。
- 状態抽出は `updateStoryStateFromAcceptedScene` が採用シーンごとに実行し、`mergeStoryState` で適用する。温度0.15、応答は `application/json`。
- 抽出プロンプトの `presentCharacters` / `learnedBy` は「保存せず、knownBy に畳む」既存方針（K1/K2）を維持する。
- 生成プロンプトの区切りは `\n\n---\n\n`。
- NG表現の上限（`BAN_LIMIT_TOTAL = 12`）・登録側の正規化・UI表示は本設計書では変更しない。

## 互換性方針

- **新規フィールドはすべてオプショナル**。読み込み時にサービス層で欠損を補正する。既存プロジェクトの「未指定」項目は、以降**新規に採用したシーン**の抽出結果からのみ埋まる。
  - `processedGenerationIds` に載っている既存シーンは自動では再抽出されない（[generationService.ts:1113](../../src/server/services/generationService.ts:1113) のドレインは未処理分だけを対象とする）。過去シーンの `actor` / `recipient` / `explicitlyUnknownBy` / `knowledge` を埋めたい場合は K7 の revert で当該 diff を巻き戻し、K6 のバックログドレインで再抽出させるか、K8 の物語状態編集画面で手動編集する。
  - 本設計書では「既存プロジェクトの遡及再抽出」は行わない（自壊リスクとコスト過大のため）。既存プロジェクトへの効き方は、以降の新規シーンから徐々にという緩やかな適用になる旨をリリースノートに明記する。
- `schemaVersion` は据え置く。
- 既存の `characters` / `visibility` / `knownBy` / `explicitlyUnknownBy` フィールド自体は残す。actor/recipient は追加情報。トラック2Aで `explicitlyUnknownBy` の**差分パッチ**の受け付け方（フィールド上書きから追加/削除操作への移行）は変わるが、保存済みデータの読み取り側は変わらない。
- クライアント（`src/client`）は本設計書のスコープでは変更しない。物語状態編集画面（`WorkSettingsTab.tsx` の物語状態タブ）は現在生JSON編集で、新フィールドも JSON 経由でそのまま編集可能。人物選択UI等の追加は別トラック（「未対応の残課題」参照）。

## 実装順の推奨

依存関係が浅いため、以下の順で独立に実装できる（並行実装可）:

1. **Phase A**: トラック3A（プロンプト内NG位置の移動）— 数行修正、即効性最大。
2. **Phase B**: トラック1A（actor/recipient 追加）— 抽出プロンプト・型・レンダラの拡張。
3. **Phase C**: トラック2A（explicitlyUnknownBy 既定反転）— 抽出プロンプトの方針転換と上限緩和。
4. **Phase D**: トラック2B（characterStates.knowledge 自動更新）— C の後段で、抽出プロンプト・マージロジックの追加。

Phase C と D はどちらも `buildUpdatePrompt` を触るため、続けて実装するとコンフリクトを避けやすい。

## トラック3A: 本文生成プロンプトのNG配置修正

### 3A. NGセクションを【今回の希望】の直前へ移動、頻出フレーズと分離

**問題**

`buildPrompt` の現行構成は以下の順（[promptBuilder.ts:44-188](../../src/server/prompts/promptBuilder.ts:44)）:

1. 【この作品の核】
2. 【作品設定】/【参考資料】
3. 【現在状態スナップショット】
4. 【重要な過去イベント】
5. 【好み・NG】（メモ由来のNG）
6. 【これまでの要約】
7. 【出力形式】
8. 【表現上の注意】（登録NG + 直近本文の頻出フレーズ）
9. 【これまでの作品本文（直近）】
10. （書き直しモードのみ）【今回書き直し／別案対象の場面】
11. 【文体見本】
12. 【今回の希望】

登録NGは 8 に置かれるが、直後の 9（直近本文）にNG該当フレーズが出ていると、末尾追従の強い弱モデルほど直近本文を模倣する。ロールプレイ側は末尾寄り（scenario → summary → recent → banned → 指示）に配置しており、こちらに揃える。

また現行の【表現上の注意】は「使わない表現」と「直近本文で頻出した表現」を並置しているが、後者は「多用を避け」の弱い指示で、前者の強度を希釈している。

**方針**

- NG（登録済み）は本文の最後尾、【今回の希望】の直前に置く。
- 頻出フレーズは別セクション（【表現の重複を避ける】）として分離し、位置は現状同等（直近本文の直後）。
- 【好み・NG】セクションのNG部分は残す（プリファレンス寄りのNG。手動登録の言い回しとは意味が違うため）。ただし冒頭注記で「言い回し単位の禁止は末尾の【使わない表現】を参照」と1行付ける。

**実装**

1. `buildPrompt` の `parts` 積み込み順を以下に変更:

   1. 【この作品の核】
   2. 【作品設定】/【参考資料】
   3. 【現在状態スナップショット】
   4. 【重要な過去イベント】
   5. 【好み・NG】（プリファレンス由来のみ。冒頭注記追加）
   6. 【これまでの要約】
   7. 【出力形式】
   8. 【これまでの作品本文（直近）】
   9. （rewrite / variate モードのみ）【今回書き直し／別案対象の場面】
   10. 【表現の重複を避ける】（頻出フレーズ。0件なら省略）
   11. 【文体見本】
   12. 【使わない表現】（登録NG。0件なら省略）
   13. 【今回の希望】

   rewrite / variate モードでは【今回書き直し／別案対象の場面】が【これまでの作品本文（直近）】と【表現の重複を避ける】の間に入る。以降のセクション順は continue モードと同一。

2. `renderBannedExpressions` を2関数に分割:
   - `renderRegisteredBannedExpressions(expressions)`: 手動登録NGのみ。0件なら空文字。
   - `renderFrequentPhraseNotice(frequentPhrases)`: 頻出フレーズのみ。0件なら空文字。
3. `renderRegisteredBannedExpressions` の文面は現行から次に強化:
   ```
   【使わない表現】
   以下の言い回しは今回の本文に出さないこと。同じ意味は別の言い方で書く。
   引用符・括弧内・地の文の別を問わず、部分一致も避ける。
   - 「〜」
   ```
4. `renderPreferenceNotes` のNG部分の直後に注記を1行追加:
   ```
   ※言い回し単位で禁止したい語句は末尾の【使わない表現】に登録する。
   ```
5. `getContextSummary` / `getStoryState` / `getRecentContext` などの読み出しは変更しない。

**受け入れ条件**

- 登録NGありのとき、【使わない表現】が【今回の希望】の直前に1回だけ出る。
- 頻出フレーズがあるとき、【表現の重複を避ける】は continue モードでは【これまでの作品本文（直近）】の直後、rewrite / variate モードでは【今回書き直し／別案対象の場面】の直後に出る。
- 登録NG 0件・頻出フレーズ 0件 のとき、対応セクションは出ない（区切り「---」も追加されない）。
- `expressionService.resolveBannedExpressions` の返り値・件数上限には変更なし。既存の `expressionService.test.ts` が全て通る。
- **既存の `promptBuilder.test.ts` の破壊的改修が必要**（追加だけでは通らない）。ラッパー見出し【表現上の注意】は撤去、【直近本文で頻出した表現】は【表現の重複を避ける】に改名されるため、下記アサーションを新仕様に沿って書き換える:
  - `promptBuilder.test.ts:608` `bannedIndex = userPrompt.indexOf('【表現上の注意】')` の順序検証は、【使わない表現】に置換した上で「【使わない表現】が【今回の希望】の直前」に反転する（現行は【出力形式】と【今回の希望】の中間位置検証、新仕様では【使わない表現】が【今回の希望】の直前になる）。
  - `promptBuilder.test.ts:629` の `expect(userPrompt).not.toContain('【表現上の注意】')` は「【使わない表現】と【表現の重複を避ける】の両方が現れないこと」の2件チェックに置換する。
  - `promptBuilder.test.ts:682` / `:696` / `:703-704` / `:759` の `【直近本文で頻出した表現】` 参照は `【表現の重複を避ける】` に置換する。
  - `promptBuilder.test.ts:705` の `split('【表現上の注意】')` によるセクション抽出は `split('【使わない表現】')` に変更する（手動NGが登録されているケースの検証なので置換先は【使わない表現】）。
- **ユニットテスト（新規追加）**（`promptBuilder.test.ts` / `promptBuilderLayering.test.ts`）:
  - continue モードの13セクション順序スナップショット（登録NG・頻出フレーズあり/なし の4パターン）。
  - rewrite / variate モードの順序スナップショット（【今回書き直し／別案対象の場面】が【これまでの作品本文（直近）】と【表現の重複を避ける】の間に入る）。
  - `resolveBannedExpressions` が空配列を返したときに【使わない表現】が現れない。
  - rewrite / variate モードでも【使わない表現】が【今回の希望】の直前に来る。
  - 【好み・NG】直下に「※言い回し単位で禁止したい語句は末尾の【使わない表現】に登録する。」の注記が出る（プリファレンス由来NGがあるケース）。

## トラック1A: importantEvents に actor / recipient を持たせる

### 1A. 発話主体・受け手の構造化

**問題**

`StoryEventRecord.characters` は「関係人物」のフラット配列（[types.ts:205-216](../../src/shared/types.ts:205)、[storyStateService.ts:210-222](../../src/server/services/storyStateService.ts:210)）。「太郎が花子に告白した」という出来事も `characters: ['太郎', '花子']` としか残らない。生成プロンプトの【重要な過去イベント】でも `関係人物: 太郎 / 花子` としか表示されないため、以降の生成でモデルが「花子が太郎に告白した」と主客を反転させても止められない。

**方針**

`StoryEventRecord` に **`actor`**（主体 characterId）と **`recipient`**（受け手 characterId）を追加する。抽出プロンプトで「発話・宣言・告白・約束・秘密の開示・攻撃・受諾・拒絶などの主客が意味を持つ出来事」にのみ埋めさせる。生成プロンプトでは `関係人物` の前に `主体→受け手` の形で出す。

`characters`（関係人物）は残す（当事者以外の観客・言及される第三者も含めるため）。

**注意**: `characters` は自由文字列（本文中の呼称の配列、`asStringArray` 経由。[storyStateService.ts:505](../../src/server/services/storyStateService.ts:505)）、actor / recipient は characterId。**別空間の別軸データで、包含関係は保証しない**（コード上の同一性検証もない）。生成プロンプトの描画側でも【重要な過去イベント】の `関係人物` は生の名前、`主体` は `characterNameForId` 解決名（[promptBuilder.ts:404-406](../../src/server/prompts/promptBuilder.ts:404)）と経路が違うため、両者の表示が食い違うケースが原理的に存在する（例: characters に「兄さん」、actor に characterId 経由の「太郎」）。運用上の一致は抽出LLMの品質に依存するため、後任者は「食い違いをコードで是正する」のではなく「両方見せて読者/著者が判断する」設計として扱う。

**実装**

1. 型変更（`src/shared/types.ts`）:
   ```ts
   export interface StoryEventRecord {
     eventId: string;
     sceneId: SceneId | null;
     summary: string;
     characters: string[];
     visibility: string;
     knownBy?: CharacterId[];
     explicitlyUnknownBy?: CharacterId[];
     actor?: CharacterId | null;      // 追加。主体
     recipient?: CharacterId | null;  // 追加。受け手
     importance: MemoryImportance;
     status: StoryItemStatus;
     updatedAt: string;
   }
   ```
   両フィールドとも省略可・null 可。差分パッチにおける挙動は以下:
   - **キー不在**（`hasField(patch, 'actor') === false`）: 既存値を保持。差分未言及として扱う。
   - **`actor: null`（明示 null）**: 既存値を null に上書き。「主体不明への訂正」を許可する意味論。
   - **`actor: 'char-xxx'`**: 既存値を上書き。人物ヒントに無い ID は `normalizeSingleCharacterId` で null に落とす。
   - recipient も同様。

2. 抽出プロンプト（`storyStateService.buildUpdatePrompt`）の差分JSON形式に追加:
   ```json
   {
     "eventId": "既存eventIdまたは空文字",
     "sceneId": "...",
     "summary": "...",
     "characters": ["関係人物"],
     "presentCharacters": ["char-id"],
     "learnedBy": ["char-id"],
     "knownBy": ["char-id"],
     "explicitlyUnknownBy": ["char-id"],
     "actor": "主体のcharacterIdまたはnull",
     "recipient": "受け手のcharacterIdまたはnull",
     "importance": "high",
     "status": "active"
   }
   ```
   指示に以下を追加:
   ```
   - actor は「その出来事の主体（発話者・行為者・宣言者）」のcharacterId。単独行為なら recipient は null。
   - recipient は「その出来事の受け手・宛先（宣告された相手、告白された相手など）」のcharacterId。
     宛先が特定できない出来事（独白・情景・自然現象）では null にする。
   - actor / recipient は当事者性が明確な場合にのみ埋める。誰が主体か本文から読み取れない場合は null にする（推測しない）。
   - actor / recipient に指定する characterId は、人物ヒントに存在するIDに限る。名前一致で複数候補ある場合はnullにする。
   - actor / recipient は「主客の構造」を残すためのものであり、知識状態（knownBy / explicitlyUnknownBy）とは独立に扱う。actor だからといって自動で knownBy に含める、といった暗黙の連動はしない。知識状態は presentCharacters / learnedBy / knownBy / explicitlyUnknownBy 側で別途判断する。
   ```

3. 正規化（`normalizeEventRecord` / `normalizeEventPatch`）:
   - `normalizeCharacterIdList` の単数版として `normalizeSingleCharacterId(value, characters): CharacterId | null` を追加。人物一覧に存在しない ID は null に落とす。
   - **`characters` が空配列のときの挙動**: `normalizeCharacterIdList` は空時に `allowAny = true` で任意文字列を通す（[storyStateService.ts:711-713](../../src/server/services/storyStateService.ts:711)）。本設計の `normalizeSingleCharacterId` は**この allowAny 分岐を継承せず、`characters` が空なら常に null を返す**。理由は (a) 受け入れ条件「人物ヒントに存在しない ID は null に落ちる」の一貫性、(b) 単数値なので「呼称のまま保存」しても表示側で `characterNameForId` 解決に失敗するだけで実益がないため。この差異は関数の doc コメントに明記する。
   - `normalizeEventRecord` で `actor` / `recipient` を読み取り、正規化して保存する。
   - `normalizeEventPatch` は `hasField` で差分の言及有無を判定する。キー不在なら `existing?.actor ?? null` を維持、明示 null なら null に上書き、ID指定なら正規化のうえ上書き。recipient も同様。
   - actor / recipient と knownBy / explicitlyUnknownBy は独立して抽出・保存する。「actor は必ず knownBy に含まれる」といった自動包含はしない（例: 背後からの攻撃・催眠中の宣告など、recipient が actor を認識していない出来事があるため）。抽出プロンプト側で actor が明らかに知っているなら presentCharacters / learnedBy 経由で knownBy に入る前提。

4. `setupCommitService.normalizeStoryEvent`（[setupCommitService.ts:515](../../src/server/services/setupCommitService.ts:515)）にも actor / recipient の正規化を追加する。セットアップ経路でLLMが返した StoryEvent に actor/recipient が含まれるケースを想定し、`normalizeSingleCharacterId` を経由して保存する。存在しない ID は null に落とす。

5. 生成プロンプトの描画（`renderImportantPast` in `promptBuilder.ts`）:
   - 現行の `関係人物: 太郎 / 花子` の前に `主体: 太郎 → 花子` を追加。
   - `recipient` が null のときは `主体: 太郎`。
   - `actor` が null のときは `主体` 行を出さない（従来通り `関係人物` のみ）。
   - `characterNameForId` で name に解決。解決不能（人物削除済み）なら ID をそのまま出す。

   出力例:
   ```
   - 太郎が海辺で花子に想いを打ち明けた（重要度: high、主体: 太郎 → 花子、関係人物: 太郎 / 花子、知っている人物: 太郎 / 花子）
   ```

6. K7 の diff サマリ（`summarizeDiff`）は変更不要（イベント単位で追加/更新を検出するため、actor/recipient のみの変化も `updatedEvents` に入る）。

**受け入れ条件**

- 抽出で「太郎が花子に告白した」に相当する場面から `actor: 'char-taro'`, `recipient: 'char-hanako'` が入る。
- 独白（一人だけの場面）で actor が入り recipient が null になる。
- 差分で `actor: null` を明示すると既存の actor が消える（主体不明への訂正が可能）。
- 差分で actor キーが不在なら既存値が保持される（`hasField` 判定）。
- 生成プロンプトの【重要な過去イベント】に `主体: 太郎 → 花子` が出る。
- 人物ヒントに存在しない ID が指定された場合は null に落ちる（人物削除後も破綻しない）。
- 既存プロジェクト（actor/recipient 未定義）でもエラーなく読み込め、【重要な過去イベント】の描画から主体行だけがスキップされる。
- actor / recipient と knownBy は独立して抽出される。「actor だから knownBy に自動包含」は行わない。背後からの攻撃・催眠中の宣告等で recipient が knownBy に含まれない抽出結果も許容される。
- **ユニットテスト**（`storyStateService.test.ts` / `promptBuilder.test.ts` に追加）:
  - 正規化: キー不在・null・存在しない ID・存在する ID の4ケース。
  - マージ: 差分の actor/recipient が既存に上書きされる。差分明示 null で消える。キー不在で既存値が保持される。
  - actor / recipient 抽出は knownBy に影響しない（両フィールドを独立に検証）。
  - プロンプト描画: actor あり/なし、recipient あり/なし の4ケースの文字列。

**懸念点と対応**

- 抽出LLMが actor / recipient に「太郎」のような名前文字列を返す誤りは、`normalizeSingleCharacterId` で ID 検証時に null に落とす。抽出プロンプトで「characterId のみ」を強調する。
- 三人以上の会話（例: 家族会議）で recipient が単数だと表現力が足りない。今回は単数のみとし、必要があれば将来 `recipients?: CharacterId[]` へ拡張する。単数優先の理由は、多数の場合 `characters` / `presentCharacters` で十分表現できるため。

## トラック2A: explicitlyUnknownBy の既定を「反転」

### 2A. 「その場に居なかった人物は原則 explicitlyUnknownBy」に方針転換

**問題**

現行の抽出プロンプトは以下の運用（[storyStateService.ts:185](../../src/server/services/storyStateService.ts:185)）:

```
- explicitlyUnknownBy は通常0〜2名。知らないことが物語上の緊張や皮肉を生む人物だけを入れ、
  その場にいなかった人物を機械的に列挙しない。
```

コードの上限も `MAX_EXPLICITLY_UNKNOWN = 4`（[storyStateService.ts:31](../../src/server/services/storyStateService.ts:31)）。結果、大半のイベントで `explicitlyUnknownBy` は空。生成プロンプトの【人物の情報状態】は「まだ知らない」がほぼ空欄になり、モデルへのネガティブ制約が効かない。POV違反の根本原因はここ。

**方針**

「その場に居合わせず、伝聞・観察でも記録の閲覧でも、その場面が終わった時点でその情報を知り得ない人物」は原則として `explicitlyUnknownBy` に入れる。**時間軸を場面終了時点に固定**する（「後日伝えに行く予定があるから」という理由で除外してはいけない。報告される場面が来たら、その場面の抽出で `learnedBy` 経由で `knownBy` に移すことで状態遷移させる）。上限は 4 → 12 に上げる。

差分パッチの受け付け方は**フィールド上書きから追加/削除操作へ移行**する。現行 `explicitlyUnknownBy` フィールドをパッチキーで指定すると既存を全置換する挙動（[storyStateService.ts:563-570](../../src/server/services/storyStateService.ts:563)）は、LLMが偶発的に空配列を返した際に既存値を全消去してしまう。`knownBy` が累積型（`presentCharacters` / `learnedBy` を和集合で追加）である運用と非対称でもある。`addUnknownBy` / `removeUnknownBy` の分離操作に統一する。

**トレードオフの明示**: 本トラックは以下の2変更を同時に入れる:

1. `explicitlyUnknownBy` を既定で埋める（居合わせない主要人物を全員）
2. 生成プロンプトの否定制約強化（「噂話・比喩・伏線としても既知扱いしない」、実装 5）

この組み合わせは**真のPOV違反（症状2）を減らす代わりに、抽出LLMが `knownBy` に取り込み損ねた「実は知っている人物」を誤って `explicitlyUnknownBy` に入れた場合、強化された制約がその人物の正当な描写をより強く抑制する**（false-negative の増幅）。反転は「真の unknown」も「偽の unknown」も同時に増やし、強化ワードが後者の害を大きくする構造。

本設計はこのトレードオフを「症状2の抑止 > 時々の過剰抑制」として受容する。誤抑制が観測された場合の緩和策（プロンプト否定文の弱化、`removeUnknownBy` での手動訂正、K7 revert での再抽出）を用意しているため、実運用で問題化したら受け入れ後にチューニングする。

**実装**

1. 定数変更（`storyStateService.ts` および `setupCommitService.ts`）:
   ```ts
   const MAX_EXPLICITLY_UNKNOWN = 12; // 4 → 12
   ```
   `storyStateService.ts:31` と `setupCommitService.ts:524` の両方を更新。`MAX_EVENT_KNOWN_BY` は 12 のまま。合計が人物総数を超えても、`explicitlyUnknownBy` から knownBy を除外する既存フィルタ（[storyStateService.ts:499](../../src/server/services/storyStateService.ts:499)、[storyStateService.ts:570](../../src/server/services/storyStateService.ts:570)、[setupCommitService.ts:525](../../src/server/services/setupCommitService.ts:525)）で整合が取れる。

2. 抽出プロンプトの指示を差し替え（`storyStateService.buildUpdatePrompt`）:

   置き換え前:
   ```
   - explicitlyUnknownBy は通常0〜2名。知らないことが物語上の緊張や皮肉を生む人物だけを入れ、その場にいなかった人物を機械的に列挙しない。
   ```

   置き換え後:
   ```
   - explicitlyUnknownBy はパッチキーとしては使わない。代わりに addUnknownBy / removeUnknownBy を使う。
   - addUnknownBy には、この場面終了時点で、その場に居合わせず、伝聞・立ち聞き・観察・記録の閲覧などによっても知り得ない人物のcharacterIdを入れる。原則としてknownBy以外の主要人物は全員入れる。
       - 判定は「この場面が終わった瞬間」で切る。次の場面で当人に伝えに行く予定があっても、この場面終了時点ではまだ知らないので addUnknownBy に入れる。実際に情報が伝わる場面の抽出で learnedBy 経由で knownBy へ移す。

   ```

3. 差分JSON形式のイベント例を修正（`buildUpdatePrompt` 内の JSON.stringify するテンプレート）:

## トラック2B: characterStates.knowledge の自動更新

### 2B. 抽出で「知った / 積極的に知らないと明かした」を捕捉する

**問題**

現行の抽出プロンプトは以下（[storyStateService.ts:181](../../src/server/services/storyStateService.ts:181)）:

```
- characterStates の knowledge は更新対象ではないため出力しない。
```

`StoryCharacterState.knowledge` は setup 時の初期値と手動編集でしか更新されない。本文が積み上がっても「A は B の秘密を知っている」といった人物固有の知識は明示化されず、生成プロンプトの【人物の情報状態】は importantEvents.knownBy 経由の情報だけに依存する。

しかし importantEvents は「事件」しか記録しないため、以下は捕捉できない:

- 「A は B が甘いものが苦手だと気づいた」（イベントではないが、以降の描写に効く）
- 「A は自分がまだ本名を知らないことに気づいた」（否定的な知識）
- 「A は父の職業について公式には知らないが、うすうす察している」（曖昧な知識）

これらが積み上がらないため、後段の生成で人物ごとの知識境界がブレる。

**方針**

`characterStates.knowledge` を抽出プロンプトで更新できるようにする。差分パッチとして「追加」「削除（誤登録の取り消し）」の2操作を受け付ける。イベント単位で記録するほどでもない小さな知識・観察を、人物単位で自然言語のまま積み上げる置き場所にする。

**実装**

1. 型は変更なし。`StoryCharacterState.knowledge: string[]` の既存フィールドを流用する。

2. 抽出プロンプト（`storyStateService.buildUpdatePrompt`）の差分JSON形式を修正:

   置き換え前:
   ```
   - characterStates の knowledge は更新対象ではないため出力しない。
   ```

   置き換え後（characterStates 内の指示を再構成）:
   ```
   - characterStates.knowledge には、今回の場面で当該人物が新たに得た小さな知識・観察・仮説を短文で追加する（1件30字以内推奨、1人物あたり今回追加は最大3件）。
     イベント化するほどでもないが「以降その人物の描写に効く」性質のもの。例: 「Bの好みが苦味に偏っていることに気づいた」「Cが左利きだと知った」「Dの職業を疑い始めた」。
   - 既存の knowledge から取り消したい項目がある場合、characterStates.removeKnowledge に完全一致する文字列を並べる。誤抽出の巻き戻し用。
   - 重複する内容や、importantEvents に既に載せた事件そのものは knowledge に書かない。
   ```

3. 差分JSON形式の characterStates 例に `knowledge` / `removeKnowledge` を追加:
   ```json
   {
     "characterId": "既存characterIdまたはnull",
     "name": "人物名",
     "currentState": "今回更新が必要な現在状態",
     "relationships": ["今回更新が必要な関係変化"],
     "knowledge": ["今回追加する知識・観察（各30字以内、最大3件）"],
     "removeKnowledge": ["既存knowledgeから削除する完全一致文字列"]
   }
   ```

4. マージロジック（`normalizeCharacterStatePatch` in `storyStateService.ts`）を修正:
   - `hasField(value, 'knowledge')` かつ配列なら、`asStringArray(value.knowledge, 3)` を既存 knowledge に追加（末尾追記）。
   - `hasField(value, 'removeKnowledge')` かつ配列なら、`asStringArray(value.removeKnowledge, 12)` を既存 knowledge から `normalizeComparableText`（[characterStateMatching.ts:143](../../src/server/utils/characterStateMatching.ts:143)）経由で照合して除去。`normalizeComparableText` は NFKC 正規化 + trim + **連続する空白の1個化** + 小文字化を行う。全角/半角の差、大文字小文字差、連続空白の差は吸収されるが、**空白の有無自体は吸収されない**（例: 「A は 気づいた」と「A は気づいた」は別物として扱われる）。この制約は受け入れ条件に明記し、抽出プロンプト側で `removeKnowledge` に指定する文字列は既存 knowledge の文字列と厳密に一致させることを求める。
   - 追加後、`mergeUniqueStrings` 相当のロジックで重複除去（同じく `normalizeComparableText` ベース）。
   - 全体の上限は既存の 12 件（`asStringArray(value.knowledge, 12)` を通す既存の `normalizeCharacterState`）を維持。上限を超えた場合は古いものから捨てる（先頭を切り落とす）。

5. マージヘルパーとして `mergeKnowledgeList(existing, add, remove): string[]` を新設し、追加削除ロジックを1関数に閉じる。テスト容易性のためエクスポートする。

6. 生成プロンプトの描画（`renderCharacterKnowledgeState` in `promptBuilder.ts`）は変更不要。既に `state.knowledge` を表示している（[promptBuilder.ts:481-483](../../src/server/prompts/promptBuilder.ts:481)）。knowledge が自動で積まれれば自然に反映される。

7. K7 の diff サマリ（`summarizeDiff`）は変更不要。characterStates 単位の JSON stringify 比較で `updatedCharacters` に載る。

**受け入れ条件**

- 抽出で「A は B の癖に気づいた」と読める場面があれば、A の characterStates.knowledge に該当文が追加される。
- 差分で `removeKnowledge: ['Aは……に気づいた']` を渡すと、A の既存 knowledge から完全一致で該当項目が消える（誤抽出の巻き戻し）。
- 1回の差分で knowledge に4件以上渡されても、追加は3件までに切り詰められる。
- 既存 knowledge が上限12件を超えた場合、古いものから削除される（先頭切り落とし）。
- 削除対象文字列と既存項目の照合は `normalizeComparableText` を通す。連続空白の差・全角/半角差・大小文字差は吸収されるが、**空白の有無自体は吸収されない**（`removeKnowledge` に指定する文字列は既存の空白位置と一致させる必要がある）。
- 生成プロンプトの【人物の情報状態】に knowledge が反映される。
- **ユニットテスト**（`storyStateService.test.ts` に追加）:
  - knowledge 追加（1件 / 3件 / 4件超）。
  - removeKnowledge による削除（完全一致 / 全角半角差 / 大小文字差の吸収、空白位置ずれは非マッチ）。
  - 追加と削除が同時に来た場合の適用順（先に削除、次に追加）。
  - 上限12件超過時の古いものからの切り捨て。

**懸念点と対応**

- LLM がイベントと重複した内容を knowledge にも書いてしまう可能性。プロンプトの指示で明示的に抑制するが、多少の重複は生成プロンプトの `dedupeText`（[promptBuilder.ts:503](../../src/server/prompts/promptBuilder.ts:503)）で吸収される（同じ文言なら表示上は1回）。
- knowledge の細粒度が高すぎると【人物の情報状態】の known 側が肥大化する。known 側は 1 人物あたり 6 件表示（`slice(0, 6)`、[promptBuilder.ts:503](../../src/server/prompts/promptBuilder.ts:503)）で頭打ちになるが、この 6 件は knowledge と importantEvents.knownBy 由来イベントの合算後に切り詰められる。knowledge が優勢になり過去イベントが押し出される場合は、混合ソートの優先度調整（例: knowledge を後段に置く）を将来検討する。今回は既存の実装順（先に knowledge、後にイベント）を維持する。
- 誤抽出の除去手段（`removeKnowledge` 経由の差分パッチ）は K8 の物語状態編集画面のUI経由で叩けるようにする必要がある。K8 の実装は既存済みのため、knowledge 編集の入力欄はすでに存在（`characterStates.knowledge` の直接編集フォーム）。本トラックはあくまで抽出LLMからも書けるようにする話。

## 横断的な注意

### 抽出LLMへの負荷（実務上の最大リスク）

トラック 1A / 2A / 2B は全て `buildUpdatePrompt` に追加指示を入れ、応答 JSON にも actor/recipient/addUnknownBy/removeUnknownBy/knowledge/removeKnowledge が加わる。特に 2A は反転により全イベントに `addUnknownBy` が大量付与され、応答が膨らむ。出力上限に達すると `storyStateService.ts:98-101` で部分適用ではなく throw されるため、そのシーンの状態更新が丸ごとエラーになる（ユーザーに見える失敗、かつ状態が未更新のまま次シーンが積まれると以降の整合がむしろ悪化する）。

このリスクは「据え置き + 受け入れ時に確認」では小さくないので、以下を**実装時の必須手当**として組み込む:

1. **初回 outputLength を先んじて 6000 に引き上げる**（`STORY_STATE_OUTPUT_LENGTH = 4000` → `6000`）。retry の `STORY_STATE_RETRY_OUTPUT_LENGTH` は `9000` にする（現行 6000 の 1.5 倍を維持）。定数変更のみで、コスト増分は最悪 1.5 倍にとどまる。
2. **抽出プロンプトの `addUnknownBy` 対象範囲をコードでも下限保証**: プロンプトの指示だけでは弱いので、`addUnknownBy` の対象を実装 2 の指示文で「role が protagonist / deuteragonist の人物、および当該場面直前の characterStates に登場する人物」に厳格化する（既に指示に含まれているが、Phase C 実装時にプロンプト文言を再チェックし、脇役全員が addUnknownBy に混入しない状態を確認する）。
3. **length 再試行率を受け入れ条件に格上げ**: Phase C（2A）と Phase D（2B）を続けて入れ終わったあと、受け入れテスト用プロジェクトで 10 シーン以上の連続採用を行い、`finishReason === 'length'` による retry 発生率が 20% 未満であることを確認する。超える場合は初回 outputLength を追加で引き上げるか、抽出指示を絞る。数値目標は暫定で 20%、実測に応じて調整する。

Phase C / D は同じ抽出プロンプトを触るため、実装は連続して入れつつ、この length 再試行率の確認は **C・D 両方完了後に一度だけ実施**する。

### K7 diff 履歴との整合

いずれのトラックも K7 の diff 記録（`appendStoryStateDiff` / `revertLatestStoryStateDiff`）と自然に噛み合う。beforeState は StoryState 丸ごとのスナップショットで保存されるため、actor/recipient/knowledge の追加も含めた完全な巻き戻しが可能。追加実装は不要。

### K8 物語状態編集画面（本設計書のスコープ外）

現行の物語状態編集は生JSON編集（`WorkSettingsTab.tsx` の物語状態タブ、[WorkSettingsTab.tsx:1809](../../src/client/components/WorkSettingsTab.tsx:1809)）で、イベント単位の編集フォーム・人物選択セレクトボックスは存在しない。

本設計書では**クライアントは変更しない**。新フィールド（actor / recipient / knowledge の追加削除 / explicitlyUnknownBy）はいずれも生JSON編集で操作可能。UI で個別編集できるようにする改修は別トラック（「未対応の残課題」参照）。

生JSON編集で操作する際の注意:
- `actor` / `recipient` は characterId 文字列または `null` を指定する。存在しない ID は次回保存時にサーバー側で null に落ちる。
- `explicitlyUnknownBy` は配列を直接編集可能（クライアント→サーバーの JSON PUT は差分パッチではなく全体置換のため、`addUnknownBy` / `removeUnknownBy` 経由の追加削除操作は不要）。
- `characterStates[].knowledge` は文字列配列を直接編集可能（同上）。

### ロールプレイモードへの影響

`RoleplayContextSnapshot` は StoryState を含まないため、本設計書の変更はロールプレイ会話には直接影響しない。ただし、setup モード（`SetupDraft`）からロールプレイセッションが作成される経路で `characters[].currentState` が使われる（[roleplayPromptBuilder.ts:200](../../src/server/services/roleplayPromptBuilder.ts:200)）。本設計書は Character 型を触らないため、こちらへの波及もない。

### プロバイダ差分

- Gemini / OpenAI / xAI / OpenRouter / DeepSeek すべて JSON 応答対応済み（`responseMimeType: 'application/json'`）。追加フィールドの取り扱いに差は出ない想定。
- 弱いモデル（無料枠のOpenRouter）で actor/recipient の取りこぼしが多い場合、プロンプト側で明示例（1つ）を追加することで安定化する余地がある。実装時は例を入れずに素の指示で試し、受け入れテストで判断する。

## 受け入れテスト（統合）

以下の手順で3症状の改善が体感できることを確認する。

**検証範囲の宣言**: 本テストは**新規プロジェクト + 新規シーンのみ**を対象とする。既存プロジェクトへの遡及適用は本設計のスコープ外であり、旧イベントは `explicitlyUnknownBy` がほぼ空のまま残るため、既存プロジェクトでは 2A/2B の効きが部分的になる。これは仕様上の受容判断で、本受け入れテストの検証対象ではない旨をここに明記する（後日「効いていない」との誤診を防ぐため）。

1. テスト用プロジェクトを新規作成し、setup で人物3〜4名（protagonist / deuteragonist / supporting）を登録する。
2. 第1話〜第3話まで生成・採用する。第2話には「A と B が2人だけの秘密の会話」を、第3話に「その秘密が絡む場面（C は同席せず）」を含める。
3. 物語状態タブの生JSON表示で以下を確認:
   - 第2話の秘密が `importantEvents` に載り、`actor: A`, `recipient: B` が入る（症状1）。
   - 同イベントの `explicitlyUnknownBy` に C が含まれる（症状2A）。
   - C の characterStates.knowledge には該当情報が入らない（症状2B が誤動作しない）。
4. 「第4話は C の視点で」と wish に書いて生成:
   - 【使わない表現】に登録済みNGが【今回の希望】直前に出ることをプロンプトログ（`generation-prompts/gen-*.md`）で確認（症状3）。
   - 生成本文で C が第2話の秘密に触れないことを目視確認（症状2A/2B、**過剰抑制なし**の裏側の症状）。
   - 生成本文で秘密の主体（A → B）が反転していないことを目視確認（症状1A）。
5. **消しすぎ検証**（症状2A/2B の逆方向確認）: A 視点で第5話を「A が第2話の秘密を反芻する内面描写」を含む wish で生成し、A が自分の秘密を語れる（＝ `explicitlyUnknownBy` の強化制約が正当な描写を殺していない）ことを目視確認する。加えて A の characterStates.knowledge に第2話由来の項目があれば、それを踏まえた自然な連続性が本文に出るかを確認。もし「A 視点なのに秘密に触れない不自然な地の文」が観測された場合、以下のいずれかで対処する:
   - 該当イベントの `explicitlyUnknownBy` から A を `removeUnknownBy` で除去し、K7 revert → 再抽出。
   - `renderCharacterKnowledgeState` の否定制約の強化ワードを弱める（実装 5 の変更後文言を暫定的に戻す）。
6. 手動で NG 表現を 2 件登録して同じシーンを再生成し、両方ともプロンプト末尾に出ることを確認。
7. K7 履歴から第2話の抽出結果を1度巻き戻し、再抽出して actor/recipient/explicitlyUnknownBy がおおむね同一の値で復元されることを確認（抽出の再現性チェック）。
8. 追加確認: 「第3話で C が第2話の秘密を偶然立ち聞きする」場面を作って採用し、抽出後に C が該当イベントの `knownBy` に加わり `explicitlyUnknownBy` から自動除去されることを確認（時間軸に沿った状態遷移の検証）。
9. **length 再試行率の確認**（横断的な注意「抽出LLMへの負荷」3 で格上げした条件）: 手順1〜8を通じて連続 10 シーン以上を採用したうえで、`processedGenerationIds` に載った件数と、実際の抽出試行回数（サーバーログ）から `finishReason === 'length'` の retry 発生率を算出し、20% 未満であることを確認する。超える場合は `STORY_STATE_OUTPUT_LENGTH` を追加で引き上げる。

## 未対応の残課題（別トラックへ）

以下は本設計書のスコープ外。別トラックで対応する。

- NG表現の事後検証と自動再生成（レビューの3B）。
- NG表現マッチングの正規化強化（レビューの3C）。
- NG登録件数上限の可視化・並び替えUI（レビューの3D）。
- 視点人物検出の拡張（「目線で」「から見て」対応、レビューの1B）。
- 参考資料の視点フィルタ（visibility メタ、レビューの2C）。
- 直近本文の視点フィルタ（レビューの2E、副作用大のため要別途調査）。
- 要約プロンプトの主体保存指示（レビューの1C）。
- **物語状態編集画面（K8）の UI 拡張**: 現状の生JSON編集を、イベント単位の編集フォーム＋人物選択セレクトボックス（actor / recipient / explicitlyUnknownBy）＋ knowledge の項目単位追加削除UI に置き換える。本設計書はバックエンド改善のみで、UI 追加は別トラックとする。
- **既存プロジェクトの遡及再抽出手段**: 本設計書適用後、既存プロジェクトへの効き方は「以降の新規シーンから徐々に」であり、過去の全採用シーンを再抽出する自動化された手段は用意しない。「全シーン再抽出」を安全に行うためには、抽出LLMの温度・出力上限・逆順マージ耐性・K7 diff 履歴のトリミング方針まで含めた別設計が必要。
