# 人物スキーマ再設計 設計書

作成日: 2026-07-19
対象: 人物(Character)の固定フィールド `want` / `fear` を廃止し、可変軸 `traits`(ラベル付き自由記述の配列)へ置き換える。`secrets` は JSON プロパティ名を維持したままラベルと描画指示を「見せない面」寄りに書き換える。

## 0. 設計レビュー結果

現行コードとの突合で見つかった重要事項を、本書の修正方針として先に固定する。

| 優先度 | 指摘 | 修正方針 |
|---|---|---|
| 高 | `storageService.readCharacters` は現状 `normalizeCharacterForStorage` を呼ばないため、「読み込み時に旧データを移行する」という元案は成立しない | 正規化を循環依存のない共有モジュールへ移し、読み込み境界と全書き込み境界の双方から呼ぶ |
| 高 | `Character` から `want` / `fear` を削除した後も同じ型の引数から旧キーを読む設計は型として矛盾する | 旧保存形式を表す `LegacyCharacterInput` を明示し、互換入力だけが旧キーを扱う |
| 高 | 既存 `traits` と旧 `want` / `fear` が混在した場合、旧値を先頭へ置いて4件で切ると明示的な新データを失う | 新形式の `traits` を優先し、空き枠にだけ旧値を補う |
| 高 | SetupSession はディスクへ保存され再開できるため、「一時データなので移行不要」ではない | `normalizeSetupDraft` を保存済みセッションの読み込みにも適用し、旧 draft と旧 patch を受け入れる |
| 中 | `secrets` を独立フィールドとして残す一方、commit 例では trait の「見せない面」へ入れており責務が二重化していた | 正式な出力は常に `secrets`。trait からの昇格は壊れた/旧LLM出力への防御的フォールバックだけにする |
| 中 | SetupWorkspace は現状 `want` / `fear` / `secret` を編集表示しておらず、「該当箇所があれば変更」では新 traits が不可視になる | 既存・新規人物の両フォームへ `traits` / `secrets` 編集を明示的に追加する |
| 中 | 「読者が人物を好きになる理由」の必須化は、固定軸を外す目的に反して悪役や不快さが魅力の人物まで同じ型へ寄せる | 「読者の関心を引く点（魅力・可笑しみ・違和感など）」へ緩和し、必要な人物だけに求める |

## 1. 目的と設計原則

現行の Character は `description / speechStyle / relationshipNotes / want / fear / secrets` の固定フィールドで人物を定義している。これは相談モードの提案指示「何に揺れるか・何を望むか・何を恐れるか・何を隠しているか」と組み合わさって、実質的にすべての作品で同じ3軸(望み・恐れ・秘密)を人物に付与する構造になっている。結果として:

- 生成される作品の毛色が偏る。全人物が「欲求と恐れと秘密を抱えた陰のある存在」として描かれ、コメディ・日常もの・爽快な冒険譚でも不穏な底流が混ざる。
- 相談モードが毎回律儀に3軸を埋めようとするため、軽い役にも過剰な内面が付く。
- 秘密フィールドは属性ではなくプロット装置なので、常設すると全作品が「隠された真実の話」の重力に引かれる。

新設計の原則:

1. **物語を動かす軸はLLMが物語ごとに決める**。固定スキーマで軸を強制しない。
2. **機構の部品は固定フィールドのまま残す**(name, aliases, role, description, speechStyle, greeting, dialogueExamples, currentState, relationshipNotes)。これらは識別・UI・生成処理が意味を解釈するため、自由記述に溶かさない。
3. **意味付きの描画指示は保持する**。`secrets` の「滲むように描き、軽々に明かさない」型の指示は、フィールドを残す理由になる。
4. **未設定フィールドはプロンプトに出さない**。プリセット再設計と同じ原則を人物にも適用する。
5. **相談モードの心得は残す**。プロフィール羅列を避け、揺れと関係性を中心にする方針は、スキーマではなくプロンプトの指示側で維持する。

## 2. 新しい Character スキーマ

```ts
export interface CharacterTrait {
  label: string;   // 「望み」「こだわり」「意地の張り方」など、LLM が物語ごとに決める
  text: string;    // 自由記述
}

export interface Character {
  characterId: CharacterId;
  name: string;
  aliases?: string[];
  role: CharacterRole;
  description: string;
  speechStyle?: string;
  relationshipNotes?: string;

  // 意味付きフィールド(維持・描画指示付き)
  secrets?: string;              // ラベル「見せない面」に変更

  // 新設: 物語を動かす可変軸。最大4件。
  traits?: CharacterTrait[];

  // 機構の部品(維持)
  currentState?: string;
  greeting?: string;             // ロールプレイ用
  dialogueExamples?: string[];   // ロールプレイ用
}
```

**削除するフィールド**: `want`, `fear`。

**`secrets` の扱い**: JSON プロパティ名は維持する(既存データとの互換のため変更しない)。日本語ラベルと描画指示だけを「見せない面」寄りに書き換える(§5.1)。

**`traits` の制約**:

- 各 trait の `label` は 12 文字以内、`text` は 200 文字以内(共有正規化関数で丸める)。
- 配列は最大 4 件(超過分は先頭から採用して切り落とす)。
- `label` / `text` のどちらかが空文字の trait は落とす。
- `label` は改行・タブを半角スペースへ畳み、前後空白を除去する。プロンプトの行構造を壊すラベルは保存しない。
- 同一 `label` の重複は、正規化後の完全一致で先勝ちにする。
- setup の正式な LLM 出力では trait ラベル「見せない面」「秘密」を使わず、`secrets` へ出力する。これらの trait は setup parser だけが逸脱出力として救済する(§6.2)。

### 3.2 `src/shared/characterSchema.ts` — 共有正規化(新設)

現行の `normalizeCharacterForStorage` は `projectService.ts` にあり、`projectService` 自身が `storageService` を import している。このまま `storageService.readCharacters` から呼ぶと循環依存になる。人物スキーマの純粋な正規化を `src/shared/characterSchema.ts` へ分離し、サーバーの読み書き経路と各 LLM パーサーから共有する。

```ts
export type LegacyCharacterInput = Omit<Character, 'traits'> & {
  traits?: unknown;
  want?: unknown;
  fear?: unknown;
};

export function normalizeCharacterTraits(value: unknown): CharacterTrait[];
export function normalizeCharacterForStorage(value: LegacyCharacterInput): Character;
export function normalizeCharactersForStorage(
  value: LegacyCharacterInput[]
): Character[];
```

`normalizeCharacterTraits` の仕様:

- 配列でなければ空配列。
- 各要素は `{ label, text }` のオブジェクト以外は落とす。
- `label` は改行・タブ・連続空白を半角スペース1つへ畳んで trim する。`text` は改行コードを `\n` に統一して trim する。
- 正規化後に `label` または `text` が空なら落とす。
- label は 12 文字、text は 200 文字で切る。
- 同一 label の重複は先勝ち。
- 4件を超えたら先頭4件を採用する。
- 結果が空なら Character の出力に `traits` プロパティを含めない。

`normalizeCharacterForStorage` は既存の greeting / dialogueExamples 正規化も引き継ぎ、`want` / `fear` は §4.1 の規則で traits へ寄せる。`projectService.ts` は既存 import 元を壊さないよう、この共有関数を import して再 export してよい。

### 3.3 `src/server/services/storageService.ts` — 読み込み境界

[storageService.ts:249](../../../src/server/services/storageService.ts#L249) の `readCharacters` は現在 JSON を型キャストして返すだけであり、旧データを変換しない。次のように共有正規化を必ず通す:

```ts
export async function readCharacters(projectId: string): Promise<Character[]> {
  const data = await readJsonFile<LegacyCharacterInput[]>(charactersJsonPath(projectId));
  return normalizeCharactersForStorage(Array.isArray(data) ? data : []);
}
```

これにより GET、生成、ロールプレイ、refine、複製のすべてが正規化済み Character を受け取る。読み込みはメモリ上の変換だけとし、GET だけでファイルを書き換えない。

### 3.4 `src/server/routes/settings.ts` — isCharacterInput

[settings.ts:280](../../../src/server/routes/settings.ts#L280) は移行期間の互換性のため `optionalString(value.want)` / `optionalString(value.fear)` を残しつつ、`optionalCharacterTraits(value.traits)` を追加する。型ガードの戻り値は新形式の `Character` ではなく、互換入力を表す `LegacyCharacterInput` に変える。

```ts
function optionalCharacterTraits(value: unknown): boolean {
  if (value === undefined) return true;
  if (!Array.isArray(value)) return false;
  return value.every(
    (item) =>
      isRecord(item) &&
      typeof item.label === 'string' &&
      typeof item.text === 'string'
  );
}
```

**互換のため、旧 `want` / `fear` キーが文字列として混ざっていても検証を落とさない**。新しい出力は `traits` に寄せるが、旧クライアント・旧保存データからの入力だけは共有正規化で変換する。数値やオブジェクトの旧キーまで黙って受け入れない。

### 3.5 refine 系サービスのフィールド参照

- `src/server/services/refineScanService.ts`([refineScanService.ts:306](../../../src/server/services/refineScanService.ts#L306), [:506](../../../src/server/services/refineScanService.ts#L506)): static input hash と LLM 入力の両方へ traits を登録順で含める。並び替えでもプロンプトが変わるため、hash 側でも traits は sort しない。
- `src/server/services/refineChatService.ts`([refineChatService.ts:663](../../../src/server/services/refineChatService.ts#L663), [:795](../../../src/server/services/refineChatService.ts#L795), [:822](../../../src/server/services/refineChatService.ts#L822)): character-add / character-update の双方で traits を扱う。string 配列など壊れた形は `normalizeCharacterTraits` で空配列へ丸める。
- character-update で `traits` が存在する場合は**配列全体の置換**とする。`[]` は全削除、キー自体が無い場合は現状維持、非配列はそのフィールドだけ無視する。この規則により LLM パッチでも削除が可能になる。
- trait の text に改行がある場合、LLM 向け表示では2行目以降も継続行としてインデントする。データを壊さず、プロンプト上の別フィールドに見えないようにする。

## 4. 旧データの移行(読み込み境界での正規化)

一括変換バッチは作らない。**共有正規化を読み込み境界と書き込み境界の双方で実行し、旧フィールドを新フィールドへ変換する**。変換は冪等であり、何度通しても結果が変わらないことを必須とする。

### 4.1 変換ルール

`normalizeCharacterForStorage` では、まず明示的な `traits` を正規化し、残り枠にだけ旧 `want` / `fear` を補う:

```
入力 { want: "承認されたい", fear: "見捨てられること" }
  ↓
出力 { traits: [
  { label: "望み", text: "承認されたい" },
  { label: "恐れ", text: "見捨てられること" }
] }
```

- 新形式の `traits` を常に優先する。既に「望み」「恐れ」があれば対応する旧値は追加しない。
- 新形式が4件あれば旧値は追加しない。3件なら `want`、空きが残れば `fear` の順に補う。明示的な新データを旧データが押し出してはならない。
- 新形式が無い純粋な旧データでは「望み」「恐れ」の順になる。
- 旧値は文字列かつ trim 後に非空の場合だけ移行する。
- 入力の `want` / `fear` プロパティ自体は出力に含めない。
- 変換後の `traits` だけを再入力しても同じ結果になる(冪等性)。

### 4.2 SetupDraftCharacter の移行

`SetupSession` の draft は commit 前でもディスクへ保存され、再開される。`want` / `fear` / `secret` は新しい draft モデルでは `traits` / `secrets` に寄せる一方、読み込みと patch 適用では旧キーも受け入れる。

- `setupDraftPatchService.normalizeCharacter` で新 traits を優先し、残り枠へ旧 want/fear を補う。
- 旧 `secret` は、新 `secrets` が無い場合だけ `secrets` へ移す。
- `lockedFields` の旧 `want` / `fear` は `traits`、旧 `secret` は `secrets` へ読み替える。既存ユーザーの固定指定を失わない。
- `setupSessionService.normalizeStoredSession` は scenarioSeeds だけでなく `normalizeSetupDraft(session.draft)` を呼び、保存済みセッションを返す前に人物も移行する。
- schemaVersion 1 は上記正規化後に 2 へ上げる。schemaVersion が未知の値なら黙って解釈しない。個別取得/更新は既存の setup エラー形式で読込不能を返し、一覧取得ではそのセッションだけを除外して他の一覧を壊さない。
- 読み込み時の変換はメモリ上だけでよい。次のセッション更新で新形式として保存される。

### 4.3 適用箇所

共有正規化の適用境界:

- 読み込み: `storageService.readCharacters`
- 新規作成・複製: `projectService.createProject` / `copySettings`
- 人物設定保存: `PUT /projects/:id/characters`
- setup commit: `setupCommitService` の出力と `createProject` の保存前
- refine patch 適用: `refineChatService`

`refineScanService` は読み取り専用であり更新経路には含めない。**ディスクへの書き戻しは次回保存時でよい**。既存プロジェクトを開いた時点では、GET が正規化済みデータを返すため UI 表示は新形式になる。

### 4.4 ダウングレード対策

新形式を保存した後に旧アプリへ戻すと、旧アプリは traits を描画できず、さらに保存時の既知フィールドコピーで traits を落とす可能性がある。前方移行だけでなく、このデータ損失経路も扱う。

- `storageService.writeCharacters` が既存 `characters.json` を上書きする直前に、旧 want/fear を含むかを raw JSON で確認する。
- 旧形式を検出し、`characters.pre-traits-v1.json` がまだ無い場合だけ、既存ファイルを同じプロジェクトディレクトリへ退避する。
- バックアップは一度だけ作り、以後上書きしない。新規プロジェクトでは作らない。
- 通常の読み込み対象は従来どおり `characters.json` だけとし、バックアップは復旧用に限定する。
- リリースノートに「旧バージョンへ戻す場合はバックアップの手動復元が必要」と記載する。

## 5. プロンプト組み立ての変更

### 5.1 `src/server/prompts/promptBuilder.ts` — renderCharacters

[promptBuilder.ts:200](../../../src/server/prompts/promptBuilder.ts#L200) を書き換える。

- `if (c.want) parts.push('  欲求: ...')` と `if (c.fear) parts.push('  恐れ: ...')` を削除。
- 代わりに `c.traits` を回して trait ごとに `  ${label}: ${text}` を出す。
- text が複数行なら2行目以降を4スペースでインデントする。
- `secrets` の描画指示を書き換える:

```ts
if (c.secrets) {
  parts.push(
    `  見せない面: ${c.secrets}（普段の言動には出さない。ふとした瞬間や限られた相手にだけ滲むように描き、地の文で軽々に説明しないこと）`
  );
}
```

**指示文の意図**: 隠された事実(秘密)はもちろん、外面と内面のずれ・建前・普段見せない一面までを含めて「滲ませる」型の描写を求める。これによりシリアスな秘密として使ってもコメディのギャップとして使っても機能する。

出力順は: `名前 → 呼び名 → 概要 → 口調 → 見せない面 → traits(登録順)` とする。traits を最後にまとめて出すのは、機構情報の後に「その物語ならではの軸」が並ぶ読み順を意図している。

[roleplayPromptBuilder.ts:183](../../../src/server/services/roleplayPromptBuilder.ts#L183) を書き換える。

- `push(lines, '望むもの (want)', character.want)` と `push(lines, '恐れ (fear)', character.fear)` を削除。
- `character.traits` を回して `push(lines, trait.label, trait.text)` を並べる。
- `secrets` のブロックを書き換える:

```ts

```

**指示文の意図**: ロールプレイでは秘密を絶対不可侵にすると会話が発展しないため、「自分からは明かさないが、親密度や状況に応じて滲んでよい」と解禁条件を含める。これは元の指示より緩いが、ロールプレイという文脈では自然。

## 6. 相談モード(setup)の変更

### 6.1 `src/server/services/setupPromptBuilder.ts`

#### 6.1.1 系統プロンプトの心得を書き換える

[setupPromptBuilder.ts:45](../../../src/server/services/setupPromptBuilder.ts#L45) を書き換える。novel 相談のシステム指示から:

```
- 人物はプロフィールの羅列にせず、何に揺れるか・何を望むか・何を恐れるか・何を隠しているかを中心に提案してください。
```

を次に置き換える:

```
- 人物はプロフィールの羅列にせず、その物語を動かすのに必要な軸を2〜4個、traits に自由なラベルで入れてください。
- 軸のラベルは物語の毛色に合わせて選んでください。恋愛なら「意地の張り方」、日常なら「こだわり」、コメディなら「癖」、シリアスなら「望み」「恐れ」「動機」など、決まった枠に縛られないでください。
- 「見せない面」(secrets)は、隠している事実だけでなく、外では見せない一面や建前と本音のずれも含めて構いません。どの人物にも付ける必要はありません。
- 主要人物の description には、必要なら読者の関心を引く点（魅力、可笑しみ、違和感、危うさなど）を含めてください。全員を好人物に寄せないでください。
```

roleplay 相談側も固定の「望み・恐れ・秘密」を必須にしない。継続会話を動かす軸を2〜4件選ばせ、例として「会話で望むこと」「触れられたくない話題」「距離の詰め方」「意外な弱点」などを示す。`greeting` / `dialogueExamples` は機構フィールドとして従来どおり必須候補にする。

### 6.2 `src/server/services/setupCommitService.ts` — normalizeCharacter

```ts
// NOTE: 正式な出力は value.secrets。予約ラベルの昇格は旧/逸脱出力の救済だけ。
```

`extractSecretsFromTraits` は4件上限を適用する**前**の raw 配列から、`label` が「見せない面」「秘密」の項目をすべて除き、その先頭の text を fallback として返す。明示的な `value.secrets` があれば常にそちらを優先する。予約ラベルが traits の件数枠を消費したり、通常 trait として残って描画指示が外れたりする状態を作らない。共有正規化へ旧 want/fear も渡し、direct commit の旧LLM応答を同じ優先順位で救済する。

- `want: character.want, fear: character.fear` を削除。
- 代わりに `traits: character.traits` を追加(SetupDraftCharacter に traits フィールドが増える)。
- 旧 draft(want/fear を持つもの)への互換は、SetupDraftCharacter の入力正規化側(session 更新経路)で行う。§6.3 参照。

### 6.3 `src/server/services/setupDraftPatchService.ts` / `setupSessionService.ts`

実際の charactersAdd / charactersUpdate 適用責務は `setupDraftPatchService.ts` にある。

- add: `normalizeCharacter` で traits / secrets を正規化する。旧 want/fear は新 traits の空き枠だけに補い、旧 secret は新 secrets が無い場合だけ使う。
- update: `traits` キーがあれば配列全体を置換し、`[]` で削除できるようにする。キー無しは維持、非配列は無視する。
- update: `secrets` キーがあれば文字列を trim して置換する。空文字は削除として扱う。旧 `secret` は `secrets` が無い場合だけ同じ規則で扱う。
- update: 新 traits と旧 want/fear が同時に来た場合は新 traits を優先する。旧値だけの場合は既存 traits の空き枠へ補う。
- lock: `traits` は `lockedFields: ['traits']`、secrets は `['secrets']` で保護する。旧 lock 名は §4.2 の読み込み移行で寄せる。
- session read: `setupSessionService.normalizeStoredSession` から `normalizeSetupDraft` を呼ぶ。

`normalizeSetupDraft(value: unknown)` と `applySetupDraftPatch(... patch: unknown)` が互換境界であり、内部では `LegacySetupDraftCharacterInput` / `LegacySetupDraftPatchInput` として検査する。新しい `SetupDraftCharacter` へ旧キーを戻して帳尻を合わせてはならない。

### 6.4 相談モードの JSON 出力例の見直し

相談モードの出力する JSON 例に含まれる旧キーは、**出力例からは削除する**必要がある。grep では `"want"` / `"fear"` / `"secret"` を含む「出力例・テンプレート」だけを確認し、入力互換の parser / normalize 分岐は残すこと。

### 新規テスト

1. **normalizeCharacterForStorage 移行**:
   - 旧 `{want, fear}` 入力 → `traits: [{望み}, {恐れ}]` 出力。
   - 旧 `{want, traits:[{望み,別}]}` 入力 → traits の望みが残り、旧 want は捨てられる(先勝ち)。
   - traits が 5 件以上 → 4 件に切られる。
   - trait の label が 13 文字以上 → 12 文字に切られる。
   - trait の text が 201 文字以上 → 200 文字に切られる。
   - 空 label または空 text の trait → 落とされる。
   - 同一 label の重複 → 先勝ち。
   - `traits: []` や `traits: undefined` → 出力に traits フィールドが含まれない。
   - label 内の改行/タブ → 単一行へ正規化される。
   - 新 traits が4件ある混在入力 → 旧 want/fear が明示 traits を押し出さない。
   - 正規化済み出力を再入力 → 同じ出力になる(冪等性)。

2. **readCharacters 移行**:
   - ディスク上の旧 `{want, fear}` → GET/各サービスでは traits として見える。
   - 読み込みだけでは characters.json の内容・mtime が変わらない。
   - 次回 PUT 後の characters.json には want/fear が残らない。
   - 初回の移行保存前に `characters.pre-traits-v1.json` が作られ、2回目以降も上書きされない。
   - 新規形式だけのプロジェクトではバックアップを作らない。

3. **renderCharacters**:
   - traits が登録順で `  <label>: <text>` の並びで出力される。
   - 出力順が `名前 → 呼び名 → 概要 → 口調 → 見せない面 → traits` になる。
   - secrets の描画指示文が新しい文言に変わる。
   - traits なし・secrets なしの人物では該当行が出ない。

4. **buildPersonaCard(roleplay)**:
   - traits が `- <label>: <text>` で並ぶ。
   - 「見せない面」の指示文が緩和版になっている。

5. **setup / setupCommit の移行**:
   - LLM が traits に「見せない面」ラベルで返した場合、secrets に振り分けられ traits からは消える。
   - LLM が旧 `want` / `fear` キーだけを返した場合、空き枠に「望み」「恐れ」で追加される(§6.3 の互換)。
   - traits が壊れた形(string 配列など)で返っても落ちず、`traits: undefined` になる。
   - 保存済み旧 SetupSession を再開すると traits / secrets へ変換される。
   - schemaVersion 1 の SetupSession はメモリ上で 2 へ移行され、次回更新で version 2 として保存される。
   - 未知の schemaVersion は個別取得でエラーになり、一覧では当該セッションだけが除外される。
   - `lockedFields: ['want', 'secret']` が `['traits', 'secrets']` へ変換される。
   - update の `traits: []` / `secrets: ''` で値を削除できる。

6. **settings API バリデーション**:
   - 旧 `{want, fear}` を含む Character PUT を受け付ける(検証を落とさない)。
   - 新 `{traits: [{label, text}]}` を受け付ける。
   - `traits` が非配列や壊れた形なら 400 を返す(§3.4 の optionalCharacterTraits)。
   - 応答は正規化済みであり、旧キーと空 trait を含まない。

7. **refine**:
   - traits の追加・置換・全削除が patch 適用後の characters.json に反映される。
   - traits の内容または順序が変わると static input hash が変わる。
   - character-update / character-add の traits が RefineChatPanel で `label: text` として表示される。

8. **UI とサーバー正規化の同期**:
   - WorkSettingsTab から空 label/text を含む draft を保存し、API が不完全行を除いた配列を返した場合、画面 state が API 応答へ置き換わる。
   - 保存直後に再読込しなくても、閲覧モードの表示と characters.json が一致する。
