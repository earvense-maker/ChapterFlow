# 設定レイヤー分離（不変設定と進行状態の区別）設計書

## 0. 背景

現行の物語設定は、仕組み上は二層に分かれている。

- **静的層（ユーザーが書き、自動更新されない）**: `Project.coreConcept` / `world.md` / `Character` の各フィールド / 参考資料（knowledge） / `Memory`
- **動的層（シーン採用のたびに自動更新される）**: `StoryState`（currentSituation / characterStates / importantEvents / openThreads / clock）

さらに生成プロンプトには信頼順位「採用済み本文 ＞ 派生データ ＞ 作品設定・参考資料」が明文化されている（`promptBuilder.renderWishSection`）。しかし調査の結果、この二層の**境界が崩れている箇所**が確認された。

1. **`Character.currentState` の二重出力（最重要）**: `promptBuilder.renderCurrentState` は、静的な `Character.currentState` と動的な `storyState.characterStates` の両方を**同じ【人物の現在状態】セクションに並記**する。静的側はセットアップ時に「開始時点の状態」として生成された後、自動更新されない（クライアントに編集UIも存在しない）。物語が進むほど、開始時点の状態と最新の状態が**同格の事実として同一セクション内で矛盾**する。信頼順位はセクション間の原則なので、ここには効かない。
2. **静的セクションに時点の明示がない**: 【作品設定】【関係性設定】が「いつの時点の記述か」を示さないため、物語中で変わった事柄（関係の変化など）と食い違ったとき、LLMは同格の情報として扱う。
3. **`world.md` の不変/初期状況の混在**: 魔法体系のような不変の法則と、「王国は平和である」のような開始時点の状況が区別なく混在する。後者が本文中で変わっても world.md は毎回古い状態を主張し、変化を書いた本文が直近文脈から流れ落ちた後は、StoryState が捕捉していない限り古い設定への引き戻しが起きうる。
4. **設定の陳腐化に気づく導線がない**: refine scan（矛盾検出）は横断チェックができるが手動ボタンのみで、本文がどれだけ進んだら実行すべきかのシグナルがない。

本書はこの4点への対処を、コスト順に L1〜L5 の項目として設計する。

対象コード:

- `src/server/prompts/promptBuilder.ts`（renderCurrentState / renderCharacterKnowledgeState / 【作品設定】組み立て）
- `src/server/services/setupPromptBuilder.ts`（worldText 生成指示）
- `src/server/services/storyStateService.ts`（人物ヒントの currentState）
- `src/server/services/refineScanService.ts`（scan プロンプトの currentState ラベルと陳腐化検出指示）
- `src/server/services/refineChatService.ts`（人物レンダリングの currentState ラベル）
- `src/server/utils/characterStateMatching.ts`（人物と StoryState の副作用なし照合）
- `src/client/components/WorkSettingsTab.tsx`（人物編集フォーム、設定レビューカードのナッジ）
- `src/server/routes/refine.ts` / `src/client/clientApi.ts` / `src/shared/types.ts`（設定レビュー状態API）
- `tests/unit/promptBuilder.test.ts` ほか対応テスト

前提となる現状仕様（変更しないもの）:

- 信頼順位の一般則（`renderWishSection` の①〜④と情報源の信頼順）は維持する。
- StoryState 更新パイプライン（`updateStoryStateFromAcceptedScene` → `mergeStoryState` → diff 記録）は変更しない。
- `WorkSettingsTab` の StoryState JSON 編集・diff 取り消しUIは既存のまま。
- `roleplayPromptBuilder` は `Character.currentState` を「会話開始時点の状態」として使用しており、**既に本書の再定義と同じ意味**。変更不要。
- `setupPromptBuilder` のコミット指示も currentState を「開始時点の状態」として生成させており（novel: `'開始時点の状態'`、roleplay: `'会話開始時点の状態'`）、生成側の意味は既に「初期状態」である。ずれているのは**消費側（promptBuilder）のレンダリングだけ**。

互換性方針:

- **スキーマ変更なし**。`Character.currentState` のフィールド名は変えない（保存データ・API・refine patch の `CharacterFieldPatch.currentState` に波及するリネームはコストに見合わない）。意味の再定義は、プロンプト上のラベルとUI上のラベルで表現する。
- world.md は自由記述のまま。L4 の見出し規約は**あれば使う**オプトインで、規約のない既存プロジェクトの挙動は変えない。
- Character / StoryState / world.md のドメインスキーマは変更しない。
- L5 だけは、走査がどの StoryState diff までを確認したかを示す任意メタデータを `refineScan.json` に追加する。既存キャッシュでは未設定を許容し、読み込み時に保守的に扱う。バックログ数自体は永続化しない。

実装順の推奨:

1. **Phase 1（意味変更と可視化を同時リリース）**: L1 → L2 → L3
2. **Phase 2（world 構造化）**: L4
3. **Phase 3（ナッジ）**: L5

L2 と L3 は同一リリースに含める。既存データの値自体は変換しないが、`Character.currentState` の意味を「手動の現在状態」だと解釈していた既存利用者にとっては意味上の移行になるため、プロンプト側の抑制と同時に値をUIで可視化する。実装は L1+L2 と L3 を別コミットにしてもよいが、片方だけの配布はしない。

## L1. 【作品設定】セクションに時点の明示を追加

**問題**: 【作品設定】（世界設定・人物設定・関係性設定）は「物語開始時点の記述」なのか「常に正しい記述」なのかをLLMに伝えていない。関係性設定は特に、初期関係を毎回主張し続け、StoryState 側の「関係変化」と並走する。

**方針**: セクション冒頭に一文で時点と優先関係を明示する。ただし、世界の法則や人物の恒常的な属性まで「過去の情報」とは扱わない。作品設定を基礎設定と位置づけ、そのうち時間で変化しうる記述は開始時点の情報である、と明示する。

**実装**:

1. `promptBuilder.buildPrompt` の【作品設定】組み立て箇所で、ヘッダ直後に注記を加える:
   ```
   【作品設定】
   以下は作品の基礎設定である。このうち時間とともに変化しうる記述は物語開始時点の情報として扱う。
   物語の進行によって変わった事柄は、採用済み本文を最優先し、次に【現在状態スナップショット】を優先する。
   ```
2. 文言は `renderWishSection` の信頼順（採用済み本文 ＞ 派生データ ＞ 作品設定）と矛盾しないこと。「スナップショットを優先」とだけ書いて本文への言及を落とさない。

**受け入れ条件**:

- worldText または characters がある生成プロンプトで、【作品設定】ヘッダ直後に注記が出る。
- 【作品設定】自体が出ない（設定が空の）場合は注記も出ない。
- ユニットテスト: 注記の有無と位置。

## L2. `Character.currentState` を「初期状態」に再定義し二重出力を解消

**問題**: `renderCurrentState`（`promptBuilder.ts` 268〜279行）は次の2系統を同じ【人物の現在状態】に並べる。

- `characters.filter((c) => c.currentState?.trim())` — 静的。セットアップ時の「開始時点の状態」のまま更新されない。
- `storyState.characterStates` — 動的。シーン採用のたびに更新される。

同一人物が両方に載ると、古い状態と新しい状態が同格で矛盾する。しかも静的側はクライアントに表示・編集UIがなく、**ユーザーから見えない値が毎回プロンプトに注入されている**。

**方針**: `Character.currentState` の意味を「**物語開始時点の初期状態**」と定義し直す（生成側=setupPromptBuilder は既にこの意味で生成しており、消費側を合わせるだけ）。プロンプトでは、StoryState にその人物の現在状態があれば静的側を出さない。StoryState がまだその人物を捕捉していない間だけ、初期状態を「開始時点」と明示して出す。

検討した代替案: 静的側を「ユーザーが手で維持する現在状態の上書き」と再定義し StoryState より優先させる案は不採用。編集UIが存在せず手で維持される実態がないこと、setup が「開始時点の状態」として生成している事実と食い違うことが理由。

**実装**:

1. `src/server/utils/characterStateMatching.ts` に副作用のない人物マッチング共通関数を切り出す:
   ```ts
   function matchStoryCharacterStates(
     characters: Character[],
     states: StoryCharacterState[]
   ): {
     byCharacterId: Map<CharacterId, StoryCharacterState>;
     unmatchedStates: StoryCharacterState[];
     diagnostics: CharacterStateMatchDiagnostic[];
   }
   ```
   マッチングは配列全体を一度に解決し、1つの StoryState を複数人物へ割り当てない。優先順位は characterId → 一意な正規化名 → 一意な aliases。空文字は候補にしない。名前または alias が複数人物に一致する曖昧な場合は、誤って初期状態を抑制しないよう未照合にする。同じ characterId の StoryState が複数ある破損データでは、妥当な `updatedAt` が最も新しいものを採用し、残りを diagnostics に返す。純粋関数内ではログを書かず、呼び出し側が diagnostics を警告ログへ出す。既存の `renderCharacterKnowledgeState` も同じ照合結果を使う。
2. `renderCurrentState` は「一人物一行」に組み直す:
   - Character ごとに共通関数で StoryState を照合する。
   - StoryState の `currentState` が非空ならそれだけを状態本文として使い、`Character.currentState` は出さない。
   - StoryState の `currentState` が空または未照合なら、`Character.currentState` を `初期状態（現在状態未取得）: ...` と明示して使う。
   - StoryState の `relationships` は同じ人物行に `関係変化: ...` として追記する。
   - `knowledge` はここでは出さず、既存の【人物別の認識状態】だけで出す。現状の二重出力を解消する。
   - Character に照合できない孤立 StoryState は、動的情報を失わないよう末尾に一度だけ出す。同じ StoryState を複数人物へ割り当てない。
   - 状態本文も関係変化もない行は出さない。
3. `storyStateService.buildUpdatePrompt` の人物ヒント（126行）: `currentState` キーを `initialState` に改名する（入力プロンプトの人物ヒントJSONだけ。型変更なし）。状態抽出LLMが初期状態を「現在の状態」と誤認して差分に混ぜるのを防ぐ。LLMに返させる差分JSONのキーは従来どおり `characterStates[].currentState` であり、`normalizeCharacterState` も変更しない。更新方針に「人物ヒントの initialState を出力キーとして模倣せず、現在状態は currentState に返す」と明記する。
4. `refineScanService.renderCharactersForPrompt`（260〜262行）: ラベルを `currentState:` から `initialState（開始時点）:` に変える。あわせて scan の判断基準に一文追加: 「initialState は開始時点の状態であり、storyState の現在と食い違うこと自体は正常。矛盾として指摘しない。」これがないと、再定義後は**物語が進んだプロジェクト全件で偽陽性の contradiction が出る**。
5. `refineChatService` の人物レンダリング（607行）も同様にラベルを変更。`CharacterFieldPatch.currentState` によるパッチ適用は「初期状態の修正」として引き続き許可する（機能変更なし）。
6. `roleplayPromptBuilder` は変更しない（既に「会話開始時点の状態」ラベル）。
7. `Character.currentState` と `CharacterFieldPatch.currentState` の型コメント、および refine chat のパッチ生成指示を「物語/会話の開始時点の状態」に更新する。フィールド名は互換性のため維持する。

**受け入れ条件**:

- StoryState に currentState 非空のエントリがある人物: 【人物の現在状態】に StoryState を使った1行だけが出る。
- StoryState にエントリがない人物: 初期状態が `初期状態（現在状態未取得）` 付きで1行出る。
- StoryState エントリはあるが currentState が空（knowledge / relationships のみ等）の人物: 状態本文は初期状態を使い、relationships は同じ行へ追記する。knowledge は【人物別の認識状態】だけに出る。
- Character に対応しない StoryState は欠落せず、かつ重複せずに出る。
- 1人物につき状態行は最大1行、同じ knowledge は1セクションだけに出る。
- `renderCharacterKnowledgeState` の既存テストが共通関数化後も通る。
- scan のシステム指示に「initialState と storyState の不一致は正常」が含まれることをテストする。
- ユニットテスト: 上記3ケース + characterId / 正規化名 / aliases 経由の一致 + 全角・空白差 + 重複名・alias衝突 + 重複characterId（最新採用とdiagnostic）+ 空名。

## L3. 初期状態の編集UI（WorkSettingsTab 人物編集）

**問題**: `Character.currentState` はクライアントのどこにも表示・編集欄がない（サーバー側 `routes/settings.ts` のバリデーションは既に受け付ける）。プロンプトに入る値をユーザーが見られない・直せない。

**方針**: `WorkSettingsTab` の人物編集フォームに「初期状態（物語開始時点）」欄を追加する。ラベルで L2 の再定義をUI上も固定する。

**実装**:

1. `WorkSettingsTab.tsx` の人物編集フォーム（speechStyle / want / fear / secrets の並び、1321〜1350行付近）に `currentState` のテキストエリアを追加。ラベル「初期状態」、プレースホルダ「物語開始時点の状態。進行中の現在状態は『物語の状態』側で管理される」。
2. 人物カードのサマリー表示（want / fear / secrets を出している箇所）にも `初期状態: ...` を追加。
3. サーバー変更なし（バリデーション済み）。`project.projectType === 'roleplay'` ではラベルを「会話開始時点の状態」、novel（未設定を含む）では「初期状態（物語開始時点）」と出し分ける。

**受け入れ条件**:

- 人物編集で初期状態を入力・保存でき、再読込後も保持される。
- 空にして保存すると、プロンプトの【人物の現在状態】から静的行が消える（L2 の挙動と接続）。

## L4. world.md の「開始時点の状況」セクション規約

**問題**: world.md 内で不変の法則と開始時点の状況が混在し、後者が陳腐化しても全文が【世界設定】として同格に注入され続ける。

**方針**: world.md は自由記述のまま、見出し規約 `## 開始時点の状況` をオプトインで導入する。規約があればプロンプト側で分割し、開始時点部分にだけ時点注記を付ける。規約がなければ従来どおり（既存プロジェクトの挙動不変）。

**実装**:

1. `promptBuilder` に、worldText を行単位で分割する純粋関数を追加する。
   - 対象見出しは `^(#{2,4})\s*開始時点の状況\s*$`。前後空白を許容し、コードフェンス内の同名行は無視する。
   - 対象見出しの直後から、次の「同じか上位レベル」の見出しの直前までを開始時点セグメントとする。下位レベルの見出しはセグメント内に残す。
   - 対象見出しより前と、セグメント終了後の本文は通常セグメントへ戻す。結果は `normal | initial` の ordered segments とし、元の出現順を変えない。隣接する同種セグメントだけを結合する。
   - normal → 【世界設定】。initial → 【世界設定（開始時点の状況）】として出し、注記「以下は物語開始時点の状況である。進行によって変わった事柄は、採用済み本文を最優先し、次に【現在状態スナップショット】を優先する」を付ける。同種セクションが離れて複数回現れることは許容する。
   - 見出し行そのものは出力しない。片側が空なら、その側のセクションは出さない。
   - 見出しがなければ worldText のレンダリングは従来どおり。L1 の【作品設定】冒頭注記を除き、【世界設定】本文は変更しない。
   - 未閉じコードフェンスなど解析の確信が持てない入力では、本文を落とさないことを優先し、全文を従来の【世界設定】として出す（fail-open）。
2. `setupPromptBuilder` の worldText 生成指示（novel 用 345行、roleplay 用 402行）に追記: 「時間の経過や物語の進行で変わりうる状況（勢力関係、人物の所属・所在、季節など）は、`## 開始時点の状況` という見出しの下にまとめる。変わらない法則・地理・文化はその見出しより前に書く」。新規プロジェクトは最初から分離された world.md を持つ。
3. `storyStateService.buildUpdatePrompt` の【世界設定抜粋】（4000字切り出し）は変更しない。切り出しで見出しが欠ける可能性はあるが、状態抽出の参考情報にすぎないため許容する。
4. docs/利用ガイドに見出し規約の説明を1段落追加。

**受け入れ条件**:

- 見出しありの world.md で、プロンプトに【世界設定】と【世界設定（開始時点の状況）】が分かれて出る。
- 見出しなしの world.md は、L1 の共通注記を除き従来と同じ【世界設定】出力になる。
- ユニットテスト: 分割あり/なし/見出しが先頭/末尾、次の同レベル・上位・下位見出し、複数セグメント、コードフェンス内の同名行、未閉じコードフェンス、CRLF。

## L5. 設定陳腐化ナッジ（refine scan 導線強化）

**問題**: 本文が進んで設定が陳腐化しても、ユーザーが自発的に refine scan を押さない限り気づけない。また現行 scan は「initialState と storyState の食い違い」以外の陳腐化（description / relationshipNotes / world の記述が現在の物語状態と食い違う）を明示的に探す指示を持たない。

**方針**: (a) 最後に**成功した** scan が確認した StoryState diff の位置をカーソルとして保持し、それ以降の差分量からナッジを判定する。(b) world / characters / 有効システムプロンプトの静的入力フィンガープリントも保持し、設定自体が編集された場合は件数を待たず再レビュー対象にする。(c) scan プロンプトに「進行による陳腐化」の検出指示を加える。`generatedAt` は走査完了時刻であり、走査中に適用された diff を誤って確認済みにしてしまうため、判定基準には使わない。動的→静的の自動パッチ提案（storyState refresh が refine patch を生成する案）は複雑すぎるため今回は見送り、既存の refine scan → refine chat 導線に乗せる。

**実装**:

1. scan 基準カーソル:
   - `RefineScanResult` に任意フィールド `reviewedStoryStateDiffId?: string | null`、`reviewedStoryStateUpdatedAt?: string | null`、`reviewedStaticInputHash?: string | null` を追加する。
   - StoryState と diff は既存の StoryState mutex の内側でスナップショットとして読み、最新の保持済み diff の `diffId` と StoryState の `updatedAt` を今回走査の候補基準にする。これにより、スナップショット取得後から scan 完了までに追加された diff は確認済みにしない。
   - hash は scan に実際に渡す world、正規化済み characters、解決済み systemPrompt、タイトルを安定した順序で連結し、SHA-256で算出する。StoryState は場面ごとに変わるため hash へ含めず、diff カーソルで扱う。
   - `lastError === null` の成功結果だけが候補基準と hash を新しい確認済みメタデータとして保存する。失敗・部分失敗時は、直前キャッシュの確認済みメタデータをすべて引き継ぎ、ナッジを消さない。
   - 既存キャッシュにカーソルがない場合は「未走査相当」とし、保持済み diff 全件をバックログ候補にする。
2. バックログ算出:
   - `story-state-diffs.json` は最大20件しか保持しないため、厳密な累計件数は返さない。確認済み `diffId` が履歴内にあれば、それより新しい diff 数を数える。カーソルが履歴から脱落している場合は `needsReview = true` とし、件数表示は `backlogCountLowerBound = REFINE_NUDGE_DIFF_COUNT` とする。
   - scan 未実行なら保持済み diff 数を数える。0件ならナッジなし、閾値未満ならナッジなし、閾値以上ならナッジあり。
   - `reverted` の diff も、採用本文が進んだ事実を示すため件数に含める。将来「本文自体の採用取り消し」を導入した場合は再検討する。
   - 現在の StoryState `updatedAt` が「最新の非reverted diff の `resultUpdatedAt`」（該当diffがなければ確認済み `reviewedStoryStateUpdatedAt`）と一致しない場合は、StoryState JSON の手動編集とみなし、件数に関係なく再レビュー対象にする。さらに新規の自動 diff には直前の `StoryState.updatedAt` を `previousUpdatedAt` として残し、最新の非reverted diff から確認済み時点までの更新連鎖をたどる。途中に対応する自動更新がない場合も、手動編集または差し戻しが挟まったものとして再レビュー対象にする。これにより、手動編集の直後に自動更新が続いても検知を失わない。L5 導入前の diff に `previousUpdatedAt` がない場合は、誤検知を避けて前者の直接比較だけを使う。
   - 現在の静的入力 hash が確認済み hash と異なる場合も、件数に関係なく再レビュー対象にする。既存キャッシュで hash がない場合は、初回移行時に常時ナッジを出さないため、diff 条件だけで判定し、次の成功 scan から hash 判定を有効にする。
3. API: 既存 `GET/POST /projects/:id/refine/scan` のレスポンス型は変えない。`GET /projects/:id/refine/status` を追加し、次を返す。
   ```ts
   type RefineReviewReason =
     | 'story_progressed'
     | 'history_truncated'
     | 'settings_changed'
     | 'story_state_edited';

   interface RefineReviewStatus {
     backlogCountLowerBound: number;
     needsReview: boolean;
     threshold: number;
     reasons: RefineReviewReason[];
   }
   ```
   算出は `refineScanService` に置き、route はプロジェクト存在確認と委譲だけにする。`ProjectState.storyStateBacklogCount` は「StoryState に未反映の採用シーン数」で意味が異なるため流用しない。
4. 閾値はサーバー側の定数 `REFINE_NUDGE_DIFF_COUNT = 10` に一元化し、UIは `needsReview` を使う。`WorkSettingsTab` は設定レビューカードの初期ロード時と scan 完了後に status を取得し直す。status 取得失敗は既存の scan 表示を壊さず、ナッジだけを非表示にする。
5. `WorkSettingsTab` 設定レビューカード: `needsReview` のとき、reason が設定変更系なら「設定が前回のレビューから変更されています」、本文進行系なら「前回のレビューから本文が進んでいます。設定と現状のずれを確認しますか？」と表示する。ボタンは既存の scan 実行ボタンを流用し、新規UIは追加しない。件数は下限値であり得るため、正確な「N件」とは表示しない。
6. `refineScanService` の判断基準に追記: 「description / relationshipNotes / world の記述が storyState の現在の状況と食い違う場合は、物語の進行による陳腐化として contradiction で指摘し、suggestedFix に現状を反映した書き換え案を書く。initialState と storyState の食い違いは正常であり指摘しない（L2 の4と同一文面を維持）」。

**受け入れ条件**:

- 成功した scan の直後はナッジが消え、diff が閾値まで溜まると再表示される。
- scan が失敗または部分失敗した場合は、直前の確認済みカーソルを維持し、ナッジが誤って消えない。
- scan 未実行のプロジェクトでは diff 総数で判定される。
- scan 実行中に新しい diff が追加された場合、その diff は次回レビュー対象として残る。
- 保持上限によりカーソルが diff 履歴から脱落しても `needsReview` は true になる。
- world / characters / 有効システムプロンプトの編集後は、diff が0件でも `settings_changed` で再表示される。
- StoryState の手動編集または差し戻し後は、diff カーソルが進んでいなくても `story_state_edited` で再表示される。
- APIテスト: status の正常系、プロジェクト不在、既存 scan API のレスポンス互換性。
- ユニットテスト: バックログ算出（scan 前/成功後/失敗後、0件、9件、10件、カーソル脱落、reverted、走査中の追加）、hash の決定性と変更検出、手動 StoryState 更新とその後の自動更新、既存キャッシュのフォールバック。

## 将来課題（本書のスコープ外）

- **storyFact メモの失効**: 「AはまだBを知らない」型のメモは進行で偽になるが、手動アーカイブしか手段がない。refine scan の入力にメモを加えて陳腐化を指摘させる案があるが、入力肥大とのトレードオフのため見送り。
- **want / fear のアーク管理**: 長期連載でのキャラクターアーク（欲求・恐れの変化）は現状表現できない。StoryState 側に arc フィールドを設ける案は、抽出精度の検証を含めて別トラック。
- **動的→静的の自動橋渡し**: storyState refresh が「設定レベルの変化」を検知したとき refine patch を自動提案する仕組み。L5 のナッジで運用上の穴を塞いだ後、必要性を再評価する。
