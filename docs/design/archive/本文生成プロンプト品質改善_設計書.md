# 本文生成プロンプト品質改善 設計書

- 作成日: 2026-07-20
- 対象バージョン: 0.1.0-beta.5 → 次リリース
- 実装後の扱い: 実装完了後に `archive/` へ移動する

## 背景と目的

本文生成の出力が「均質で説明的」になりやすい傾向を減らし、場面ごとの自然なリズムと具体性を高める。調査の結果、本文生成プロンプトの組み立て経路（`generationService` → `promptBuilder` → 各アダプタ）に以下の要因を特定した。

1. **字数指定が全場面同一かつ狭すぎる帯**（主因）。既定 6,000 字で許容幅 ±500 字（`outputLength.ts` の tolerance 上限）。下限がノルマ化し、薄い場面を説明文で埋める動機になる。
2. **直近本文 12,000 字による自己模倣ループ**。プロンプト末尾近くの直近本文が最強の文体シグナルとなり、単調な文体が再生産される。語句レベルの対策（手動NG表現）はあるが自動対抗策がない。
3. **文体見本（1,000 字）が直近本文に埋没する配置**。見本の直後に 12,000 字の直近本文が続き、recency と分量で見本が負ける。
4. **文脈切り詰めが文の途中で起こる**。直近本文の先頭（`slice(-maxChars)`）と文体見本の末尾（1,000 字切り）が文の途中で壊れる。
5. **temperature 既定 0.7 が創作用途としてやや保守的**。

場面無関係な人物データの全量投入（人物設定・現在状態・情報状態）も説明調を誘発する要因として特定したが、誤検出リスクがあるため本設計では任意項目（F）とする。

## 修正項目

### A. 字数指定の緩和（主因・最優先）

**対象**: `src/server/prompts/promptBuilder.ts` の `renderOutputConditions`

**変更**: 下限〜上限の帯提示をやめ、「上限＋標準目安＋場面密度への委任＋埋め草禁止」に変える。

現行:

```
- 目安文字数: 約${target}字（${lower}〜${upper}字程度）。字数を合わせるために急停止せず、文や段落の切りがよいところで自然に終える。上限に届きそうな場合は少し短くても自然な区切りを優先する。
```

変更案:

```
- 文字数: 上限は約${upper}字。${target}字前後を標準としつつ、場面が求める密度に応じてそれより短くてよい。字数を満たすための説明・要約・感情の言い換えによる引き延ばしはしない。場面が自然に閉じる位置で、文や段落の切りがよいところで終える。
```

- `getApproximateOutputRange`（`outputLength.ts`）は変更しない。`upper` は `estimateMaxOutputTokens` のトークン見積りにも使われており、プロンプト文言だけを変える。
- `【今回の希望】` の優先順位「④目安文字数」は「④文字数の上限」に文言を合わせる。

**リスク**: 出力が想定より短くなる場合がある。標準値（target）をアンカーとして残すことで緩和する。受け入れ確認で複数場面の生成長のばらつきを見る。

**テスト**: `tests/unit/promptBuilder.test.ts` の文言アサーションを更新。下限値がプロンプトに現れないことを確認。

### B. temperature 既定の引き上げ 0.7 → 0.9

**対象**:

- `src/server/services/projectService.ts` の `DEFAULT_TEMPERATURE`（0.7 → 0.9）
- `src/server/services/generationService.ts` の `TEMPERATURE_DEFAULT`（0.7 → 0.9）
- `src/client/components/TechSettingsTab.tsx` の表示フォールバック `?? 0.7`（→ 0.9。UI 要素の追加はしない）

**据え置き**: `TEMPERATURE_VARIATE_DELTA`（+0.15）、`TEMPERATURE_MAX`（1.3）、`TEMPERATURE_SUMMARY`（0.25。要約・StoryState 抽出は事実整理タスクのため低温のまま）。

**重要な制約**: temperature はプロジェクト作成時に明示保存されるため（`projectService.ts` の作成処理）、**既存作品には自動では効かない**。保存済みの 0.7 が「旧既定の自動保存」か「利用者の意図的選択」か区別できないため、データ移行はしない。既存作品は技術設定のスライダで手動変更する運用とし、リリースノートに明記する。

**テスト**: `projectService` 系テストの既定値アサーションを更新。新規作成プロジェクトの `samplingConfig.temperature` が 0.9 になることを確認。

### C. 文体見本の配置換え（自己模倣ループの緩和・その1）

**対象**: `src/server/prompts/promptBuilder.ts` の `buildPrompt` セクション順

**変更**: `【文体見本】` を直近本文の後ろへ移動する。

- 現行順: … → 出力形式 → 表現上の注意 → **文体見本** → 直近本文 → （書き直し対象） → 今回の希望
- 変更後: … → 出力形式 → 表現上の注意 → 直近本文 → （書き直し対象） → **文体見本** → 今回の希望

直近本文より後（= より高い recency）に見本を置き、「直近本文の調子より見本を優先する」という既存指示と実際の影響力を一致させる。セクション文言は変更しない。

**テスト**: `promptBuilder.test.ts` / `promptBuilderLayering.test.ts` にセクション出現順のアサーションを追加・更新。

### D. 文境界での切り詰め

**対象**: 新規ユーティリティ `src/server/utils/textBoundary.ts`（仮称）＋適用 2 箇所

1. `contextAssembler.ts` の `getRecentContext`: `joined.slice(-maxChars)` の後、最初の段落境界（`\n`）まで、なければ最初の文末（`。」`『』考慮は不要、`。`／`！`／`？`＋閉じ括弧程度）まで先頭を捨てる。境界が見つからない場合（maxChars 全体が 1 文など異常系）はそのまま返す。
2. `promptBuilder.ts` の文体見本 `slice(0, 1000)`: 末尾側を最後の文末境界まで切り詰める。保存層（`projectService` の 1,000 字正規化）は変更せず、描画時のみ整える。

**テスト**: 新規 `tests/unit/textBoundary.test.ts`（境界あり／なし／改行のみ／短文）。`contextAssembler` 経由の結合テストは既存テストの枠内で更新。

### E. 頻出表現の自動注入（自己模倣ループの緩和・その2）

**対象**: `src/server/prompts/promptBuilder.ts`（`renderBannedExpressions` 周辺）

**変更**: `buildPrompt` 内で取得済みの直近本文に対して `extractFrequentPhrases`（`utils/phraseFrequency.ts`、既存）を適用し、上位の頻出語句を `【表現上の注意】` に自動追記する。

- 手動NG（最大 12 件、現行どおり「使わないこと」）と区別し、自動検出分は小見出しを分けて最大 8 件、指示は「直近の本文で繰り返し使われている。多用を避け、同じ意味は別の言い方で書くこと」とする（完全禁止にしない。口癖など意図的な反復を壊さないため）。
- **除外フィルタ**（誤検出対策・必須）:
  - 人物名・呼び名（`Character.name` / `aliases`）を含む語句
  - 手動NGと重複する語句（正規化して比較）
  - `phraseFrequency` 既存の閾値（出現 3 回以上・スコア順）に加え、採用は上位 8 件まで
- 対象テキストは `getRecentContext` の戻り値をそのまま使う（追加 I/O なし）。分析上限は既存の `ANALYZE_MAX_CHARS` に従う。

**リスク**: 意図的なモチーフ反復（象徴として繰り返す語）を抑制しうる。「多用を避ける」という弱い指示にとどめることと、件数上限で緩和する。

**テスト**: `promptBuilder.test.ts` に、頻出語句がセクションに載ること・人物名が除外されること・手動NGとの重複排除・件数上限のケースを追加。

### F. 人物データの場面連動圧縮（任意・最後に判断）

**対象**: `src/server/prompts/promptBuilder.ts` の `renderCharacters` / `renderCurrentState` / `renderCharacterKnowledgeState`

**変更案**: 「直近本文・今回の希望・現在の状況のいずれにも名前／呼び名が現れない `supporting` / `other` の人物」は、人物設定を 1 行（名前・役割・概要）に圧縮し、現在状態・情報状態からは除外する。`protagonist` / `deuteragonist` は常に全量。

**リスク**: 希望文で名指しせずに登場させたい脇役の情報が痩せる。効果（プロンプトの説明調データ削減）と引き換えのトレードオフであり、A〜E の効果を確認してから着手要否を判断する。**本設計では実装保留を既定とする。**

## 実施順序

1. A（字数指定の緩和）
2. C（文体見本の配置換え）
3. D（文境界トリム）
4. B（temperature 既定変更）
5. E（頻出表現の自動注入）
6. F は保留。A〜E 適用後の生成結果を見て判断。

A〜D は独立で小さく、E のみ新ロジックを含む。1〜4 をひとまとまり、E を別コミットに分ける。

## 影響範囲

- 変更するのは小説本文生成経路（`promptBuilder.ts` とその入力）のみ。ロールプレイ（`roleplayPromptBuilder.ts`）、セットアップ相談、要約生成、StoryState 抽出のプロンプトは変更しない。
- UI・設定項目・保存データのスキーマは変更しない（B の表示フォールバック定数のみ）。
- データ移行なし。

## 検証

- 単体テスト: 上記各項目のテスト更新・追加。`npm test`（vitest）全通過。
- 手動確認: 既存プロジェクトで continue / regenerate / variate 各モードの生成を行い、(1) プロンプトプレビュー（コンテキスト見積り経路）にセクション順・文言が正しく出ること、(2) 生成長が場面によりばらつくこと、(3) 頻出語句セクションに人物名が混入しないことを確認。
- 受け入れテスト手順書の該当項目（生成系）を通す。
