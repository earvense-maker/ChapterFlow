# 種メモ強化・物語整合性 設計書

## 0. 背景

本書は2つの改善トラックをまとめた設計書であり、これ単体で実装作業へ渡せることを目的とする。

- **トラックS（種メモ強化）**: 「相談して作る」で蓄積した種メモ（`SetupDraft`）の情報が、コミット後の本文生成に届くまでに目減りしている問題への対処。
- **トラックK（物語整合性）**: 本文生成時にLLMが事実関係を間違えにくくする仕組み、および人物ごとの「知っていること／知らないこと」の混同を防ぐ仕組み。

対象コード:

- `src/shared/types.ts`（Project / Character / StoryState / Setup系型）
- `src/server/prompts/promptBuilder.ts`
- `src/server/services/setupPromptBuilder.ts`
- `src/server/services/setupCommitService.ts`
- `src/server/services/setupDraftPatchService.ts`
- `src/server/services/storyStateService.ts`
- `src/server/services/generationService.ts`
- `src/server/services/projectService.ts`
- `src/server/routes/generate.ts` / `src/server/routes/state.ts` / `src/server/routes/projects.ts`
- `src/client/components/SetupCommitReview.tsx` / `WorkSettingsTab.tsx` / 物語状態編集画面（生成フォーム・リーダーの新規UIは追加しない）
- `tests/unit/` の対応テスト

前提となる現状仕様（変更しないもの）:

- プロジェクトデータは `data/projects/{projectId}/` 配下のJSON。`safeWriteJson` / `readJsonFile` を使う。
- 本文生成プロンプトは `promptBuilder.buildPrompt` が組み立てる。セクションは `\n\n---\n\n` 区切り。
- シーン採用時に `storyStateService.updateStoryStateFromAcceptedScene` がLLMで差分抽出し `mergeStoryState` で適用する。温度0.15。
- `ProjectState.storyStateRefresh`（fresh / pending / stale）と `POST /projects/:id/story-state/refresh` は実装済み。
- LLM呼び出しはadapter経由。エラー時は既存のエラー伝搬経路に従う。

互換性方針:

- **新規フィールドはすべてオプショナル**とし、読み込み時にサービス層で欠損を補正する。`schemaVersion` は変更しない。
- 既存プロジェクトの `StoryEventRecord.visibility`（自由文）は残す。新規抽出では `knownBy` / `explicitlyUnknownBy` を主とし、`visibility` はフォールバック表示にのみ使う。
- `types.ts` はクライアントと共有のため、型追加時はクライアントのコンパイルが通ることを確認する。

実装順の推奨（依存関係は各項目に記載）:

1. Phase A（プロンプト・コミット変換の修正。UI変更ほぼなし）: S1 → S3 → S6 → S2
2. Phase B（コミットレビューUIを含む）: S4 → S5
3. Phase C（抽出パイプライン改修）: K1+K2（一体で実装） → K3 → K4
4. Phase D（仕上げ）: K5 → K6 → **K8（物語状態編集画面の新設）** → K7

UX方針（全項目に共通）: 複雑さは機械側に寄せ、ユーザー操作面は「自動で埋まる、直したい時だけ触れる」を原則とする。生成フォームとリーダー画面には新規UI要素を追加しない。

重要な前提: **コミット後のstoryStateを閲覧・手動編集する画面とAPIは現状存在しない**（クライアントの編集UIはコミット前のプラン編集のみ、サーバーのstory-state系ルートはrefreshのみ）。S5の未確定管理・K1/K2の誤抽出修正（knownByから人物を外す操作）・K7の履歴と取り消しは、すべてK8で新設する画面を受け皿とする。S5・K7のUI記述はK8に依存するため、UI部分の実装はK8以降になる。

## トラックS: 種メモ→生成の情報伝達強化

### S1. tone（文体の好み）が本文生成に届かない問題の修正

**問題**: `setupCommitService.mergeDraftPreferenceMemories` は `draft.tone` を `importance: 'medium'` の preference memory として保存するが、`promptBuilder.buildPrompt` は `importance === 'high'` のメモリしか使わない（`promptBuilder.ts` の `highMemories` フィルタ）。結果、相談で集めた文体の好みはコミット後の生成に一切反映されない。

**方針**: コミット側のimportanceは変えない（重要度の意味を保つ）。`promptBuilder` 側で preference / negative は medium も出力対象にする。storyFact は従来どおり high のみ。

**実装**:

1. `buildPrompt` 内のメモリフィルタを分離する:
   - `renderImportantPast` へ渡すのは従来どおり `importance === 'high'` の storyFact。
   - `renderPreferenceNotes` へ渡すのは `status === 'active'` かつ `importance` が `high` または `medium` の preference / negative。
2. 出力行数の上限を設ける: preference と negative の合計16行。超過時は high 優先、次に `updatedAt` の新しい順。

**受け入れ条件**:

- tone由来（medium preference）のメモリが生成プロンプトの【好み・NG】に現れる。
- low importance の preference は出力されない。
- ユニットテスト: medium preference の出力、16行超過時の優先順位。

### S2. openingSeeds を初回生成の「今回の希望」に引き継ぐ

**問題**: `draft.openingSeeds` は、コミットLLMが `worldText` を返した場合 `buildFallbackWorldText` が使われず完全に失われる。冒頭候補は本来「第1話の今回の希望（wish）」に流すのが最も効く。

**方針**: コミットプランに `firstWishSuggestion`（初回生成のwish提案文、1件）を追加し、プロジェクトに保存。クライアントは初回生成時にwish欄へプリフィルする。

**実装**:

1. 型変更: `Project` / `CreateProjectBody` / `SetupCommitPlan` に `firstWishSuggestion?: string`（上限300字）を追加。
2. `setupPromptBuilder.buildSetupCommitPrompt` の出力JSON形式に `firstWishSuggestion` を追加。指示: 「openingSeeds と相談の流れから、第1話冒頭への希望を1文で書く。openingSeedsが空なら省略可」。
3. `setupCommitService.normalizeSetupCommitPlan`: LLM出力が無い場合のフォールバックとして `draft.openingSeeds` の先頭要素を使う（300字で切る）。※openingSeeds は `string[]` で status を持たない。
4. `SetupCommitReview.tsx`: プラン確認画面に編集可能テキスト欄として表示。
5. クライアント: プロジェクトを開いた際、**採用済み生成が0件**（`state.lastAcceptedGenerationId === null` で判定）かつ wish入力欄が空のとき `firstWishSuggestion` をプリフィルする（ユーザーは編集・削除可能）。プリフィルは**最初のシーン採用まで毎回**行う（消して画面を離れ、再訪した場合は復活する。一度きりにするためのフラグ管理はしない）。採用後は条件から外れるため自然に出なくなる。

**受け入れ条件**:

- openingSeedsがある状態でコミットすると、プロジェクトに firstWishSuggestion が保存される。
- 新規プロジェクトの初回生成フォームにwishがプリフィルされ、シーン採用後は再プリフィルされない。
- ユニットテスト: LLM出力あり/なし両方の正規化。

### S3. 「作品の核」を生成プロンプトに常駐させる

**問題**: `draft.coreConcept` はコミット時にworldTextへ溶けて第一級の情報でなくなる。連載が長くなり要約圧縮が進むほど、テーマのドリフトが起きる。

**方針**: `Project.coreConcept` を新設し、生成プロンプトの先頭に毎回固定で出す。

**実装**:

1. 型変更: `Project` / `CreateProjectBody` / `UpdateProjectBody` に `coreConcept?: string`（上限300字）を追加。`projectService` の作成・更新で受け付ける。
2. `buildSetupCommitPrompt` の出力JSONに `coreConcept` を追加。指示: 「この作品が何の話で、どんな読み味を約束するかを1〜2文で書く」。フォールバック: `draft.coreConcept`。
3. `promptBuilder.buildPrompt`: `coreConcept` があれば **partsの先頭**（【作品設定】より前）に追加する:
   ```
   【この作品の核】
   ${coreConcept}
   この核から外れた作風・テーマの展開を書かないこと。
   ```
4. `WorkSettingsTab.tsx`: 作品設定画面で編集可能にする。

**受け入れ条件**:

- coreConcept設定済みプロジェクトの生成プロンプト先頭に【この作品の核】が出る。未設定なら出ない。
- ユニットテスト: promptBuilder のセクション順（核 → 作品設定）。

### S4. 文体見本の自動設定と「もっとこう」再生成

**問題**: 相談中の試し書き（previews、最大3件）はコミットプロンプトに参考として渡るだけで、本文生成には届かない。文体をLLMに守らせるには形容詞の指示より実文例の提示が効く。

**方針**: ユーザーに「見本を選ぶ」作業をさせない。コミットプラン作成時に自動で `styleSample` を設定し、コミットレビューでは「気に入らなければ指示を書いて作り直す」「見本なしにする」だけを提供する。既存の書き直しリロールと同じ操作感にする。

**実装**:

1. 型変更: `Project` / `CreateProjectBody` / `SetupCommitPlan` に `styleSample?: string`（保存上限1000字）を追加。
2. 自動設定: `createSetupCommitPlan` 内で、最新の preview があればその本文を `styleSample` に採用する。previews が空なら試し書き生成を1回実行して採用する。**生成に失敗した場合は styleSample 空のままプラン作成を続行する**（コミットをブロックしない）。
   - **デッドロック注意**: `createSetupCommitPlan` は `withSessionLock` 内で実行され、`generateSetupPreview` も同じロックを取るため、そのまま呼ぶと**確実にデッドロックする**（mutexは非再入）。試し書き生成のコア処理（プロンプト構築＋LLM呼び出し）を**ロックを取らない内部関数に分離**し、`createSetupCommitPlan` からはそれを呼ぶこと。
   - この経路で生成した試し書きは **session.previews には積まない**（plan.styleSample にのみ入れる）。積むとプラン作成中に revision が余分に進み、クライアントのrevision追跡が複雑になるため。ユーザー起点の再生成（下記3）は従来どおり previews API 経由で、previews にも積まれる。
3. 指示付き再生成:
   - `buildSetupPreviewPrompt` に任意の `styleHint?: string` を追加し、指定時は【文体への希望】セクションとして入れる。
   - 試し書きAPI（previews系エンドポイント）のbodyに `instruction?: string` を追加。
   - `SetupCommitReview.tsx`: 見本本文＋指示入力欄（プレースホルダ例: 「もっと会話を増やして」「湿度高めに」）＋「この指示で作り直す」＋「見本を使わない」を表示。再生成結果で `plan.styleSample` を置換する。
4. 指示の学習化: 再生成に使われた指示テキストは `draft.tone` にも追加する（`normalizeComparableText` で重複排除）。S1経由で以後の本文生成にも効く。
5. `promptBuilder.buildPrompt`: `styleSample` があれば【出力条件】の直前に追加:
   ```
   【文体見本】
   以下は文体・リズム・描写の密度の見本である。内容・人物・出来事は本編と無関係であり、参照しないこと。書き方だけを参考にすること。

   ${styleSample.slice(0, 600)}
   ```
6. `WorkSettingsTab.tsx`: 作品設定画面では表示・貼り替え・削除のみ（本編の気に入った一節を貼り直す使い方を想定）。再生成UIはコミットレビュー限定とする。

**受け入れ条件**:

- previewがある状態でプラン作成すると自動でstyleSampleが入る。previewが無ければ自動生成され、生成失敗時もプラン作成自体は成功する。
- 指示付き再生成でstyleSampleが置換され、指示がdraft.toneに追加される（重複は排除）。
- 「見本を使わない」でstyleSample空のままコミットでき、その場合プロンプトに見本セクションが出ない。
- ユニットテスト: 自動採用の優先順位（最新preview→自動生成→空）、失敗時継続、tone追加の重複排除、600字トリミング。

### S5. 「作者が未確定」と「物語内の未解決」を区別する

**問題**: コミットプロンプトは未確定事項を `storyState.openThreads` に入れさせるが、生成側の【未解決事項】は「回収してよい伏線」として読まれる。作者がまだ決めたくないこと（事件の真相など）を生成LLMが勝手に確定させるリスクがある。

**方針**: `StoryState.authorUndecided` を新設し、生成プロンプトでは「確定させるな」という逆向きの指示で別枠表示する。

**実装**:

1. 型変更: `StoryState` に追加:
   ```ts
   authorUndecided?: Array<{
     id: string;            // 'und-' プレフィックスのID
     text: string;          // 未確定の内容
     reason?: string;       // 未確定にしている理由
     status: 'active' | 'resolved' | 'archived';
     updatedAt: string;
   }>;
   ```
   上限12件。`storyStateService.normalizeStoryState` / `createEmptyStoryState` で欠損補正。
2. `buildSetupCommitPrompt`: 出力JSONの `storyState` に `authorUndecided` を追加し、指示を変更: 「作者が決めていない事項は openThreads ではなく authorUndecided に入れる。openThreads は作中で提示済みの謎・伏線だけにする」。
3. `setupCommitService.normalizeSetupCommitPlan`: **LLM出力に依存せず**、`draft.undecided`（active）を authorUndecided に必ずマージする（F1の `mergeDraftPreferenceMemories` と同じ思想。重複は `normalizeComparableText` で排除）。
4. `promptBuilder.renderCurrentState`: 【未解決事項】とは別に追加:
   ```
   【まだ確定させないこと】
   以下は作者がまだ決めていない事項である。作中で真相・答え・正体を確定させず、曖昧さを保ったまま書くこと。
   - ${text}（${reason}）
   ```
5. `storyStateService` の抽出プロンプト: authorUndecided は**抽出・更新の対象外**であることを明記（既存JSONに含めて渡すが、差分形式に含めない）。解決済みにするのはユーザーの手動操作のみ。
6. UI: authorUndecided の一覧・追加・resolve・削除は **K8で新設する物語状態編集画面**に実装する（現状この画面は存在しないため、本項のUI部分はK8以降に実施。データ構造・コミット変換・プロンプト出力はK8を待たず実装できる）。

**受け入れ条件**:

- undecidedを含む相談をコミットすると、openThreadsではなくauthorUndecidedに入る。
- 生成プロンプトに【まだ確定させないこと】が【未解決事項】と別枠で出る。
- 状態抽出の差分でauthorUndecidedが変化しない。
- ユニットテスト: コミット時の必須マージ、抽出マージでの不変性。

### S6. 人物の内面駆動フィールドの追加と、secrets のプロンプト出力

**問題（2つ、セットで直す）**:

1. `SetupDraftCharacter` には description / speechStyle / relationshipNotes しかなく、コミットプロンプトが要求する「物語上の揺れ」を入れる構造的な置き場がない。
2. `Character.secrets` はスキーマに存在するのに `promptBuilder.renderCharacters` が出力しない。**このままではフィールドを増やしても死蔵になる**ため、レンダリング拡張を本項のスコープに含める。

**方針**: 種メモとCharacterの両方に want / fear を追加し、種メモの secret を `Character.secrets` へマップ。生成プロンプトの人物設定に欲求・恐れ・秘密を出力する。

**実装**:

1. 型変更:
   - `SetupDraftCharacter` に `want?: string; fear?: string; secret?: string` を追加。
   - `Character` に `want?: string; fear?: string` を追加（secrets は既存）。
2. `buildSetupChatPrompt` の draftPatch 形式（charactersAdd / charactersUpdate）に want / fear / secret を追加。指示: 「人物には可能なら、欲しいもの(want)・恐れ(fear)・秘密(secret)を短く入れる。ユーザーが明言していない場合は候補として提案してよい」。
3. `setupDraftPatchService`: 新フィールドの正規化・ロック対象パスへの追加（既存フィールドの扱いに準拠）。
4. `setupCommitService.normalizeCharacter` / `normalizeDraftCharacters`: want / fear / secret(→secrets) をマップ。`buildSetupCommitPrompt` の出力形式にも追加。
5. `promptBuilder.renderCharacters` を拡張:
   ```
   - ${name}（${role}）
     概要: ...
     口調: ...
     欲求: ${want}
     恐れ: ${fear}
     秘密: ${secrets}（本人だけが知る。知らない人物の前では言動に出さず、地の文でも軽々に明かさないこと）
   ```
   - NOTE: secrets は「初期設定としての秘密」。作中で誰かに開示された場合の管理はK1のイベント（knownBy）が担う。開示済みかどうかの判断はここではしない。
6. `WorkSettingsTab.tsx` の人物編集に want / fear / secrets の入力欄を追加（secrets欄が既に無い場合）。

**受け入れ条件**:

- 相談チャットで人物のwant/fear/secretがdraftに入り、コミット後のCharacterに引き継がれる。
- secrets設定済み人物の生成プロンプトに秘密行と取り扱い注記が出る。
- ユニットテスト: パッチ正規化、コミットマップ、renderCharacters出力。

## トラックK: 物語整合性・知識管理

### K1. イベントの knownBy / explicitlyUnknownBy 構造化と、人物別情報状態のレンダリング

**問題**: 「誰が何を知っているか」が `characterStates[].knowledge`（人物側の自由文配列）と `importantEvents[].visibility`（イベント側の自由文）の2箇所にあり、矛盾しても検出できない。また「まだ知らない」という否定形の情報がどこにもなく、LLMの秘密漏洩を防ぐ最有力手段が使えていない。

**方針**: イベントを知識の唯一の真実にする。`knownBy` / `explicitlyUnknownBy` を characterId 配列で持ち、生成時に人物ごとの「知っている／まだ知らない」を**コードで導出して**レンダリングする。

- NOTE: 場面キャスト選択（生成前に登場人物を絞る機能）は今回スコープ外。本項では全人物分を導出して出す。後日キャスト選択を導入する場合、この導出結果にフィルタを掛けるだけでよい構造にしておく。

**実装**:

1. 型変更: `StoryEventRecord` に追加:
   ```ts
   knownBy?: string[];             // characterId。このイベントを知っている人物
   explicitlyUnknownBy?: string[]; // characterId。「知らない」ことが物語上重要な人物
   ```
   `visibility`（自由文）は残すが、新規抽出では使わない。
2. `storyStateService` の抽出プロンプト（`buildUpdatePrompt`）を変更:
   - 差分JSON形式の importantEvents に `knownBy` / `explicitlyUnknownBy` を追加し、`visibility` を削除。
   - 指示: 「knownBy / explicitlyUnknownBy には【人物ヒント】のcharacterIdだけを使う。該当IDが無い人物（モブ等）は含めない」。
   - **過剰生成の抑制**: 「explicitlyUnknownBy は通常0〜2名。『知らないこと』が物語上の緊張や皮肉を生む人物だけを入れる。その場にいなかった人物を機械的に列挙しない」を明記する（LLMは不在者全員を入れる方向に過剰生成しがちで、膨らむとプロンプト長と禁止規則の希釈に直結する）。
   - characterStates の差分形式から `knowledge` を**削除**する（今後は更新させない。既存データの knowledge は保持・表示のみ）。
3. `mergeStoryState` / `normalizeEventRecord` / `normalizeEventPatch`: 新フィールドのマージ対応。**機械的検証**を追加: knownBy / explicitlyUnknownBy 内のIDが characters に存在しない場合はそのIDを黙って除外する（マージに characters リストを渡す。呼び出し元 `updateStoryStateFromAcceptedScene` は `input.characters` を持っている）。
   - **「知らない→知る」の遷移**: knownBy に追加されたIDは、同イベントの explicitlyUnknownBy から**自動的に除去**する（最頻出のライフサイクル。放置すると同一人物が両方に入ったまま恒久化し、プロンプトに「知っている」と「まだ知らない」が同時出力される）。
   - **explicitlyUnknownBy の上限**: イベントあたり4件。超過分はマージ時に先頭4件だけ残して切り捨てる（プロンプト指示の抑制をすり抜けた過剰生成の防波堤）。
   - **旧形式 knowledge の保護**: `normalizeCharacterStatePatch` は現状 knowledge フィールドが来ると既存値を上書きする（`storyStateService.ts` の `hasField(value, 'knowledge')` 分岐）。プロンプトから knowledge を外すだけでは、LLMが旧形式で返した場合に既存値が消える。**マージコード側でも characterStates パッチの knowledge を無視する**（既存値を常に維持）こと。
4. `promptBuilder.renderCurrentState` に人物別情報状態を追加:
   ```
   【人物の情報状態】
   - ${name}
     知っている: ${knownBy にこの人物が入るイベントのsummary、importance high優先で最大6件}
     まだ知らない: ${explicitlyUnknownBy にこの人物が入るイベントのsummary、全件}
   ```
   - 「まだ知らない」が1件も無い人物は「まだ知らない」行を省略。「知っている」「まだ知らない」両方空の人物は出さない。
   - 対象は全人物（`Character` に active/status フラグは無いため、登録済み全員）。
   - **防御**: 同一イベントで knownBy と explicitlyUnknownBy の両方に同じ人物が入っていた場合（自動除去をすり抜けた旧データ等）、knownBy を優先し「まだ知らない」には出さない。
   - 既存の `characterStates[].knowledge`（旧形式）が残っている人物は「知っている」に併記する（重複は文字列比較で除外）。
5. `renderOutputConditions` に禁止規則を追加:
   ```
   - 【人物の情報状態】で「まだ知らない」とされた事実を、その人物の台詞・内心・行動の根拠・その人物視点の地の文に使わないこと。
   ```
6. `renderImportantPast` のイベント表示: `visibility` の代わりに knownBy を人物名解決して「知っている人物: A, B」形式で出す。knownBy が無い旧イベントは従来どおり visibility を出す。

**受け入れ条件**:

- knownBy付きイベントがあるとき、生成プロンプトに【人物の情報状態】が人物別に出る。
- explicitlyUnknownBy の人物に「まだ知らない」行が出て、出力条件に禁止規則が出る。
- 存在しないcharacterIdがマージ時に除外される。
- 旧データ（visibilityのみ・knowledgeあり）でもエラーなくレンダリングされる。
- ユニットテスト: 導出ロジック（knownBy→知っている、上限6件、旧knowledge併記）、ID検証、マージ。

### K2. 抽出時の「居合わせた人物」「新たに知った人物」の必須化

**問題**: 現行の抽出形式はイベントに `characters`（関係人物）しかなく、「イベントの当事者」と「イベントを知っている人物」の混同が起きやすい。知識伝播の根は「その場に誰がいたか」である。

**方針**: 抽出プロンプトで `presentCharacters`（居合わせた人物）と `learnedBy`（この場面で新たに知った人物。又聞き・立ち聞き含む）を要求し、コード側で knownBy に畳み込む。

**実装**（K1と一体で実装する）:

1. 抽出プロンプトの差分JSON形式のイベントに `presentCharacters` / `learnedBy` を追加。指示:
   - 「presentCharacters: その出来事の場に居合わせた人物のcharacterId」
   - 「learnedBy: その場にいなかったが、この場面で伝聞・立ち聞き・観察などにより知った人物のcharacterId」
   - 「characters: 出来事の当事者（従来どおり）」
2. マージ時のコード処理: `event.knownBy = 既存knownBy ∪ presentCharacters ∪ learnedBy`（重複除去、ID検証はK1と同様）。`presentCharacters` / `learnedBy` 自体は**保存しない**（knownByに畳む。スキーマを増やさない）。
3. 既存イベントの更新パッチでも同様に和集合で反映。knownByから人物を外す操作は手動編集（K8の物語状態編集画面）のみとする。

**受け入れ条件**:

- 立ち聞きシーンの抽出で、learnedBy経由の人物がイベントのknownByに入る。
- 既存イベントへの更新で knownBy が単調増加（勝手に消えない）。knownBy に入った人物が同イベントの explicitlyUnknownBy から自動除去される。
- ユニットテスト: 畳み込み、和集合マージ、explicitlyUnknownBy自動除去。

### K4. 物語内時間（storyClock）

**問題**: 時間の構造化表現が無く、「翌朝の次にまた朝が来る」「一晩に1週間分の出来事」系の時系列エラーを防げない。

**方針**: 相対時間（◯日目・時間帯）を storyState に持たせ、抽出で更新し、生成プロンプトに常時出す。

**実装**:

1. 型変更: `StoryState` に追加:
   ```ts
   clock?: {
     day: number;         // 物語開始からの日数。1始まり
     timeOfDay?: string;  // 朝/昼/夕/夜 など自由文
     note?: string;       // 「文化祭前日」等の補足
   };
   ```
   イベント側（`StoryEventRecord`）へのclock追加は**行わない**。抽出・レンダリングでの利用先が無いフィールドは死蔵になるため（S6の問題2と同じ理由）。イベントの発生時刻が必要になったら、レンダリング仕様とセットで追加する。
2. `setupCommitService.normalizeStoryState`: コミット時に `clock: { day: 1 }` で初期化（LLM出力があればそれを優先）。
3. 抽出プロンプト: 差分JSONに `clock` を追加。指示: 「場面終了時点の物語内時間を更新する。経過が読み取れない場合は既存の値をそのまま返す。日をまたいだ描写があればdayを進める」。
4. `mergeStoryState`: clockのマージ（差分にあれば置換、なければ維持）。dayが既存より小さい値に戻る差分は無視する（回想シーンでの誤更新ガード）。
5. `promptBuilder.renderCurrentState`: 【現在の状況】の先頭行に `物語内時間: ${day}日目・${timeOfDay}${note ? `（${note}）` : ''}` を出す。出力条件に「物語内時間と矛盾する時間経過・時間帯を書かないこと。時間を進める場合は本文中で自然に示すこと」を追加。

**受け入れ条件**:

- コミット直後のstoryStateに clock(day:1) がある。
- 抽出でclockが更新され、生成プロンプトに時間行が出る。
- dayが逆行する差分が無視される。
- ユニットテスト: マージ（更新・維持・逆行ガード）。

### K5. 視点束縛（wish駆動・UIなし）

**問題**: pov preset（三人称一元など）はあるが「誰の視点か」の指定が無く、視点人物が知らない事実が地の文から漏れる、他人物の内心を断定するなどのPOV違反を防げない。

**方針**: 視点人物の指定UI・設定項目は**設けない**。「今回の希望」に「◯◯の視点で」と書けば適用される、という自然な操作を正とし、サーバー側で機械的に補強する。型変更なし・UI変更なし。

**実装**:

1. 常設ルール: `renderOutputConditions` に無条件で追加する（視点指定の有無に関わらず入れる）:
   ```
   - 地の文は視点人物の認識範囲で書くこと。視点人物が【人物の情報状態】で「まだ知らない」とされた事実を地の文で開示しないこと。
   - 視点人物以外の内心は、外から観察できる言動として描き、断定で書かないこと。
   - 視点人物は、今回の希望に指定があればその人物、指定が無ければ直近本文の視点を維持すること。
   ```
2. wishの決定的解析（LLM不使用）: `buildPrompt` 内で、characters の `name` / `aliases`（K3）を用いて wish 文字列から「${名前}の視点」「${名前}視点」パターンを検索する。**複数の名前が一致した場合は最長一致を優先する**（例: 「美咲」と「咲」が両方登場する作品で「美咲の視点で」と書かれたとき、「咲の視点」の部分一致に負けないこと）。一致した場合のみ、束縛ブロックを具体名で展開して【出力条件】に追加:
   ```
   - 視点人物: ${name}。この場面は${name}の視点で書く。
   ```
   さらに【人物の情報状態】（K1）で該当人物の行を先頭に並べ替える。
3. 一致しない場合は常設ルールのみ。`Project` / `GenerateRequestBody` / `GenerationRequest` への型追加、生成フォーム・設定画面へのUI追加は行わない。

**受け入れ条件**:

- wishに「◯◯の視点で」と書くと、具体名入りの束縛文が出力条件に出る（aliasでの指定も一致する）。
- 視点指定の無いwishでは常設ルール3行のみが加わり、それ以外は従来と同一のプロンプト。
- 人物名がwishに含まれるだけ（「視点」を伴わない）では発火しない。
- ユニットテスト: name一致 / alias一致 / 不一致 / 誤発火なし / 最長一致優先（部分文字列衝突） の5ケース。

### K6. 状態抽出の未反映バックログ管理（既存実装の拡張）

**問題**: `storyStateRefresh`（fresh/pending/stale）と手動refreshは実装済みだが、**最新1件のgenerationしか追跡しない**。抽出が失敗した後に別のシーンを採用すると、失敗したシーンの差分は再抽出対象から漏れ、永久に反映されない。

**方針**: 処理済みgenerationIdの集合を持ち、「採用済みだが未反映」のバックログを機械的に計算できるようにする。refreshはバックログ全件を古い順に処理する。

**実装**:

1. 型変更: `StoryState` に `processedGenerationIds?: string[]` を追加。**トリミングしない**（IDは短い文字列であり数百件でも数KB。上限を設けて古いIDを削ると、削られたシーンがバックログ判定に復活して再抽出され、イベント・スレッドが重複する自壊バグになる）。
2. `updateStoryStateFromAcceptedScene`: 抽出・マージ成功時に対象generationIdを追記。
3. 互換初期化: `processedGenerationIds` が未定義の既存プロジェクトは「現在の全採用済みgenerationIdを処理済み」として初期化する（過去分を遡って再抽出しない）。実行場所は**バックログ計算関数内のlazy初期化**とする（`readStoryState` はエピソード一覧にアクセスできないため。計算関数はサービス層に置き、エピソード一覧とstoryStateの両方を読み、未定義なら初期化して永続化してから計算する）。
4. バックログ計算をサービス関数として追加: 全エピソードの `scenes[].acceptedGenerationId`（非null）のうち `processedGenerationIds` に無いものを、エピソード順・シーン順で返す。
5. `GET /projects/:id/state`（または reader state）のレスポンスに `storyStateBacklogCount` を追加。
   - 鮮度表現の正: **バックログを正とする**。`backlogCount > 0` はUI上 stale と同義に扱う。既存の `storyStateRefresh` は進行中表示（pending）とエラーメッセージの伝達用として残す。
6. バックログのドレイン処理を共通関数として実装する: バックログを**古い順**に、各シーンについて `updateStoryStateFromAcceptedScene` を順次呼ぶ（1回の実行で上限10件）。
   - **失敗時は中断する**: 途中のシーンで抽出が失敗したら、そのシーン以降は処理せず、処理済み件数・残数・エラーを返す。抽出は「前の場面の状態が適用済み」という前提で積み上がるため、失敗をスキップして次を適用すると順序整合（本設計の眼目）が壊れる。
7. `POST /projects/:id/story-state/refresh` の挙動変更: 最新1件ではなく、上記ドレイン関数を呼ぶ。レスポンスに処理件数と残数を含める。
8. **採用時の自動抽出もドレイン関数に統一する**: 現行の採用フロー（`refreshStoryStateAfterAcceptance`）は採用したgenerationだけを抽出するため、古いシーンの抽出失敗が残った状態で新シーンを採用すると、新しい方が先に適用される逆順が起きる。採用時も「バックログを古い順にドレイン」に置き換える（採用直後のバックログには今回のgenerationが末尾に入っているので、正常時の挙動は従来と同じ。失敗残があるときだけ古い方から処理される）。これでrefreshと採用時のコードパスが1本になる。
9. UI: 既存のstale表示を拡張し、バックログ件数を表示（「物語状態: 2場面未反映」）。refreshボタンは既存のものを流用。

**受け入れ条件**:

- 抽出失敗→別シーン採用 のとき、採用時のドレインで古いシーンから順に処理される（新しい方だけ先に適用される逆順が起きない）。
- ドレイン中の失敗で処理が中断され、失敗シーン以降のstoryStateが変化しない。残数がレスポンスで返る。
- 既存プロジェクトを開いてもバックログ0から始まる（遡及抽出しない）。
- 処理済みIDが多数（数百件）あっても古いシーンがバックログに復活しない。
- ユニットテスト: バックログ計算、lazy初期化の一回性、ドレインの順序・失敗中断・上限10件。

### K7. 抽出差分の記録と取り消し（裏側のみ）

**問題**: 状態抽出の誤りは複利で蓄積するが、誤った自動更新を元に戻す手段が無い（state編集画面での全文手動修正のみ）。

**方針**: **リーダー・生成フローには一切表示しない**。差分の記録とLIFO取り消しをバックエンド機能として持ち、唯一の入口を**K8で新設する**物語状態編集画面の「自動更新の履歴」セクションにする。誤抽出の発見は「ユーザーが読んでいて矛盾に気づいた時」で構わないという割り切りであり、その時に履歴が復旧手段として機能する。チップ・通知・採用後のポップアップは実装しない。

**実装**:

1. 新規ストレージ: `data/projects/{projectId}/story-state-diffs.json`
   ```ts
   interface StoryStateDiffRecord {
     diffId: string;             // 'diff-' プレフィックス
     generationId: string;
     sceneId: string;
     appliedAt: string;
     summary: {                  // チップ表示用にコードで集計
       addedEvents: string[];    // 追加イベントのsummary
       updatedEvents: string[];
       addedThreads: string[];
       resolvedThreads: string[];
       updatedCharacters: string[]; // 人物名
       clockChanged: boolean;
     };
     beforeState?: StoryState;   // 適用直前のスナップショット（revert用）
     resultUpdatedAt: string;     // マージ適用後に書き込んだ storyState.updatedAt（staleness検査用）
     reverted: boolean;
   }
   ```
   summaryは最新20件を保持。**beforeState は直近3件のみ保持**し、古い行からは削除する（revertはLIFOで最新しか戻せないため全量保持は不要。上限規模のStoryStateを20件抱えるとdiffファイルが数百KBになり、採用のたびに全書き換えされる）。
2. `updateStoryStateFromAcceptedScene`: マージ適用時に beforeState・summary・resultUpdatedAt を記録して diff を保存する。
3. Revert API: `POST /projects/:id/story-state/diffs/:diffId/revert`
   - **取り消せるのは「未取り消しの最新diff」のみ**（LIFO）。それ以外は409を返す（理由: 途中のdiffだけ戻すには逆適用が必要で、後続diffとの整合が壊れるため。古い誤りはK8の編集画面での手動修正へ誘導する）。
   - **staleness検査**: 現在の `storyState.updatedAt` が diff の `resultUpdatedAt` と一致しない場合は409を返す（diff適用後にK8で手動編集されている。revertすると手動編集が黙って消えるため）。エラーメッセージで「状態が手動編集されているため取り消せません」と案内する。
   - 処理: storyState を `beforeState` に置換 → diffの `reverted: true`。
   - **updatedAtは更新しない**: beforeState を `updatedAt` ごとそのまま書き込むこと（現在時刻で上書きしない）。beforeState.updatedAt は1つ前のdiffの resultUpdatedAt と一致するため、これにより連続revert（beforeState保持分の最大3段）のstaleness検査が自然に通る。「書き込み時はupdatedAtを更新する」という他経路の慣習に従うと、2段目のrevertが誤って409になる。
   - revert も `withStoryStateLock` を通す（横断注意参照）。
   - NOTE: `processedGenerationIds` からの明示的な除去は不要。beforeState は当該generationIdの追記前のスナップショットなので、置換だけでK6のバックログに自動的に再計上される。
4. 一覧API: `GET /projects/:id/story-state/diffs`（最新20件のsummaryとreverted可否）。
5. UI（K8の物語状態編集画面のみ）: 画面末尾に「自動更新の履歴」セクションを追加。各行にsummary（例: 「イベント+2 / 人物2名 / 時間→3日目」）と日時を表示し、**最新の未取り消し行にのみ**「この更新を取り消す」ボタンを出す。取り消し後は「取り消しました。再抽出するか、状態を手動で編集できます」の案内を出す。リーダー画面・生成フォーム・採用フローには新規UI要素を追加しない。
6. 個別項目の微修正はK8の編集機能に任せる（本項では全体revertのみ）。
7. **採用取り消し（unaccept）との連動**: `unacceptCurrentScene` は現状、抽出結果をstoryStateに残したまま採用だけを取り消す。unaccept時、当該generationIdのdiffが「未取り消しの最新」でありstaleness検査も通る場合に限り、**revertを自動実行**する。条件を満たさない場合は何もしない（警告ログのみ。古い残骸はK8での手動修正に委ねる）。取り消し→別案採用という高頻度フローで旧案のイベントが残骸として蓄積するのを防ぐ。LIFO制約と自然に噛み合い、ユーザー操作なしでK7の仕組みが活きる唯一の自動化ポイント。

**受け入れ条件**:

- 物語状態編集画面（K8）で履歴が確認でき、最新diffのrevertでstoryStateがbeforeStateへ戻り、K6のバックログに該当シーンが再計上される。
- 2つ前のdiffのrevert、および手動編集後（updatedAt不一致）のrevertは409。
- revert後の連続revert（2段目・3段目）がstaleness検査を通る（updatedAt非更新の検証）。
- unaccept時、当該generationのdiffが最新なら自動revertされ、最新でなければstoryStateが変化しない。
- リーダー画面・生成フォームに新規UI要素が追加されていない。
- ユニットテスト: diff保存・summary20件/beforeState3件のトリミング、revertのLIFO制約、staleness検査、連続revert、unaccept連動、バックログ再計上。

### K8. 物語状態の閲覧・編集画面の新設（S5・K7の前提タスク）

**問題**: コミット後の storyState を閲覧・手動編集する手段が存在しない。クライアントの storyState 編集UIはコミット前のプラン編集（`SetupCommitReview.tsx`）のみで、サーバー側にも読み書きAPIが無い（`routes/state.ts` は ProjectState のみ、story-state系は `generate.ts` の refresh のみ）。このままでは、S5の未確定管理、K1/K2の誤抽出修正（knownByから人物を外す唯一の手段）、K7の履歴・取り消しに受け皿が無い。

**方針**: 作品設定まわりに「物語状態」画面を新設する。閲覧が主・編集が従（「自動で埋まる、直したい時だけ触れる」原則）。リーダー・生成フローからは独立させる。

**実装**:

1. サーバーAPI:
   - `GET /projects/:id/story-state`: 正規化済み StoryState を返す。
   - `PUT /projects/:id/story-state`: 編集可能フィールド（currentSituation / characterStates / importantEvents（knownBy・explicitlyUnknownBy含む） / openThreads / authorUndecided / clock）の全置換。既存の `normalizeStoryState` で検証し、`updatedAt` を更新する。
   - **characterIdの実在検証**: knownBy / explicitlyUnknownBy のID検証（K1-3）は characters リストが必要で `normalizeStoryState` 単体ではできない。PUT処理内でも characters を読み込み、存在しないIDを除外すること。
   - **`processedGenerationIds` は編集対象外**とし、PUTのbodyに含まれていてもサーバー側で既存値を維持する（手動編集でバックログ管理が壊れるのを防ぐ）。
   - **排他**: PUTは `withStoryStateLock`（`storyStateService` のmutex）を通す。採用直後の非同期抽出と手動編集が競合すると、どちらかの書き込みが黙って消えるため（横断注意も参照）。
2. クライアント: `WorkSettingsTab` に「物語状態」セクション（またはタブ）を追加。内容:
   - 現在の状況・物語内時間（K4）の表示と編集。
   - 人物の情報状態: イベント由来の「知っている／まだ知らない」を人物別に表示し、イベント側の knownBy / explicitlyUnknownBy を編集できる（人物の追加・除去）。
   - 重要イベント・未解決事項の一覧・編集・アーカイブ。
   - まだ確定させないこと（S5の authorUndecided）の一覧・追加・resolve・削除。
   - 自動更新の履歴（K7）: 履歴一覧と最新diffの取り消しボタン。
3. 編集の保存は全置換PUT一発（項目別APIは作らない。単一ユーザー・ローカルアプリなので楽観ロック等は不要とするが、K7のstaleness検査が成立するよう `updatedAt` の更新は必須）。

**受け入れ条件**:

- コミット済みプロジェクトの storyState が画面で閲覧・編集でき、編集が次回生成のプロンプトに反映される。
- knownBy から人物を外す操作ができる（K1/K2の誤抽出の唯一の修正手段）。
- PUT で processedGenerationIds が変化しない。
- ユニットテスト: PUTの正規化・processedGenerationIds保護・updatedAt更新。

## 実装上の横断注意

- **プロンプト長の管理**: K1・K4・S3・S4・S6でプロンプトセクションが増える。`contextEstimate` の見積もり対象に新セクションを含めること。【人物の情報状態】は人物あたり「知っている」6件上限で抑制済みだが、人物数が多い作品では【これまでの作品本文（直近）】の `maxChars` とのバランスを実測で確認する。
- **抽出プロンプトの負荷**: K1+K2+K4で抽出タスクの出力項目が増える。few-shot例を1つ抽出プロンプトに入れることを推奨（立ち聞きシーン→learnedBy反映の例が最も誤りやすい）。抽出精度の受け入れテストとして、正解付きシーン数本のゴールデンテストを `tests/unit` に置くとよい。
- **story-state.json を書くすべての経路は `withStoryStateLock` を通す**: 対象は 抽出（既存）・K8のPUT・K7のrevert（unaccept連動含む）。採用直後の非同期抽出と手動編集・revertが競合すると、後勝ちでどちらかの書き込みが黙って消える。読み取り→変更→書き込みの一連をロック内で行うこと。
- **StoryStateへのフィールド追加は3点セット**: `normalizeStoryState` は2実装ある（`setupCommitService` と `storyStateService`）。さらに `mergeStoryState` は previous を正規化して再構築するため、authorUndecided（S5）・clock（K4）・processedGenerationIds（K6）のいずれかを足し漏れると、**マージのたびに黙って消える**。StoryStateにフィールドを追加する際は「両normalize＋mergeStoryState」の3箇所を必ず同時に対応すること。可能なら正規化ロジックの共通化（`storyStateService` の normalize を `setupCommitService` から使う）を先行タスクにするとよい。
- **順序依存**: K7はK6の `processedGenerationIds` とK8の編集画面に依存する。S5のUI部分もK8に依存する（データ・プロンプト部分は独立）。K1とK2は同一プロンプト・同一マージ処理を触るため一体で実装する。K5のwish解析はK3の `aliases` を使うが、K3未実装でも `name` のみの照合で動作させてよい。S6のレンダリングとK1のレンダリングは同じ `renderCharacters` / `renderCurrentState` 周辺を触るため、実装順が離れる場合はコンフリクトに注意。
- **設定画面の見せ方**: S3・S6・K3で増えるフィールド（作品の核・文体見本・欲求・恐れ・秘密・呼び名）は、いずれも相談・コミットで自動的に埋まる「直したい時だけ触る欄」である。設定画面では既存の基本項目と分け、折りたたみの「詳細設定」にまとめて既定では畳んでおく。人物編集のフォームが縦に長くなりすぎる場合、欲求・恐れ・秘密・呼び名は1つの折りたたみグループにする。
- **コミットプラン作成のレイテンシ**: S4の自動見本生成により、previewsが空の場合はプラン作成でLLM呼び出しが2回（commit変換＋試し書き）になる。順次実行で構わないが、クライアントの待機表示が2段階になることを許容する（「試し書きを作成中…」→「作成プランを整理中…」等）。
- **後方互換の確認観点**: 旧project.json（新フィールド無し）、旧story-state.json（visibility自由文・knowledge配列あり・clock無し）、旧セッションJSONのすべてで、読み込み→生成→抽出が例外なく通ること。
