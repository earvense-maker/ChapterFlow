# 表現調整機能 設計書

## 0. 背景と設計方針

本文生成に使うモデルは Gemini Flash / DeepSeek クラスを前提とする。このクラスのモデルは同じ言い回し・同じ比喩を反復しやすく、読者が最も敏感に反応する劣化要因になっている。

設計の基本原則は「賢さをモデルに求めず、構造で解決する」こと。

- 反復の**検出**は、採用済み本文の全履歴がローカルにあるという本アプリの強みを使い、LLMを使わない決定的処理で行う。
- 反復の**予防**は、抽象的な指示（陳腐な表現を避けよ）ではなく、弱いモデルでも追従できる「具体的な文字列の禁止リスト」と「サンプリングパラメータ」で行う。
- 読者ごとに気になる表現は異なるため、汎用の陳腐表現辞書は持たず、**プロジェクト単位の個人辞書**を育てる方式にする。

本設計書は以下の4機能を対象とする。

| 機能 | 概要 | 主な変更箇所 |
|---|---|---|
| A. サンプリングペナルティ | frequency/presence penalty をアダプタ経由で指定可能にする | adapters, Project型, SettingPanel |
| B. 頻出表現レポート | 採用済み本文の文字n-gram頻度分析と表示 | 新規 util/service, 新規API, クライアント |
| C. NG表現リスト | ユーザーが「見たくない表現」を登録する個人辞書 | 新規 expressions.json, 新規API, クライアント |
| D. 禁止リスト注入 | B・Cの結果を生成プロンプトへ注入する | promptBuilder, generationService |

### スコープ外（将来検討）

- 推敲パス（生成後に頻出表現を含む文だけをアンカーつきで書き直させる）。生成コストが倍近くなるため、まずB/Dの効果を見てから判断する。
- 比喩の領域ローテーション（今回使った比喩領域を抽出・記録し、次回避けさせる）。実験的要素が強い。
- 意味的な言い換え反復の検出（「息を呑んだ」と「言葉を失った」を同類と見なす等）。文字列一致の範囲に限定する。
- 相談モード・要約・storyState抽出へのペナルティ適用。構造化タスクの安定性を優先し、本文生成のみを対象とする。

## 1. 機能A: サンプリングペナルティ

### 1.1 目的

Gemini（generationConfig.frequencyPenalty / presencePenalty）と DeepSeek・OpenAI（frequency_penalty / presence_penalty）はいずれも反復抑制パラメータを持つが、現在のアダプタは temperature と maxOutputTokens しか送っていない。これを開放する。

直近本文はプロンプトに含まれるため、ペナルティは「今回の生成内での反復」に加えて「直近シーンで使った語の再使用」にもある程度効く。ただしペナルティの参照範囲（プロンプトを含むか出力のみか）はプロバイダ実装に依存するため、効果はモデル依存であると割り切る。

### 1.2 データ

`Project` に任意フィールドを追加する（`src/shared/types.ts`）。

```ts
export interface SamplingConfig {
  frequencyPenalty: number; // 0.0〜1.0
  presencePenalty: number;  // 0.0〜1.0
}

export interface Project {
  // 既存フィールド…
  samplingConfig?: SamplingConfig;
}
```

- `UpdateProjectBody` にも `samplingConfig?: Partial<SamplingConfig>` を追加。
- サーバー側検証: 有限数でなければ 0、0未満は 0、1.0 超は 1.0 に丸める（`projectService` の既存検証と同じ場所）。API仕様上は -2.0〜2.0 だが、日本語は強くかけると助詞まで痩せて文が崩れるため、UI・保存とも 0〜1.0 に制限する。
- 既定値は両方 0（= 送信しない）。既存プロジェクトはフィールド無しのまま動く（後方互換）。

### 1.3 アダプタ

`AdapterGenerateRequest` に任意フィールドを追加する。

```ts
export interface AdapterGenerateRequest {
  // 既存フィールド…
  frequencyPenalty?: number;
  presencePenalty?: number;
}
```

- 値が `undefined` または `0` のときは、**リクエストボディにフィールド自体を含めない**。非対応モデルとの互換性を守るため必須。
- GeminiAdapter: `generationConfig.frequencyPenalty` / `generationConfig.presencePenalty`（generateText / generateTextStream 両方）。
- DeepSeekAdapter / OpenAIAdapter: `frequency_penalty` / `presence_penalty`。

### 1.4 適用箇所とフォールバック

- 適用するのは `generationService.generateScene` / `generateSceneStream` の本文生成のみ。`compressProjectContext`、`storyStateService`、相談モード（setupSessionService）には渡さない。
- ペナルティ付きリクエストが HTTP 400（Gemini: INVALID_ARGUMENT）で失敗した場合、**ペナルティなしで1回だけ再試行**し、`console.warn` を残す。モデルが非対応でも生成自体は成立させる。再試行はリクエスト送信前の失敗（レスポンス確立前）に限る。ストリーミングでチャンク受信後に失敗した場合は再試行せず既存のエラー処理に乗せる。

### 1.5 UI

SettingPanel に「表現の反復抑制」セクションを追加。

- スライダー2本（frequencyPenalty / presencePenalty、0〜1.0、step 0.05）。
- 説明文: 「値を上げると同じ語の反復が減ります。上げすぎると文が不自然になります。目安 0.2〜0.5」。
- 保存は既存の `updateProject` 経由。

## 2. 機能B: 頻出表現レポート

### 2.1 目的

採用済み本文から「繰り返されている言い回し」を機械的に抽出し、ユーザーに見せる。体感ではなくデータで反復を確認でき、機能C（NG登録）と機能D（自動禁止リスト）の土台になる。

### 2.2 分析対象テキスト

- 全エピソードの採用済み本文（`buildEpisodeMarkdown` と同じソース: 各シーンの `acceptedGenerationId` の本文）を episode 順に結合する。
- 末尾 `ANALYZE_MAX_CHARS = 50_000` 字のみ分析する。読者が体感する反復は直近のものなので、古い本文まで遡る必要はない。

### 2.3 抽出アルゴリズム

新規 `src/server/utils/phraseFrequency.ts`（純関数、LLM不使用、Node依存なし）。

1. テキストを句読点・鉤括弧・改行（`。、「」『』！？\n`）でセグメントに分割し、各セグメント内の空白を圧縮する。
2. 各セグメントから文字 n-gram（n = 4〜10）を列挙し、出現回数を数える。n-gram はセグメント境界をまたがない。
3. フィルタ:
   - 出現回数 `MIN_COUNT = 3` 未満は捨てる。
   - ひらがなのみの候補は長さ5未満なら捨てる（「という」「ながら」等の機能語ノイズ対策。「まるで〜のように」級の慣用句は残る）。
   - 数字・英字・記号のみの候補は捨てる。
4. 極大化: 同じ出現回数で、より長い候補に完全包含される短い候補は捨てる（「息を呑ん」は「息を呑んだ」に吸収）。
5. スコア = 出現回数 × (文字数 − 2)。スコア降順で上位 `MAX_REPORT_ITEMS = 30` 件を返す。

計算量は 50,000字 × 7種のn ≒ 35万エントリの Map 構築で、オンデマンド実行で十分（キャッシュは初期実装では持たない）。

### 2.4 サービス層

新規 `src/server/services/expressionService.ts`。

```ts
export interface FrequencyReportItem {
  text: string;
  count: number;
  score: number;
  isNg: boolean; // NG表現リストに登録済みか
}

export interface FrequencyReport {
  generatedAt: string;
  analyzedChars: number;
  phrases: FrequencyReportItem[];
}

export async function buildFrequencyReport(projectId: string): Promise<FrequencyReport>;
```

- サービス層で、`characters.json` の人物名（`name` フィールド）を**含む**候補を除外する。人物名は頻出して当然であり、禁止リストに乗ると致命的なため。
- NG表現リスト（機能C）に登録済みの表現は `isNg: true` を付けて返す（除外はしない。登録済み表現がまだ出現しているかの確認に使う）。

### 2.5 API

```
GET /api/projects/:id/expressions/report
→ 200 FrequencyReport
```

### 2.6 UI

- Reader のツールバー（文脈圧縮ボタンの並び）に「表現レポート」ボタンを追加し、パネル表示。
- 各行: 表現、出現回数、「NGに登録」ボタン（登録済みならバッジ表示に変わる）。

## 3. 機能C: NG表現リスト

### 3.1 目的

読者が「もう見たくない」と感じた言い回しをプロジェクト単位で永続登録する個人辞書。既存の `negative` メモリは展開・設定内容のNG用であり、表現レベルのNGは性質が異なる（プロンプト内の置き場所も注入形式も違う）ため、**別ファイルに分離**する。

### 3.2 データ

新規ファイル `data/projects/{projectId}/expressions.json`。

```ts
export type NgExpressionSource = 'manual' | 'report' | 'selection';
export type NgExpressionStatus = 'active' | 'archived';

export interface NgExpression {
  id: string;            // generateTimestampId('ngx')
  text: string;
  source: NgExpressionSource;
  status: NgExpressionStatus;
  createdAt: string;
  updatedAt: string;
}

export interface ExpressionsFile {
  schemaVersion: 1;
  ngExpressions: NgExpression[];
}
```

- storageService に `expressionsJsonPath` / `readExpressions` / `writeExpressions` を追加（既存の `safeWriteJson` と `assertSafePathSegment` を使用）。
- ファイル不存在時は空リストとして扱う（既存の `readMemories` と同じ流儀）。

### 3.3 検証ルール

- `text` は trim 後 1〜30 文字。改行を含む場合は拒否（プロンプト注入の区画を壊さないため）。
- 空白圧縮＋前後trim で正規化した文字列が既存 active 項目と一致する場合は重複として拒否（409 ではなく既存項目を返す）。
- active 上限 `MAX_NG_EXPRESSIONS = 50`。超過時は 400 を返し、UIで整理を促す。
- 削除はソフトアーカイブ（`status: 'archived'`）。メモリ削除と同じ流儀。

### 3.4 API

```
GET    /api/projects/:id/expressions
→ 200 { ngExpressions: NgExpression[] }   // active のみ

POST   /api/projects/:id/expressions
body: { text: string, source?: NgExpressionSource }
→ 201 NgExpression（重複時は 200 で既存項目）

DELETE /api/projects/:id/expressions/:expressionId
→ 200 { ok: true }   // アーカイブ
```

### 3.5 UI

登録経路は3つ。

1. **表現レポートから**: 各行の「NGに登録」（source: 'report'）。主経路。
2. **本文選択から**: Reader の本文テキストを選択したとき、選択範囲が1〜30字ならフローティングの「この表現をNGに登録」ボタンを表示（source: 'selection'）。Reader に選択UIは現存しないため新設。将来の線引きフィードバック機能と同じ選択基盤になる想定で、初期実装は「選択→登録」だけの最小形にする。
3. **手入力**: SettingPanel の「NG表現」セクションに一覧＋追加入力欄＋削除ボタン（source: 'manual'）。

## 4. 機能D: 禁止リストのプロンプト注入

### 4.1 リストの構成

生成のたびに `expressionService.resolveBannedExpressions(projectId)` で以下を組み立てる。

1. NG表現リストの active 項目を登録の新しい順に、最大 `BAN_LIMIT_TOTAL = 12` 件まで。
2. 残り枠を、頻出表現レポートの上位（count ≥ 3、人物名除外済み、NG登録済みと重複しないもの）で埋める。自動分の上限は `AUTO_BAN_LIMIT = 8` 件。

**リスト総数の上限を 12 に固定する理由**: 禁止リストが長いほど本文全体の指示追従が崩れる（弱いモデルの特性）。上限は定数として1箇所（expressionService）に置き、モデルの追従実測に応じて調整できるようにする。追従テストの際は「5件 / 10件 / 20件での違反率」を測り、この定数に反映する。

### 4.2 プロンプト上の配置と文言

`buildPrompt`（promptBuilder.ts）に入力 `bannedExpressions: string[]` を追加し、**出力条件セクションの直後（userPrompt の最後）**に独立セクションとして置く。弱いモデルはプロンプト末尾の指示への追従が最も良いため。

```
【表現上の注意】
以下の言い回しは直近の本文に頻出しているか、読者が避けたい表現である。
今回の本文では使わないこと。同じ意味は別の言い方で書くこと。
- 「息を呑んだ」
- 「胸の奥が」
```

- 0件のときはセクションごと省略する。
- 各表現は「」で括り、データであることを明示する（表現文字列に指示文を混入されても1項目として扱われる形にする。改行拒否（3.3）はこの区画保護のため）。

### 4.3 呼び出し経路

- `generateScene` / `generateSceneStream`: プロンプト構築前に `resolveBannedExpressions` を呼び、`buildPrompt` に渡す。
- `buildReaderContextUsage`（トークン見積り）: 同じヘルパーを呼び、実際の生成と同じプロンプトで見積もる（見積りと実プロンプトの乖離を作らない）。
- 相談モード・要約・storyState抽出には注入しない。

### 4.4 記録

`GenerationRecord` に `bannedExpressions?: string[]` を追加し、その生成で注入したリストを記録する。後から「禁止したのに使われたか」を検証（テスト・追従率測定）できるようにするため。

## 6. 前提と割り切り

- **形態素解析は導入しない。** 文字n-gramの誤検出（意味の切れ目とずれる候補）は、人物名除外とユーザーの取捨選択で許容する。精度が問題になったら kuromoji 系の導入を検討する。
- **禁止リストの遵守はモデル依存であり、保証されない。** 本設計は違反率を下げる仕組みであって、ゼロにする仕組みではない。確実な除去が必要になった場合の受け皿が将来の推敲パス。
- **ペナルティの参照範囲（プロンプト含む/出力のみ）はプロバイダ実装依存。** 「直近本文との反復にも効く」は期待値であり仕様ではない。
- 頻出レポートは毎回オンデマンド計算する。連載が伸びて遅くなった場合は、採用イベント時の事前計算＋キャッシュに切り替える（分析対象を末尾50,000字に固定しているため、当面は不要と判断）。
- `expressions.json` は既存のバックアップ・エクスポート対象（`data/projects/` 配下）に自然に含まれる。スキーマ移行は `schemaVersion: 1` を置くのみで、移行コードは他ファイル同様に将来課題とする。
