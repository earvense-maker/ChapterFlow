# 運用機能拡張 設計書

> 注記: 本文中の「Yumeweaving」は開発時の旧称です。正式名称は「ChapterFlow」です。旧パス・旧環境変数は互換仕様として残る場合があります。

## 0. 背景と目的

日常運用を楽にする4つの独立した機能をまとめた設計書。これ単体で実装作業へ渡せることを目的とする。

- **G1(データディレクトリ変更)**: 作品データの保存先をアプリ内から確認・変更できるようにする。移動方式(コピー→検証→旧削除→自動再起動)。
- **G2(作品ショートカット)**: `<データディレクトリ>\作品一覧\<作品タイトル>.lnk` を自動生成し、Explorer 経由で本文 md を開けるようにする。
- **G3(生成中インジケータ)**: 生成ボタンのラベルにスピナー+パルスを表示し、生成中であることを視覚的に伝える。
- **G4(続き生成で最上部へスクロール)**: 生成開始時にリーダーを最上部へスクロールする。

対象コード:

- `src/electron/main.ts`(データディレクトリ設定の読み込み、再起動、ショートカット生成の PowerShell 呼び出し)
- `src/server/config.ts`(env よりアプリ設定を優先する仕組みは main で env を上書きする方式にするため、config.ts 自体は変更しない予定)
- `src/server/routes/system.ts`(データディレクトリ確認・移動 API 追加)
- `src/server/services/`(新設: `appSettingsService.ts`、`shortcutService.ts`。既存: `projectService.ts` に rename フックを追加)
- `src/client/components/`(新設: 「アプリ設定」画面、`GeneratingLabel.tsx`。既存: `Reader.tsx` へ G3・G4 適用、`ProjectList.tsx` から「アプリ設定」動線を追加)
- `src/client/clientApi.ts`(新 API の呼び出し)
- `src/client/styles.css`(G3 の CSS)

前提となる現状仕様(変更しないもの):

- Electron main が `startServer` を呼び、`YUMEWEAVING_DATA_DIR` env が設定されていればそれを使う(`config.ts:7`)。**main はサーバー起動前に env を書き換える機会がある** → G1 の切り替えはこの経路で行う。
- 各プロジェクトは `<DATA_DIR>/projects/<projectId>/` 配下で完結し、本文は `episodes/ep-*.md` 1本(**新規プロジェクトを含めエピソード md は常に1本**という現行仕様が確認済み)。
- ファイル書き込みは `safeWrite` 経由(atomic rename)。
- 「シャットダウン」経路は `POST /api/shutdown` → `onShutdownRequest`(Electron 版では `app.quit()`)。
- Electron の `app.getPath('userData')` は Windows で `%APPDATA%\Yumeweaving`。ウィンドウ状態も既にここに保存されている。

## G1. データディレクトリ変更

### 問題

現在、作品データの保存先は `%USERPROFILE%\Documents\Yumeweaving`(または `YUMEWEAVING_DATA_DIR` env)固定で、GUI から確認・変更できない。OneDrive リダイレクト環境で「マイドキュメントに見当たらない」問題や、外付けドライブに逃がしたい要望に対処できない。

### 方針

**アプリ設定ファイルを Electron の userData に置き、main プロセスがサーバー起動前に読み込んで env を上書きする**。変更操作は「コピー → 完全性検証 → 旧削除 → 設定書き込み → 自動再起動」の順で、失敗しても両側にデータが残る「二相コミット」を採用する。

- **設定ファイル位置**: `%APPDATA%\Yumeweaving\app-settings.json`(userData 直下)。**データディレクトリ本体には置かない**(変更対象と重なるため)
- **優先順位**: env(`YUMEWEAVING_DATA_DIR`) > app-settings.json の `dataDir` > 既定(`Documents\Yumeweaving`)
  - env を最優先にすることで、テスト(playwright)や将来の開発者操作を維持する
- **選択したフォルダが空でない場合**: そのフォルダ内に `Yumeweaving` サブフォルダを作成し、そこにデータを移す(ユーザーが「D:\backup を選んだのに中身が想定と違う」となる混乱を避けつつ、既存ファイルの上書き事故もゼロにする)

### 実装

#### G1-a. app-settings.json のスキーマと読み込み

```ts
// src/electron/appSettings.ts(新設)
interface AppSettings {
  dataDir?: string;                 // 絶対パス。未設定なら既定
  pendingCleanup?: string | null;   // 二相コミットの旧データ削除待ち
}
```

`main.ts` の `bootstrap()` の**冒頭**で以下を実行:

1. `readAppSettings()` を読み込み
2. `pendingCleanup` があれば `dataDir`(新しい方)を検証(`projects/` フォルダが存在するか)し、OK なら旧を再帰削除して `pendingCleanup: null` を保存
3. env `YUMEWEAVING_DATA_DIR` が未設定かつ `dataDir` があれば `process.env.YUMEWEAVING_DATA_DIR = dataDir` をセット
4. その後 `startServer` を呼ぶ

**NOTE**: `config.ts` の `DATA_DIR` はモジュール評価時に確定するため、上記のセットは `startServer` を import する前に完了させる。ESM の import 順で担保する(dynamic import で startServer を呼ぶ)。

#### G1-b. サーバー API

`src/server/routes/system.ts` に追加:

- `GET /api/system/data-dir` → `{ current: string, defaultPath: string, isUsingDefault: boolean }`
- `POST /api/system/data-dir/preview { targetPath }` → 検証結果のみ返す(ドライラン)
  - `{ resolvedPath, targetIsEmpty, hasFreeSpace, estimatedSize, sameAsCurrentDir, invalidReason? }`
  - `resolvedPath`: 空でないなら `<targetPath>/Yumeweaving`、空なら `<targetPath>` そのまま
  - `invalidReason`: 「現在の場所と同じ」「現在の場所の子孫にあたる」「書き込み不可」「同一ドライブでの空き容量不足」のいずれか
- `POST /api/system/data-dir/apply { targetPath }` → 実行し、成功時に自動再起動を予約
  - 実装は G1-c の順序どおり
  - 完了応答後、`onShutdownRequest` に相当する新フック `onRestartRequest` を呼ぶ(main が `app.relaunch()` → `app.exit(0)`)
  - `startServer` の `StartServerOptions` に `onRestartRequest?: () => void` を追加(既存の `onShutdownRequest` と同居)

#### G1-c. 移動処理の順序(二相コミット)

`src/server/services/dataDirMoveService.ts` を新設し、以下を実装:

```
1. resolvedPath を確定(空チェック → 空でなければ /Yumeweaving を付加)
2. resolvedPath が現在の DATA_DIR と同じ、または子孫なら拒否
3. 新側フォルダを作成(既存なら中身の projects/ が無いことを確認)
4. コピー: fs.cp(currentDataDir, resolvedPath, { recursive: true, errorOnExist: false })
   - 生成中や書き込み中を避けるため、コピー前に「移動中はロック」フラグを立て、generate-stream 系のルートは 503 で拒否する
5. 完全性検証:
   - 両側の projects/ 直下のフォルダ数が一致
   - 旧側で見つけた全ファイルが新側に存在(相対パスと size が一致するか)
   - config/credentials.json など機密ファイルも含めて欠落ゼロ
6. app-settings.json を { dataDir: resolvedPath, pendingCleanup: oldDataDir } で保存
7. 200 応答を返す
8. onRestartRequest() を呼ぶ(main が再起動)
9. 次回起動時、G1-a の 手順2 が旧側を削除して pendingCleanup をクリア
```

**失敗時の挙動**:

- 手順4(コピー)途中で失敗 → 新側を巻き戻し(再帰削除)、旧側は無傷。エラー応答
- 手順5(検証)で欠落 → 同上
- 手順6(設定書き込み)で失敗 → 新側を巻き戻し、エラー応答
- 手順9(旧削除)で失敗 → warn ログのみ。アプリは新側で正常起動し、`pendingCleanup` は残る(次回起動時に再試行)。ユーザーには「アプリ設定」画面で「旧データが残っています。手動で削除できます: `<パス>`」と表示する

**書き込みロックの実装**: `src/server/services/dataDirLock.ts` を新設し、`isLocked()` と `withLock()` を提供。`generationService`・保存系サービスは lock 中は書き込み拒否(既存 `withProjectWriteLock` の外側にもう1層)。

#### G1-d. クライアント UI

`src/client/components/AppSettingsModal.tsx` を新設(`ProjectList.tsx` のヘッダーに「アプリ設定」ボタンを追加)。内容:

- **保存先(現在値)**: フルパスを表示、コピーボタン
- **保存先を変更**: 「フォルダを選択」ボタン → Electron の `dialog.showOpenDialog({ properties: ['openDirectory'] })` 経由(server 側 `/api/system/data-dir/pick` → main に IPC 委譲 or renderer 経由の直接呼び出し。**renderer から直接 dialog を呼べないため、Electron の preload を今回この目的のためだけに追加するのは避け、サーバー経由で main に伝える** → 別スレッド Q: 現状 renderer↔main の直通はないので、サーバーが `dialog.showOpenDialog` を持つ Electron を import して呼ぶ形にする)
  - NOTE(現実的な折衷): サーバーは Node 側で動いており、`electron` 側のダイアログを直接呼ぶには preload か IPC が必要。今回は「フォルダのパスを手入力+『プレビュー』ボタン」で最小実装とし、フォルダ選択ダイアログは後日タスクとする(過剰実装を避ける)。手入力の詳細:
    - テキスト入力欄 + プレビューボタン
    - プレビューで `POST /api/system/data-dir/preview` を叩き、resolvedPath や空き容量、無効理由を表示
    - 「この場所に移動する」ボタンで `apply` → 進捗モーダル → 自動再起動
- **移動中**: モーダルで「コピー中(x / y ファイル)」を表示。実装は SSE を使うほど複雑にせず、`apply` レスポンスまでスピナーで OK(単一の作品なら数秒〜数十秒で終わる)
- **警告**: 「移動中はアプリを終了しないでください」

### 受け入れ条件

- **正常系**: 空の別ドライブを指定 → コピー・削除・再起動が通り、新側に作品データが揃っている。旧側が消えている。
- **サブフォルダ作成系**: 空でないフォルダを指定 → そのフォルダ内に `Yumeweaving` が作られ、そこにデータが入る。既存ファイルは無傷。
- **拒否系**: 現在の DATA_DIR / その子孫 / 書き込み不可な場所を指定 → 移動前に日本語エラーで拒否。
- **クラッシュ耐性**: 手順5(検証)完了前にアプリを強制終了しても、次回起動で旧側からそのまま起動できる。手順6(設定書き込み)後・手順9(削除)前にクラッシュしても、次回起動で pendingCleanup が処理され破綻しない。
- **書き込みロック**: 移動中に generate-stream を叩くと 503 が返り、生成が始まらない。
- **ユニットテスト**: dataDirMoveService の resolvedPath 算出、拒否条件、pendingCleanup の反映、をカバー。

## G2. 作品ショートカット(`作品一覧` フォルダ)

### 問題

作品本文の md ファイル(`<DATA_DIR>\projects\<projectId>\episodes\ep-*.md`)は Explorer で開こうとするとプロジェクト ID の羅列から目的の作品を探す必要があり、実質使えない。作品タイトルからワンクリックで本文 md にたどり着ける導線が必要。

### 方針

**`<DATA_DIR>\作品一覧\<作品タイトル>.lnk` を自動生成する**。Windows のショートカット(.lnk)なら Explorer から開くとエディタで本文が開き、ファイル自体は数百バイトなので同期・バックアップの負担も無視できる。

- **生成タイミング**: (a) 作品作成時、(b) 作品タイトル変更時、(c) データディレクトリ変更後(G1 の再起動後)に一括再生成
- **タイトルサニタイズ**: `\ / : * ? " < > |` を `_` に置換。前後空白トリム。空になったら `無題-<projectId 先頭8文字>`
- **重複回避**: 同じサニタイズ済み名の .lnk が既に存在し、かつ別プロジェクトを指している場合は末尾に `(2)`, `(3)` を付ける
- **タイトル変更後の旧 .lnk**: 消さない(仕様)
- **.lnk 生成方法**: PowerShell の `WScript.Shell` COM を Electron の main プロセスから呼ぶ

### 実装

#### G2-a. サービス層

`src/server/services/shortcutService.ts` を新設。主要関数:

- `writeShortcut(projectId, title): Promise<void>` — 1件生成
- `regenerateAllShortcuts(): Promise<void>` — 全作品分をまとめて生成(G1 移動後、または明示的な再生成ボタンから)

**PowerShell バッチ化**: `writeShortcut` を N 回叩くと PowerShell を N 回起動して数秒 × N かかる。**複数件生成時は1回の PowerShell 起動で全件を作る**スクリプトを標準入力経由で流し込む形にする。単発の作品作成時は `writeShortcut` 1件でも許容(PowerShell 起動 ~500ms は目立たない)。

```ps1
$ws = New-Object -ComObject WScript.Shell
$(foreach ($item in $items) {
  $lnk = $ws.CreateShortcut($item.LnkPath)
  $lnk.TargetPath = $item.TargetPath
  $lnk.Save()
})
```

`$items` は Node 側から JSON 文字列で渡し、PowerShell 内で `ConvertFrom-Json` する。

#### G2-b. フック

`src/server/services/projectService.ts` の以下にフックを追加:

- `createProject`(現在の関数名を確認して合わせる)の末尾で `writeShortcut(projectId, title)` を呼ぶ
- `updateProject` でタイトルが変わった場合に `writeShortcut(projectId, newTitle)` を呼ぶ

**エラー時の挙動**: ショートカット生成に失敗しても作品作成/更新自体は成功させる(warn ログのみ)。作品操作より優先度が低いため。

#### G2-c. `作品一覧` フォルダの初期化

- 起動時、`<DATA_DIR>\作品一覧` が無ければ作成する(`main.ts` の bootstrap 内、または server の起動時)
- **G1 の移動完了後(再起動後)**: 既存 `作品一覧` フォルダを空にしてから `regenerateAllShortcuts()` を呼び、ターゲットパスを新 DATA_DIR に合わせて作り直す

#### G2-d. ショートカットのターゲット

- ターゲットは `<DATA_DIR>\projects\<projectId>\episodes\<エピソード md ファイル>`
- **エピソード md がまだ存在しないプロジェクト**(作品作成直後、生成前)はショートカットを作らない or 「作品を開いてから最初の生成を実行してください」的なプレースホルダを作る → **作らない**を採用(存在しないファイルへの .lnk は Explorer からクリックすると赤×エラーになるため)
- 初回生成完了時に `writeShortcut` を再度呼ぶ(**追加フック**: `stateService` の episode 保存箇所、または generate 採用フロー)

### 受け入れ条件

- 新規作品を作って初回シーンを採用 → `作品一覧\<タイトル>.lnk` が生成され、ダブルクリックで本文 md がエディタで開く
- タイトルを「A」→「B」に変更 → `B.lnk` ができる。`A.lnk` は残る(仕様)
- 使用不可文字を含むタイトル(例: `シリーズ:A/1`)→ `シリーズ_A_1.lnk` が作られる
- 空タイトル / 制御文字のみ → `無題-<8桁>.lnk` にフォールバック
- 別の作品を同じサニタイズ済みタイトルで作った → `<タイトル> (2).lnk` が作られる
- データディレクトリを別ドライブに移動 → 再起動後、全 .lnk がリンク切れせず新パスを指す
- ショートカット生成失敗(権限エラー等)→ 作品作成 API は成功、warn ログのみ

## G4. 続き生成で最上部へスクロール

### 問題

続き生成の直後は本文の末尾(生成ボタン付近)に視点が残るため、頭から読み直すには手動スクロールが必要。

### 方針

**生成開始時に `window.scrollTo({ top: 0, behavior: 'smooth' })` を実行する**。生成完了時ではなく開始時に動かすのは、(a) ストリーミングで生成中に上から読み進められる、(b) 完了時にスクロールすると読んでいた位置から強制的に戻され不快、という2点による。

### 実装

`Reader.tsx:142` の `handleGenerate` 冒頭に追加:

```ts
async function handleGenerate(mode: 'continue' | 'regenerate', ...) {
  window.scrollTo({ top: 0, behavior: 'smooth' });
  // (以下、既存処理)
}
```

**NOTE(対象範囲)**: `mode === 'continue'` だけでなく `regenerate` でも上部に戻す(書き直しでも「頭から見比べる」ニーズは同じ)。

### 受け入れ条件

- 続き生成ボタンを押した瞬間、リーダーが最上部までスムーズにスクロールする
- 書き直しボタンでも同様に動く
- スクロール後、生成テキストが上から流れて見える

## 5. 実装順の推奨

1. **Phase 1(自己完結・低リスク)**: G3 → G4
   - G3 は共通コンポーネントを先に作る(以降のUI追加でも再利用可能)
   - G4 は1行変更
2. **Phase 2(基盤変更)**: G1
   - 二相コミット・ロック・main の bootstrap 改修が要になる。テスト厚めに
3. **Phase 3(G1 の派生)**: G2
   - G1 の移動処理から `regenerateAllShortcuts()` を呼ぶ形にするため、G1 実装後に着手するのが自然
   - ただし G2-b(作品作成/更新フック)だけは G1 に依存しないので、Phase 1 と並行でもよい

| 2 | 選択したフォルダが空でない場合 | サブフォルダ `Yumeweaving` を作成してその中に |
| 3 | ショートカット命名衝突時 | `(2)`, `(3)` を末尾に付加 |
| 4 | タイトル変更後の旧ショートカット | 削除しない(仕様) |
| 5 | 生成中インジケータ | スピナー(回転)+ ラベル(パルス)の合わせ技 |
| 6 | 続き生成スクロールの発火タイミング | 開始時(完了時ではない) |

## 7. 作者確認が必要な残点

- G1 のフォルダ選択 UI: 「フォルダ選択ダイアログ」か「テキスト入力+プレビュー」か。設計書は preload 追加を避けるため後者を採用しているが、Electron ダイアログを好むなら preload を1つ追加する追加工数(半日程度)を許容するか判断が必要。
- G2 の「エピソード md がない状態」でのショートカット扱い: 設計書は「作らない」を採用。初回生成前でも作りたい場合は代替(プロジェクトフォルダを開く .lnk など)を検討。
