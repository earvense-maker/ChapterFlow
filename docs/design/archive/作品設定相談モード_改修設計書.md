# 作品設定相談モード 改修設計書

## 0. 背景

`作品設定相談モード_設計書.md` の Phase 1〜2 相当は実装済みである。本書は、実装レビューで見つかった修正点・改善点・追加機能をまとめた改修設計書であり、これ単体で実装作業へ渡せることを目的とする。

対象コード:

- `src/server/routes/setupSessions.ts`
- `src/server/services/setupSessionService.ts`
- `src/server/services/setupDraftPatchService.ts`
- `src/server/services/setupPromptBuilder.ts`
- `src/server/services/setupCommitService.ts`
- `src/server/services/storageService.ts`(setup系関数)
- `src/shared/types.ts`(Setup系型)
- `src/client/clientApi.ts`(setupSessions系メソッド)
- `src/client/components/SetupWorkspace.tsx`
- `src/client/App.tsx`(setup view接続)
- `tests/unit/setupSessionService.test.ts` ほか setup系ユニットテスト

前提となる現状仕様(変更しないもの):

- セッションは `data/setup-sessions/{sessionId}.json` に保存。`safeWriteJson` / `readJsonFile` / `SAFE_PATH_SEGMENT` 検証を使う。
- 楽観ロック: `revision` 不一致は 409 + 最新 `session` を返す。
- セッション単位の直列化: `setupSessionService` 内の `withSessionLock`(プロセス内mutex)。
- LLM呼び出しは `OpenAIAdapter` / `GeminiAdapter` / `DeepSeekAdapter` を `adapterMap` 経由で使用。呼び出し前に `reloadCredentials()`。
- エラーは `SetupServiceError`(code / retryable / status / session)でルートへ伝搬し、`handleSetupError` がJSONで返す。

互換性方針:

- `data/setup-sessions/` に既存セッションJSONが存在する。**新規フィールドはすべてオプショナル**とし、読み込み時にサービス層で欠損を補正する。`schemaVersion` は 1 のまま。
- 破壊的なAPI変更(ロックAPIの置き換え、commitの2段階化)は、このアプリが単一ユーザー・同梱クライアント前提のため許容する。ただし `clientApi.ts` とテストを同時に更新し、旧エンドポイントは残さない。

## 1. 修正項目(バグ・データ喪失系)

### F1. commit時にNG・好みが消える問題の修正

**問題**: `normalizeSetupCommitData`(`setupCommitService.ts`)は、LLMが `memories` を返さないと空配列のまま保存する。`buildFallbackWorldText` にも `draft.ng` / `draft.tone` は含まれないため、相談で伝えたNGが本編生成へ一切引き継がれない経路がある。

**方針**: LLMの出力に依存せず、`draft.ng` と `draft.tone` は必ず memories に反映する(マージ方式)。

**実装**:

`normalizeSetupCommitData` 内で `normalizeMemories(raw.memories, now)` の結果に対して以下を行う関数 `mergeDraftPreferenceMemories(memories, draft, now)` を追加する。

1. `draft.ng` の各項目について、既存 memories に同一内容(空白正規化 + 小文字化で比較。`setupDraftPatchService` の `normalizeComparableText` と同じ規則)が無ければ `{ type: 'negative', importance: 'high', content }` の Memory を追加する。
2. `draft.tone` の各項目について、同様に無ければ `{ type: 'preference', importance: 'medium', content }` を追加する。
3. ID・時刻・`status: 'active'`・`source: 'manual'` などは既存の `normalizeMemory` と同じ形に揃える。
4. 上限24件は維持する。**優先順位: LLM出力の memories → ng由来 → tone由来** の順で詰め、溢れた分は捨てる…ではなく、**ng由来は必ず入れる**。上限に達している場合は importance が低いLLM由来メモリ(low → medium の順)を末尾から落として ng を入れる。

**受け入れ条件**:

- LLMが memories を返さなくても、`draft.ng` の全active項目が `type: 'negative'` / `importance: 'high'` で `memories.json` に保存される。
- LLMが ng と同内容の memory を返した場合、重複追加されない。
- ユニットテスト: memories空 + ng/toneあり、memories上限超過時のng優先、重複排除。

### F2. LLMが素の日本語で返したときに返答が失われる問題の修正

**問題**: `parseChatResult`(`setupSessionService.ts`)はJSON抽出に失敗すると、モデルの本文を捨てて定型文 `UNREADABLE_CHAT_REPLY` に置き換える。モデルが普通の文章で答えた場合、内容が丸ごと失われる。元設計書6.3は「visibleReplyが取れる場合は表示」「失敗が続けば通常の文章返答モードへフォールバック」を要求している。

**方針**: JSONとして読めない場合、生テキストを平文返答として表示する。ただし壊れたJSONの断片は見せない。

**実装**: `parseChatResult` を以下のロジックに変更する。

1. `parseJsonObject(text)` が成功 → 現行どおり。
2. 失敗した場合:
   - コードフェンス(````` ```json ````` 等)を除去しtrimしたテキストを `plain` とする。
   - `plain` が空 → 現行の定型文フォールバック。
   - `plain` に `"draftPatch"` または `"visibleReply"` の文字列が含まれる → 壊れたJSONの出力とみなし、現行の定型文フォールバック(生JSONをユーザーに見せない)。
   - それ以外 → `{ visibleReply: plain, draftPatch: null, suggestedActions: [] }` を返す(メモ更新なしで平文をそのまま表示)。

**受け入れ条件**:

- モデルが自然文だけを返したとき、その文章が assistant メッセージとして保存・表示される。draftは変化しない。
- 壊れたJSON(`{"visibleReply": "...` の途中切れ等)を返したとき、生JSONは表示されず定型文になる。
- ユニットテスト: 平文 / 壊れJSON / 空文字 / フェンス付き平文 の4ケース。

### F3. 入力検証の穴(500エラー)の修正

**問題**: `sendSetupMessage` は `body.message.trim()` を型チェックなしで呼ぶ。message欠落・非文字列のリクエストで TypeError → 500 になる。

**実装**:

- `sendSetupMessage`: `typeof body.message !== 'string'` なら `SetupServiceError('メッセージを入力してください。', 'invalid_message', false, 400)`。加えて `message.length > 4000` は `'invalid_message'` の400で拒否する(プロンプト肥大防止)。
- `assertRevision` の前に `typeof body.revision !== 'number' || !Number.isInteger(body.revision)` を `'invalid_request'` の400で拒否する(`sendSetupMessage` / `updateSetupDraft` 共通)。
- `updateSetupDraft`: `body.draft` が `isRecord` でなければ 400。
- 新設エンドポイント(後述)も同じ検証方針を適用する。

**受け入れ条件**: 不正bodyで必ず400(500にならない)。ユニットテストで message欠落 / message非文字列 / revision非数値 / draft非オブジェクト を確認。

### F4. LLM patch追加時のIDをサーバー採番に統一

**問題**: `applySetupDraftPatch` の追加系(`confirmedAdd` / `candidatesAdd` / `undecidedAdd` / `charactersAdd`)は、LLMが渡した `id` をそのまま採用する(`normalizeTextItem` 等が `record.id` を読む)。既存項目と同じID、または `draft.world` のようなセクションロックパスと同名のIDを渡されると、重複IDやロック誤判定(`isPathLocked` は前方一致も見る)、`archiveIds` の誤動作が起きうる。

**実装**:

- `applySetupDraftPatch` から呼ばれる追加経路では、入力の `id` を無視して常に `generateTimestampId(prefix)` で採番する。実装としては `normalizeTextItem` / `normalizeCandidate` / `normalizeUndecided` / `normalizeCharacter` に `forceNewId: boolean` 引数(または追加専用ラッパー)を設け、patch追加時のみ強制採番する。
- 手動編集(`PUT /draft` → `normalizeSetupDraft`)では既存IDを保持する。ただし防御として `normalizeSetupDraft` 内で:
  - 同一配列内のID重複は2件目以降を再採番する。
  - `draft.` で始まるID、`SAFE_PATH_SEGMENT` 相当(`/^[A-Za-z0-9_-]+$/`)を満たさないIDは再採番する。
  - IDが空の項目は採番する(後述 I2 の手動追加はID省略で送る)。

**受け入れ条件**:

- LLM patchで `id` を指定しても無視される(ユニットテスト: 既存IDと同じidを持つ confirmedAdd が別IDで追加される)。
- `normalizeSetupDraft` がID重複・不正IDを再採番する。
- 既存の重複排除(テキスト同一)テストが引き続き通る。

### F5. 失敗後の再試行APIの追加(userメッセージ二重化の防止)

**問題**: LLM呼び出し失敗時、ユーザー入力はセッションへ保存済み(設計どおり)だが、UIに再試行手段がなく、同文を再送すると user メッセージが二重になる。

**実装**:

サーバー:

- `sendSetupMessage` から「保存済みセッションの末尾user発言に対して応答を生成し、assistantメッセージとdraftを更新する」部分を `runChatTurn(session: SetupSession): Promise<SetupMessageResponse>` として切り出す。`buildSetupChatPrompt` へは `session.messages` 末尾のuserメッセージ内容を `userMessage` として渡す(プロンプト構造は不変)。
- 新設 `POST /api/setup-sessions/:id/retry`(bodyなし、または `{ revision }` を受けて検証):
  - `requireActiveSession` → 末尾メッセージが `role: 'user'` でなければ `'nothing_to_retry'` の400。
  - userメッセージを追加せず `runChatTurn` を実行。成功時のレスポンス形は `POST /messages` と同一(`SetupMessageResponse`)。
  - 失敗時も `POST /messages` と同じエラー形(lastError更新を含む)。
- `withSessionLock` 配下で実行する。

クライアント(`SetupWorkspace.tsx`):

- `send()` 失敗時、`reloadLatestSession` 後に最新セッションの末尾が user メッセージなら「応答を再試行」バナー(またはチャット末尾のボタン)を表示する。クリックで `api.retrySetupMessage(sessionId)` を呼び、成功時は `send()` 成功時と同じ状態更新を行う。
- 再試行可能な状態では通常送信も引き続き許可する(ユーザーが文を変えて送り直したいケース)。その場合バナーは消す。

`clientApi.ts` に `retrySetupMessage: (id) => request<SetupMessageResponse>(...)` を追加する。

**受け入れ条件**:

- LLM失敗 → retry で userメッセージが増えずに assistant 応答が付く。
- 末尾が assistant のセッションへ retry すると400。
- ユニットテスト: retry成功 / nothing_to_retry / retry中のLLM失敗(lastError更新)。

### F6. ロック切り替えのアトミック化

**問題**: クライアントの `toggleLock` は「draft更新→lock追加」「lock削除×N→draft更新」を逐次呼ぶため、途中失敗で `item.locked` と `locks` が食い違う。また revision が複数回進むため409も出やすい。

**実装**:

サーバー:

- 新設 `PUT /api/setup-sessions/:id/lock-state`。body: `{ path: string, locked: boolean, revision: number }`。
- `setupSessionService.setLockState(sessionId, body)` を追加し、1回の読み書きで以下をまとめて行う:
  1. `assertRevision`。
  2. `path` が draft内項目ID(`confirmed` / `candidates` / `undecided` / `characters` を検索)に一致すれば、その項目の `locked` を `body.locked` に設定する。
  3. `locks` 配列を更新する: `locked: true` なら `addLocks(..., [path], 'user_locked', now)`、`false` なら `lock.path === path` のエントリをすべて除去する。
  4. revisionを1回だけ加算して保存。
- レスポンス: `{ session, revision }`。
- 既存の `POST /:id/locks` と `DELETE /:id/locks/:lockId` は**削除**し、`clientApi.ts` の `addSetupLock` / `removeSetupLock` も削除する。

クライアント:

- `toggleLock` を `api.setSetupLockState(sessionId, { path, locked, revision })` の単発呼び出しに置き換える。`mutateDraftLocked` コールバック機構は不要になるため削除する。

**受け入れ条件**:

- 固定/解除が1リクエストで完結し、`item.locked` と `locks` が常に一致する。
- セクションパス(`draft.world` 等)のロックも同エンドポイントで動く(項目IDに一致しない場合は locks のみ更新)。
- ユニットテスト: 項目ロック / セクションロック / 解除 / revision不一致409。

## 2. 改善項目(設計書との差分解消)

### I1. 確定・候補・未確定の間の移動操作

**問題**: 元設計書3.2の「未確定へ移す」「確定へ移す」が未実装。またLLMが `confirmedAdd` に自己申告 `source: 'user'` で何でも入れられる(検証はプロンプト頼み)。

**実装**:

サーバー(patch適用の防衛):

- `applySetupDraftPatch` で、LLM patch(`source === 'llm'` として適用されるとき)の `confirmedAdd` のうち `source` が `'user'` でない項目は、confirmed ではなく **undecided への追加として扱う**(text をそのまま `undecidedAdd` 相当に回す。reason は `'LLM提案のため未確定として保留'` を設定)。
- プロンプト(`buildSetupChatPrompt` の【重要】)に「confirmedAdd に入れられるのはユーザーが明言した内容だけで、その場合 source は必ず 'user' にする」を追記する。

クライアント(移動UI):

- 移動はすべて既存の `PUT /draft`(`saveDraft`)で実現する。サーバーAPIの追加は不要。
- 「決まってきたこと」の各行に「未確定へ」ボタン: confirmed から該当項目を除去し、同じ text で undecided に追加(id は**新規採番させるため省略**、`source: 'manual'`)。`manualEditPaths` は新idが不明なため `[]` でよい。
- 「未確定」の各行に「確定へ」ボタン: 逆方向。
- 「候補」の各行に「確定へ」ボタン: candidate を archived にし、`title`(summaryがあれば `title: summary` 形式で連結した1行テキスト)を confirmed に追加。

**受け入れ条件**:

- LLM patchの `source: 'llm'` な confirmedAdd が undecided に落ちる(ユニットテスト)。
- UI操作で confirmed ⇄ undecided、candidate → confirmed の移動ができ、リロード後も保持される。

### I2. メモ項目の手動追加UI

**問題**: 各セクションは0件だと非表示で追加手段がない。特にNGはLLM任せでしかも入らないことがある。

**実装**(`SetupWorkspace.tsx` のみ):

- 全セクション(作品の核以外の8セクション: 決まってきたこと / 候補 / 未確定 / 人物 / 世界観 / 関係性 / 好み・文体 / NG / 冒頭候補)を**0件でも常に表示**する。0件時は「まだありません」プレースホルダを出す。
- 各セクションヘッダに「+追加」ボタンを置く。クリックでそのセクション末尾に空の編集行(既存の `EditableTextRow` 等と同じ見た目、保存前はローカルstateのみ)を出す。
- 保存時は `saveDraft` で該当配列に追加する。id は省略し(F4により)サーバー採番。`source: 'manual'`。candidates は title+summary、characters は既存の編集フォームと同項目。文字列セクションはそのままpush。
- 未保存の追加行は dirty 扱い(`markDraftDirty`)にし、既存の「未保存の変更があります」ガードに乗せる。キャンセルボタンで破棄できる。

**受け入れ条件**: NGが0件の状態からユーザーが手動で追加でき、commit後 `memories.json` に反映される(F1と結合)。

### I3. 試し書きのセッション保存とcommit・相談への反映

**問題**: `previewText` は使い捨てで、元設計書3.5「試し書きは最終変換時に openingSeeds や tone の根拠として使う」が未実装。

export interface SetupPreviewRecord {

### I4. 変換前レビュー画面(2段階commit)

**問題**: 元設計書7.4の確認画面がなく `window.confirm` のみ。LLMが決めたタイトル・world等を保存前に編集できない。

**実装**:

共有型:

```ts
export interface SetupCommitPlan {
  project: {
    title: string;
    outputLength: number;
    activePresetIds: Partial<ActivePresets>;
  };
  worldText: string;
  characters: Character[];
  memories: Memory[];
  storyState: StoryState;
  customSystemPrompt: string;
}
```

(`NormalizedSetupCommitData` を `SetupCommitPlan` ベースに再構成する。`CreateProjectBody` への詰め替えはcommit実行時に行う。)

`SetupSession` に `commitPlan?: { plan: SetupCommitPlan; createdAt: string } | null` を追加。

サーバー:

- 新設 `POST /api/setup-sessions/:id/commit-plan`(bodyなし):
  - 現行 `commitSetupSession` の前半(LLM呼び出し → `parseJsonObject` → `normalizeSetupCommitData`)を実行し、結果を `session.commitPlan` に保存(revision加算)して `{ plan, session, revision }` を返す。プロジェクトはまだ作らない。
  - JSON破損時は現行どおり `'invalid_commit_json'` 503。
- `POST /api/setup-sessions/:id/commit` を変更。body: `{ plan: SetupCommitPlan, revision: number }`:
  - ユーザー編集済みplanを受け取り、**再度 `normalizeSetupCommitData` 相当の正規化**(プリセットID正規化・上限・ID/時刻補正。LLM呼び出しなし)を通す。F1のng/toneマージもここで適用する。
  - 正規化済みデータで `projectService.createProject` → `writeMemories` → `writeStoryState` → セッション `committed` 化。補償削除(`deleteProjectDir`)は現行維持。
  - `status === 'committed'` の再commitは現行どおり既存projectIdを返す。
- 正規化ロジックはLLM出力用とユーザー編集plan用で共通化する(入力が `unknown` である点は同じ)。

クライアント:

- 「この内容で作品を作る」→ `commit-plan` を呼び、**レビュー画面**(`SetupCommitReview` コンポーネント。setup-main を置き換える全面表示)を出す:
  - 編集可能: タイトル(input)、worldText(textarea)、characters(既存の人物編集行を流用)、memories(type/importance select + content textarea + 削除/追加)、storyState.currentSituation(文字列リスト)、openThreads(summary編集・削除)、customSystemPrompt(textarea)。
  - フッター: 「この内容で作成」(→ `commit` にplanとrevisionを送る)/「相談に戻る」(破棄してチャットへ)/「作り直す」(再度 `commit-plan`)。
- `window.confirm` は撤去する。
- `clientApi.ts`: `createSetupCommitPlan(id)` 追加、`commitSetup(id, { plan, revision })` にシグネチャ変更。

**受け入れ条件**:

- commit-plan → レビュー編集 → commit で、編集内容(タイトル変更等)が `project.json` / `world.md` / `memories.json` / `story-state.json` に反映される。
- 不正なplan(未知プリセットID、壊れたstoryState)を送ってもサーバー正規化で安全な値に補正される(ユニットテスト)。
- commit失敗時にプロジェクトが残らない(現行テスト維持)。

### I5. APIキー未設定の早期検出と設定導線

**問題**: キー未設定でも相談画面に入れて、最初の送信で初めて503になる。元設計書6.3は「相談を開始せず設定画面への導線を出す」。

**実装**:

サーバー:

- `GET /api/models/providers`(`src/server/routes/models.ts`)を拡張し、各プロバイダに `hasApiKey: boolean` を付ける。`credentialService.loadCredentials()`(または `listStoredProviders()`)で保存済みプロバイダ集合を取り、`ModelProviderInfo` にマージする。共有型 `ModelProviderInfo` に `hasApiKey?: boolean` を追加。

クライアント(`SetupWorkspace.tsx` / `App.tsx`):

- `SetupWorkspace` に `onOpenSettings: () => void` propを追加し、`App.tsx` で `setView('settings-app')`(既存のアプリ設定view)へ接続する。
- セッションロード時に providers を取得済みなので、**現在セッションのprovider**の `hasApiKey` が false の場合、チャット入力の上に警告バナー「APIキーが未設定です。設定画面で入力してください。[設定を開く]」を表示し、送信・試し書き・作品化ボタンを無効化する。

**受け入れ条件**: キー未設定プロバイダ選択時に送信不可+導線表示。キー設定後(設定画面から戻る)に再判定される。

### I6. セッションのライフサイクル管理(放棄・削除・一覧)

**問題**: `abandoned` ステータス未使用、削除APIなし、`data/setup-sessions/` が溜まり続ける。「新しく始める」でも旧セッションはactiveのまま残る。

**実装**:

ストレージ(`storageService.ts`):

- `deleteSetupSession(sessionId: string): Promise<void>` を追加(`fs.rm(path, { force: true })`。IDは `setupSessionJsonPath` 経由で検証)。

サーバー:

- ユニットテスト: abandon / delete / abandon済みへの送信400。

### I7. 長い相談での文脈維持(会話サマリー)

**問題**: チャットプロンプトは直近12メッセージ+draftのみで、draftに載らないニュアンス(却下理由・経緯)が消える。

**方針**: 追加のLLM呼び出しはしない。チャットLLMの出力JSONに要約フィールドを追加し、会話が長いときだけ更新させる。

**実装**:

- `SetupSession` に `conversationSummary?: string` を追加(欠損は `''` 補正)。
- `buildSetupChatPrompt`:
  - 『【これまでの相談の要約】』として `conversationSummary`(空なら省略)を含める。
  - 出力形式JSONに `conversationSummary` フィールドを追加し、【重要】に「メッセージ数が12を超えている場合、これまでの流れ(採用・却下した方向と理由、ユーザーの好みの傾向)を800字以内で conversationSummary に更新して返す。12件以下なら省略してよい」を追記する。
- `parseChatResult` の戻り値に `conversationSummary: string | null` を追加(`asString` で取得、空なら null)。
- `sendSetupMessage` / `runChatTurn`: 返ってきた `conversationSummary` が非空なら 2000字でtruncateしてセッションへ保存する。
- commit プロンプト(`buildSetupCommitPrompt`)にも要約を含める(直近24メッセージより前の文脈の補完)。

**受け入れ条件**: 13件目以降のターンで要約が保存され、以後のプロンプトに含まれる(promptBuilder / parseChatResult のユニットテスト)。

## 3. 追加機能

### A1. 候補カードの定型操作

**実装**(`SetupWorkspace.tsx` のみ。送信はすべて既存 `send()` を使う):

- 各候補行に:
  - 「これで進める」→ `send(\`候補「${title}」で進めたい。\`)`
  - 「混ぜる」→ 候補の複数選択トグル(選択状態はローカルstate)。2件以上選択中はセクション上部に「選択した候補を混ぜる」ボタンを表示し、`send(\`候補「A」と「B」を混ぜた方向にしたい。\`)` を送る。送信後に選択解除。
- 候補セクションヘッダに「別の候補をもう一度」→ `send('今の方向とは少し違う候補を、もう一度いくつか出して。')`
- いずれも `busy || hasUnsavedDraftEdits` で無効化(既存の suggestedActions ボタンと同じガード)。

### A2. 相談チャットのストリーミング(SSE)

**方針**: 出力プロトコルを「平文の返答 → 区切りマーカー → patch JSON」に変更し、マーカー前のテキストを逐次配信する。マーカーが無い場合はF2の平文フォールバックに自然に落ちる。

**実装**:

プロンプト(`buildSetupChatPrompt`)の出力形式を変更:

```text
(ユーザーへ見せる返答を平文で書く)

===DRAFT_PATCH===
{ "draftPatch": {...}, "suggestedActions": [...], "conversationSummary": "..." }
```

- 既存の「JSONオブジェクトだけにする」指示を上記2部形式の指示へ差し替える。
- `parseChatResult` を新形式対応にする:
  1. `===DRAFT_PATCH===` があれば、前半trimを `visibleReply`、後半を `parseJsonObject` で解析(壊れていれば「メモ更新なし」)。
  2. マーカーが無ければ: 全体がJSONとして読めて `visibleReply` を持つ → 旧形式として処理(後方互換)。読めなければ F2 の平文フォールバック。

サーバー:

- アダプタのストリーミング生成(既存の本文生成ストリームで使っているアダプタAPI。`generationService.generateSceneStream` が使う経路を確認して流用)を使う `runChatTurnStream` を追加する。
- 新設 `POST /api/setup-sessions/:id/messages/stream`。リクエストbodyは `POST /messages` と同じ。レスポンスはSSE(`text/event-stream`。ヘッダ・`send(event, data)` ヘルパーは `src/server/routes/generate.ts` の実装に合わせる):
  - `event: delta` `data: {"text": "..."}` — マーカー検出前のテキスト差分のみ流す。受信バッファ上で `===DRAFT_PATCH===` を監視し、マーカー以降は配信しない(マーカーが差分境界をまたぐ場合に備え、末尾15文字程度を保留してから配信する)。
  - `event: result` `data: SetupMessageResponse` — 完了時に全文をparseし、セッション保存後の最終結果を流す。
  - `event: error` `data: { error, code, retryable, session }` — 失敗時。
  - クライアント切断(`req.on('aborted')` / `res.on('close')`)でアダプタ呼び出しを中断する(既存generate routeのAbortControllerパターンに合わせる)。中断時はuserメッセージのみ保存済みの状態で終わる(F5のretryで回復可能)。
- 非ストリーミングの `POST /messages` は残す(アダプタがstream未対応の場合のフォールバック、およびテスト用)。

クライアント:

- `clientApi.ts` に `sendSetupMessageStream(id, body, handlers)` を追加(`fetch` + `ReadableStream`。SSE解析は `src/server/utils/sse.ts` と同等のパーサをクライアント用に実装するか、既存クライアントのストリーミング受信実装があればそれに合わせる)。
- `SetupWorkspace`: 送信時、まずstreamエンドポイントを使う。`delta` を受けるたびに末尾に仮のassistantメッセージ(ローカルのみ)を伸ばして表示。`result` で正式なセッションに置き換える。`error` は現行のエラー処理と同じ。
- 送信中に「中断」ボタンを表示し、`AbortController.abort()` で切断 → retryバナー表示(F5)。

**受け入れ条件**:

- 返答が逐次表示され、完了後のセッション状態は非ストリーミング版と同一。
- マーカー以降のJSONが画面に一切表示されない。
- 中断 → retry で復帰できる。
- 旧形式JSON応答(マーカーなし)でも動く。
