## 2. 見てほしい不変条件

コードだけでは意図が読みづらい箇所。**「なぜこう書いたか」の1〜2行のヒント**だけを添える。判断はレビュアーに委ねる。

### 2.1 revision 遷移(設計書 3.4)

`roleplaySessionService.ts:687` `beginTurn` と `commitTurn` の間で:
- **messages-stream**: 開始 `R` → user 保存 `R+1` → character コミット `R+2`
- **regenerate-stream**: 開始 `R` → 保存前は据え置き → コミット `R+1`
- **派生更新(要約完了)**: revision を進めない

ねらい: 「保存済みセッションが正、ストリーミング中は暫定」の境界を revision で表現する。生成中に別タブから来た send/archive を早期に 409 で弾ける。

見てほしい: この遷移がすべてのエラーパス(ストリーム途中失敗、hard cap 到達、abort、要約失敗)で崩れないか。特に `roleplaySessionService.ts:856` `commitTurn` の期待 revision 検査。

### 2.2 hard cap 到達を「正常終了」扱いする理由

`roleplaySessionService.ts:520〜540` あたり。目安字数の 2倍 or 600 のうち大きい方に到達したら upstream を abort するが、`hardCapReached` フラグを立てて **error イベントではなく done で保存**する。

ねらい: 「モデルが指示無視して長文吐いた」時にユーザー入力を破棄したくない。600字分は使える形で保存される方が体験がいい。

見てほしい: この境界で「保存はされたが event.type が error になった」ような矛盾が起きないか。abort 経路と finishReason='error' 経路が混ざりやすい。

### 2.3 contextSnapshot の意図的な固定(設計書 3.1)

`roleplaySessionService.ts:273` `buildContextSnapshot`。セッション作成時にキャラ・世界観・customSystemPrompt をコピー。**その後の設定編集は既存セッションに反映しない**。

一方、`project.roleplayOutputChars` と NG 表現は毎ターン最新を読む(`roleplaySessionService.ts:786〜802`)。**snapshot と動的の使い分けを恣意的にやっていないか**が主な懸念点。

判断根拠: 「キャラの人格変化は会話を別物にする、字数と NG は運用パラメータで会話を壊さない」という仕分け。ユーザーへの通知 UI はまだない。

### 2.4 要約カーソル(summaryThroughMessageId)と stale ジョブ検知(設計書 3.5)

`roleplaySessionService.ts:1054` `startBackgroundSummary` → `runBackgroundSummary`。
- スナップショット段階で `summaryThroughMessageId` を記録
- mutex を保持せず要約を生成(応答レイテンシに乗せないため)
- マージ段階で mutex 内、`latest.summaryThroughMessageId === snapshot.summaryThroughMessageId` の時だけ保存

ねらい: 二重要約を防ぎつつ、要約中に別経路が要約を進めた場合はこちらを捨てる。派生データなので失っても正しさに影響しない。

見てほしい: このスナップショット/マージのパターンで、要約中に **削除やアーカイブが走った** ときの整合性(現状 archived セッションは `beginTurn` で弾かれるが要約側のガードは弱め)。

### 2.5 SSE のコミット境界(設計書 3.6)

`routes/roleplay.ts:99〜170` `runStreamRoute`。
- ジェネレータの `.next()` を先に呼び、beginTurn の失敗を検出する
- 成功したらヘッダーを送る(以降のエラーは event になる)
- ヘッダー送信前のエラー = JSON、送信後 = SSE error event

ねらい: クライアントが「JSON失敗 → 通常エラー扱い」「SSE中の失敗 → 再同期して再試行」を区別できる。

見てほしい: `.next()` を先に呼ぶことで、beginTurn 内の副作用(user 保存)がヘッダー送信前に完了する順序保証が意図通りか。

## 3. 私が下した判断で他の選択もあり得たもの

トレードオフだけ中立に書く。答えは書かない。

### 3.1 要約タイミング: 応答完了後の非同期 vs 送信前の同期

**採用**: 応答完了後にサーバー側で非同期(`runOutsideDataDirWrite` の中)。予算超過時のみ同期でフォールバック。

**別案**: 送信前に毎回同期チェック。レイテンシは増えるが挙動は単純。

### 3.2 モデル設定: セッション作成時 snapshot vs 動的取得

**採用**: `session.model` にプロバイダ/モデル名を保存(セッション作成時 snapshot)。**contextSnapshot とは別のフィールド**で持つ。

**別案**: 毎ターン `project.activeModelProvider/Name` を読む。設定変更が既存会話にも反映されるが、会話中にモデルが変わると口調が急変する。

### 3.3 NG 表現: 共通ストアを roleplay にも注入 vs 独立辞書

**採用**: 既存 `expressions.json` を共通で使い、`resolveBannedExpressions` を roleplay プロンプトの末尾に注入。ユーザー登録した NG が本編・会話両方に効く。

**別案**: roleplay 用の独立辞書を持つ。分離できるが「同じ言い回しを別々に登録する」摩擦が出る。

### 3.4 応答字数: プロンプト内の目安字数 + max_tokens cap

**採用**: `outputLength` を(1) プロバイダ API の max tokens 経路 と (2) システムプロンプトの `「{n}字程度に収める」` 埋め込み の両方に流す。hard cap = `max(600, target*2)` で abort。

**別案**: max_tokens だけで制御。プロンプトが単純になるが、モデルが cap を「目標」と誤解しない。

### 3.5 「1〜3文」制約の撤去(直近のユーザーフィードバック反映)

**採用**: 文数制約を外し、括弧書き動作の例 + 目安字数のみで制御。

**別案**: 文数制約を残し、動作は独立行で許容。より予測可能だが「セリフ+動作」の自然な混在が難しい。

### 3.6 モデル生成共通層の抽出範囲

**採用**: `modelGenerationService.ts` は「adapter 解決 + credential reload + generate/stream の統一エラー変換」だけ。penalty リトライや banned expressions は本編生成側(`generationService.ts`)に残した。

**別案**: 本編生成の共通ロジックも吸い上げる。テストが増えるが再利用は増える。

## 5. 特に確認してほしい観点

自分でも不安の残る箇所を優先順に。

1. **mutex 内で LLM I/O を待たない設計**: `beginTurn` は mutex 内で user 保存まで、`commitTurn` は再取得。生成本体は mutex を持たないが、同じセッションへの他操作は `generationInFlight` フラグで弾く。この構造で **プロセス停止時に in-flight が消えて復旧できる** かの想定が妥当か(`roleplaySessionService.ts:299` 前後)。

2. **hard cap 到達時の保存パス**: `hardCapReached = true` の分岐(`roleplaySessionService.ts:520〜540`)が、後続の abort 判定・finishReason チェックを **どれも「正常終了」として通過** することが読みづらい。ここの制御フローに漏れがないか。

3. **`SetupSession.purpose` が undefined の後方互換**: 既存の novel setup セッションを開いたときに正しく `'novel'` として動くか。`normalizeStoredSession` (`setupSessionService.ts` 内)と `SetupSessionSummary.purpose` の境界。

4. **`normalizeCharacterForStorage` を全書き込み境界で通したか**: `projectService.createProject`, `routes/settings.ts` の PUT characters, `setupCommitService` の3経路で呼んでいる。**それ以外の書き込み経路(refine 適用など)を見落としていないか**。特に `refineChatService` の `character-update` パッチ適用時に normalize されていない可能性がある。

5. **クライアント側の 409 復旧**: SSE error 受信時に GET で再同期 → 入力欄は保持、というフローが `RoleplayWorkspace.tsx` の各エラーハンドラで一貫しているか。特に regenerate 失敗時に旧 character メッセージが消えていないか。

6. **選択→NG 登録の UX 詳細**: `RoleplayWorkspace.tsx:265` `handleBubbleMouseUp`。character バブルのみに `onMouseUp` を張ったが、複数バブルを跨いだ選択・空白除外・iOS Safari 挙動などのエッジで壊れないか(Reader.tsx と同型のはずだが独立検証が要る)。
