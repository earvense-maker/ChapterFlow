# ロールプレイモード 設計書

> レビュー反映版。現行コードとの照合を行い、セッション単位の排他、SSE のコミット境界、要約カーソル、設定スナップショット、既存 setup 復元との分離を Phase 1 の必須要件として追記した。

## 0. 背景と設計方針

作品のキャラクターと、短い応答・速いテンポで会話できるモードを追加する。参考体験は Zeta.ai / Character.AI 系のキャラチャット。

このモードには二つの価値がある。

1. **遊びとしてのキャラ会話体験**そのもの。
2. **キャラの声の試運転**。speechStyle や want/fear を設定しても、実際に喋らせないと口調の解像度は分からない。会話で確かめた言い回しを設定へ還元する導線になる（Phase 2 以降）。

### 全体構成: 「相談 → コミット → 会話」の同型パイプライン

小説の流れ（設定相談セッション → コミットでプロジェクト生成 → 本編生成）と同じ骨格に乗せる。

```
設定相談セッション（purpose: 'roleplay'）
  → コミットでロールプレイ型プロジェクト生成（world / characters / customSystemPrompt）
  → ロールプレイ会話（ペルソナカード + 短応答 + ストリーミング）
```

コミット済みプロジェクトの `Character[]`（speechStyle / want / fear / secrets / relationshipNotes / currentState）と `worldText` は、会話用システムプロンプトの材料としてほぼ無加工で使える。**設定づくりは既存 setup 基盤の分岐拡張で済ませ、新規実装の本体は会話ランタイムに絞る。**

### 速度の設計原則: 賢さや速さをモデルだけに求めず、構造で確保する

- **出力上限を絞る**: 1ターンの目標は200字程度、指示逸脱時のハード上限は600字。短い応答は生成完了自体が速い。
- **ストリーミング必須**: 既存の `generateTextStream` + SSE を流用し、first token で体感速度を稼ぐ。
- **プロンプトを薄くする**: 本編生成のような全文コンテキストは載せない。ペルソナカード + 世界観ダイジェスト + 直近Nターン + 会話要約で数千トークンに収める。
- **システム部をセッション内で固定する**: 可変部（要約・直近メッセージ）を userPrompt 側に寄せ、プロバイダが対応する場合に prompt caching を利用しやすくする。
- **ホットパスに構造化出力を入れない**: 応答はプレーンテキストのみ。JSON パースも draftPatch もない。

### データ整合性の設計原則

- **保存済みセッションを正とする**: ストリーミング中の表示は暫定表示とし、character メッセージの保存成功をコミット点にする。`done` は保存済み結果の確認応答である。
- **同一セッションの変更を直列化する**: `safeWriteJson` が保証するのは単一ファイル置換の原子性だけであり、read-modify-write の排他ではない。roleplay の全変更操作は sessionId 単位の mutex と revision 検査を通す。
- **会話開始時の設定を固定する**: セッション作成時にペルソナと世界観ダイジェストをスナップショットし、既存会話が後日のキャラ編集・削除で変質しないようにする。
- **正規化を全書き込み境界で共有する**: setup コミット、プロジェクト作成、設定画面、複製など、どの経路から書いても同じ上限・既定値を適用する。

### 対象機能

| 機能 | 概要 | 主な変更箇所 |
|---|---|---|
| A. 相談セッションのロールプレイ用途対応 | `purpose` フラグで相談プロンプトを分岐 | setupPromptBuilder, SetupSession型, SetupWorkspace |
| B. プロジェクト種別とロールプレイ用データ | `projectType`、greeting / dialogueExamples / scenarioSeeds | shared/types, setupCommitService, projectService |
| C. 会話ランタイム | セッション管理 + 短応答プロンプト + SSE API | 新規 roleplaySessionService / roleplayPromptBuilder / routes/roleplay.ts、app.ts、生成共通層 |
| D. チャットUI | ロールプレイ型プロジェクトの主画面 | 新規 RoleplayWorkspace, App.tsx |

### スコープ外（将来検討）

- **小説型プロジェクトのキャラと話す入口**（Phase 2）。データ構造は同じなので追加コストは小さいが、まずロールプレイ型で会話ランタイムを安定させる。
- **会話から種メモ・セリフ例を抽出して設定へ還元**（Phase 2）。ターン完了後の非同期解析として設計する。
- **作者モード**（キャラとメタ会話するトグル）（Phase 2）。
- **ユーザー発言の編集やり直し**（Phase 2）。初期実装は直近キャラ発言の再生成のみ。
- **複数キャラ同席**（Phase 3）。話者制御が必要になり難度が上がる。
- **ユーザーペルソナの詳細設定**（Phase 3）。初期実装ではシナリオ欄に「ユーザーは誰としてそこにいるか」を書けば足りる。
- **時点指定**（「第3話時点のキャラ」と話す）。Phase 1 の currentState はセッション開始時点の最新設定をスナップショットする。
- **プロジェクト非依存のフリーロールプレイ**。アプリの芯（作品づくり）とブレるため採用しない。
- **サンプリングペナルティの注入**。会話の自然さを優先し、本文生成のみの機能のままとする。

**採用に方針変更**（Phase 1 中の実装フィードバックで方針転換した項目）:
- **NG表現の会話への注入**。当初はスコープ外としていたが、ユーザーからのフィードバック
  「会話でもキャラが繰り返す言い回しを NG に入れたい」と、本編と別辞書にする摩擦を避ける
  ために、既存の `expressions.json`（手動登録の active 分のみ、頻出自動検出は skip）を
  ロールプレイ会話のプロンプトにも注入する。設計書 3.2 の userPrompt 末尾（【指示】の直前）
  に `【表現上の注意】` セクションとして挿入し、最大 12 件までを毎ターン最新値で反映する。
  Reader と同じ「本文選択 → NG 登録」導線を RoleplayWorkspace のキャラ発言バブルにも
  用意し、選択→登録の摩擦をさらに減らす。

## 1. 機能A: 相談セッションのロールプレイ用途対応

### 1.1 目的

ロールプレイ用のキャラ・世界観づくりを、フォーム入力ではなく既存の相談体験で行う。ただし掘るべき場所が小説と違うため、相談プロンプトを用途で分岐する。

### 1.2 データ

`SetupSession` に用途フラグを追加する（`src/shared/types.ts`）。

```ts
export type SetupPurpose = 'novel' | 'roleplay';

export interface SetupSession {
  // 既存フィールド…
  purpose?: SetupPurpose; // undefined は 'novel' として扱う（後方互換）
}

export interface CreateSetupSessionBody {
  // 既存フィールド…
  purpose?: SetupPurpose;
}
```

- 既存セッションファイルは `purpose` 無しのまま読めて `'novel'` 動作になる。移行コード不要。
- `SetupDraft` に `scenarioSeeds: string[]` を追加する。novel 用途では常に空配列。既存ファイルの読み込み時は不存在なら `[]` に正規化する（`openingSeeds` などと同じ流儀）。
- `SetupDraftCharacter` に `greeting?: string` と `dialogueExamples?: string[]` を追加する。
- `SetupDraftPatch` に `scenarioSeedsAdd?: string[]` を追加し、`charactersAdd` / `charactersUpdate` で greeting / dialogueExamples を受け付ける。
- `SetupSessionSummary` に正規化済みの `purpose: SetupPurpose` を追加する。保存データでは optional のまま後方互換を保つが、サービス・API 境界では `undefined` を `'novel'` に正規化して返す。
- `CreateSetupSessionBody.purpose` は `'novel' | 'roleplay'` 以外を400にする。`normalizeStoredSession` / `createEmptySetupDraft` / `setupDraftPatchService` も追加フィールドを正規化し、`lockedFields` を含む追加・更新・アーカイブの既存規則を同じように適用する。したがって、パース・検証処理は骨格を流用するが**無変更ではない**。

### 1.3 相談プロンプトの分岐

`buildSetupChatPrompt` に purpose 分岐を入れる。novel 用途の指示は現状維持。roleplay 用途では以下を差し替える。

- 役割: 「あなたはキャラクターチャットの設定づくりの相談相手です。ユーザーはこのキャラと会話して楽しみたい人です。」
- **短期決着を明示する**: 「3〜5往復で会話を始められる状態を目指してください。プロットや章立ての話はしないでください。」Zeta 系の「思い立ってすぐ話し始める」軽さを、相談の目標ターン数で担保する。
- 掘る場所を指定する:
  - **口調の実例**: 「こういう場面でこう言う」というセリフサンプルを2〜3行、必ず具体的な発話として提案する（dialogueExamples の材料）。
  - **初回メッセージ（greeting）**: 会話の開幕をキャラ側から切り出す1〜3文。シナリオと矛盾しない汎用の挨拶にする。
  - **シナリオ候補**: 会話の舞台を2〜3個（「放課後の教室で二人きり」「旅の途中の野営」等）。scenarioSeedsAdd へ。
  - **ユーザーとの関係**: ユーザーが誰としてキャラの前に立つか（幼馴染・部下・初対面の旅人等）。relationshipSeeds を流用する。
- 火種（事件・秘密・誤解）の提案は「会話が転がるきっかけ」として軽く扱い、プロットに発展させない。
- suggestedActions の intent は novel と同じ（`preview` / `commit`）。ラベルは「試しに少し話してみる」「このキャラと話し始める」のようにロールプレイの語彙にする。

出力形式（平文 + `===DRAFT_PATCH===` + JSON）は novel と完全に同一。setupSessionService のパースとロックの骨格は流用し、追加フィールドの許可・正規化・上限検証だけを拡張する。

### 1.4 試し書きプレビューの分岐

`buildSetupPreviewPrompt` も purpose で分岐する。roleplay 用途では小説冒頭サンプルの代わりに「試し会話サンプル」を出す。

- 指示: 「相談中のキャラになりきって、ユーザーとの短い会話例（3往復程度、各発話1〜3文）を書いてください。未確定事項を勝手に確定しないでください。」
- 出力は300字程度。previews への保存・表示は既存の仕組みをそのまま使う。

### 1.5 UI

- `ProjectList` の新規作成導線に3つ目のボタン「キャラと話す作品を作る」を追加し、`purpose: 'roleplay'` の相談セッションを作成する。入口の方式は Phase 1 ではこの形に固定する。
- `SetupWorkspace` は `purpose` を必須 prop として受け取り、右側の種メモ表示に「シナリオ候補」「初回メッセージ」欄を追加表示する。roleplay 用途では「冒頭候補（openingSeeds）」欄と `firstWishSuggestion` の確認UIを隠す。
- 相談セッションの localStorage キーと「最新 active セッション」の探索は purpose 別にする。roleplay 入口から novel セッションを誤復元せず、一覧APIでも同じ purpose のセッションだけを復元候補にする。
- `App.tsx` は `view: 'setup'` とは別に `setupPurpose: SetupPurpose` を状態として保持し、setup 画面への遷移・設定画面往復・再読込で同じ値を渡す。既存セッションを開く場合は `SetupSessionSummary.purpose` を設定してから `SetupWorkspace` を表示する。

## 2. 機能B: プロジェクト種別とロールプレイ用データ

### 2.1 データ

`Project` と `Character` を拡張する（`src/shared/types.ts`）。

```ts
export type ProjectType = 'novel' | 'roleplay';

export interface Project {
  // 既存フィールド…
  projectType?: ProjectType; // undefined は 'novel'（後方互換）
}

export interface Character {
  // 既存フィールド…
  greeting?: string;           // 会話開始時にキャラが最初に発するメッセージ
  dialogueExamples?: string[]; // 口調のfew-shot例。1件=1発話、最大5件
}
```

- `CreateProjectBody` / `ProjectSummary` に `projectType` を追加。`UpdateProjectBody` には含めない（種別の後変更は初期実装では不可。小説⇔ロールプレイの相互昇格は Phase 2 の課題）。保存データでは optional、API の Project / ProjectSummary では `undefined` を `'novel'` に正規化する。
- シナリオ候補はプロジェクト直下に持たせる: `Project.scenarioSeeds?: string[]`（最大10件）。会話開始時の選択肢に使う。
- 共通 `normalizeRoleplayProjectFields` / `normalizeCharacter` を用意し、greeting は trim 後500字以内、dialogueExamples は各200字以内・最大5件、scenarioSeeds は各200字以内・最大10件に丸める。この正規化は `projectService.createProject` だけでなく、設定APIの characters 書き込み、setup コミット、refine 適用、複製にも必ず通す。

### 2.2 コミットの分岐

`buildSetupCommitPrompt` に purpose 分岐を入れる。roleplay 用途の出力形式は novel と同じ骨格のまま、以下を変える。

- `characters[]` に `greeting` / `dialogueExamples` を含めるよう指示する。
- `scenarioSeeds: string[]` を出力に追加する。
- `firstWishSuggestion` は出力しない（本編生成の概念のため）。
- `storyState` は最小構成でよいと指示する: `currentSituation` に「会話開始時点のキャラの状況」を1〜2行、`characterStates` にキャラの初期状態。`importantEvents` / `openThreads` は空でよい。
- `memories` は好み・NG（type: 'preference' / 'negative'）だけに絞る。
- `customSystemPrompt` は「キャラの振る舞いに関するシステム寄りの指示だけ」（例: 「一人称は必ず『あたし』」「絵文字は使わない」）。作品メモを詰め込まない方針は novel と共通。

`SetupCommitPlan` に `scenarioSeeds?: string[]` を追加し、`normalizedToPlan()` とコミットレビューの往復で候補を失わないようにする。`projectType` はモデル出力を信用せず、`session.purpose === 'roleplay'` のとき正規化処理が必ず `'roleplay'` を設定する。roleplay では `firstWishSuggestion` を正規化・表示対象から外し、`memories` はコミット正規化時にも `preference` / `negative` 以外を除外する。

`setupCommitService` の変更:

- roleplay 用途では **第1話・第1シーンを作らない**。現行の `projectService.createProject` は元から episodes を生成しないため、新しい分岐は不要で、空の episodes ディレクトリを維持することを回帰テストで保証する。
- `project.projectType = 'roleplay'` と `scenarioSeeds` を保存する。
- それ以外（world.md / characters.json / memories.json / story-state.json / customSystemPrompt の書き込み）は既存処理をそのまま使う。

### 2.3 保存場所

会話セッションはプロジェクト配下に置く。バックアップ・エクスポート対象に自然に含まれる。

```
projects/{projectId}/roleplay/sessions/{sessionId}.json
```

storageService に `roleplaySessionPath` / `readRoleplaySession` / `writeRoleplaySession` / `listRoleplaySessions` を追加する（`safeWriteJson` と `assertSafePathSegment` を使用、既存の流儀に合わせる）。

### 2.4 プロジェクト一覧・画面遷移

- `ProjectSummary.projectType` を使い、一覧カードに「ロールプレイ」バッジを表示する。
- ロールプレイ型プロジェクトを開くと Reader ではなく RoleplayWorkspace（機能D）に遷移する。`ProjectList` の onOpen から正規化済み projectType を渡し、`App.tsx` の open/create 完了時に `read` / `roleplay` を決める。設定画面を開く前の return view も保持し、ロールプレイから設定を開いて戻ったとき Reader へ遷移しないようにする。
- setup のコミット完了 callback は `{ projectId, projectType }` を返す。後方互換の都合で projectId しか受け取れない箇所では、プロジェクトを再取得して正規化済み projectType を確認してから遷移する。
- モデル・APIキー設定は既存のプロジェクト設定（activeModelProvider / activeModelName）をそのまま使う。**ロールプレイ専用のモデル設定枠は作らない**（設定の「同名別物」問題を増やさない）。代わりに、ロールプレイ型プロジェクトの設定画面のモデル欄に「会話のテンポを重視するなら、現在利用できる低レイテンシー向けモデルを推奨」と説明する。特定モデル名はモデル情報側の更新可能なメタデータから表示し、設計へ固定しない。
- roleplay プロジェクトの複製では projectType / scenarioSeeds / greeting / dialogueExamples をコピーするが、`roleplay/sessions` はコピーしない。既存会話は複製元だけに残す。

## 3. 機能C: 会話ランタイム

### 3.1 データ

`src/shared/types.ts` に追加する。

```ts
export type RoleplaySessionId = string;
export type RoleplayMessageRole = 'user' | 'character';
export type RoleplaySessionStatus = 'active' | 'archived';

export interface RoleplayMessage {
  messageId: string;
  role: RoleplayMessageRole;
  content: string;
  createdAt: string;
}

export interface RoleplayContextSnapshot {
  character: Character;
  otherCharacters: Array<Pick<Character, 'characterId' | 'name' | 'description'>>;
  worldDigest: string;
  customSystemPrompt: string;
  capturedAt: string;
}

export interface RoleplaySession {
  schemaVersion: 1;
  sessionId: RoleplaySessionId;
  projectId: ProjectId;
  characterId: CharacterId;
  scenario?: string;            // 会話の舞台。scenarioSeeds から選ぶか自由入力
  contextSnapshot: RoleplayContextSnapshot;
  status: RoleplaySessionStatus;
  messages: RoleplayMessage[];
  conversationSummary?: string; // 古いターンの圧縮結果
  summaryThroughMessageId?: string; // conversationSummary に含めた最後のメッセージ
  summaryUpdatedAt?: string;     // 派生データ更新時刻。会話のupdatedAtとは分離
  revision: number;             // 楽観ロック（setup と同じ流儀）
  createdAt: string;
  updatedAt: string;
}

export interface RoleplaySessionSummary {
  sessionId: RoleplaySessionId;
  characterId: CharacterId;
  characterName: string;        // contextSnapshot 由来
  scenario?: string;
  status: RoleplaySessionStatus;
  messageCount: number;
  lastExcerpt: string;
  revision: number;
  createdAt: string;
  updatedAt: string;
}

export type RoleplaySessionView = Omit<RoleplaySession, 'contextSnapshot'> & {
  characterName: string;
};
```

`contextSnapshot` はセッション作成時に共通正規化済みデータから作る。既存セッション中にキャラ・世界観・customSystemPrompt を編集またはキャラを削除しても、そのセッションはスナップショットで継続できる。編集結果を試す場合は新規セッションを作る。これによりプロンプトの固定部と会話履歴の再現性を保つ。

`contextSnapshot` はサーバー内部の保存・プロンプト構築専用とし、secrets や長いプロンプト素材を含むためAPIレスポンスへ出さない。GET・SSE・アーカイブ応答は `RoleplaySessionView` を返し、UIに必要な characterName をサーバー側で付与する。

### 3.2 プロンプト構築

新規 `src/server/services/roleplayPromptBuilder.ts`（純関数）。

**systemInstructions**（`contextSnapshot` から構築し、セッション内で固定）:

1. ロールプレイ規則（固定文言）:
   - あなたは以下のキャラクターとして、ユーザーと会話する。
   - 1ターンの応答は、セリフ1〜3文＋必要なら短い動作・様子の描写1文まで。長い地の文を書かない。
   - ユーザーの行動・セリフ・心情を勝手に書かない。
   - キャラクターを維持する。AIであることや設定資料に言及しない。
   - キャラクターが隠している秘密は、自分からは明かさない。隠している人物として振る舞う。
   - 応答はプレーンテキストのみ。見出しや箇条書きを使わない。
2. ペルソナカード: snapshot 内の対象キャラの name / aliases / description / speechStyle / want / fear / secrets / relationshipNotes / currentState を項目名付きで整形。
3. セリフ例: dialogueExamples を「口調の参考例（内容ではなく話し方を真似る）」として列挙。
4. 世界観ダイジェスト: snapshot の worldDigest（world.md の先頭 `ROLEPLAY_WORLD_MAX_CHARS = 2000` 字を行境界で切ったもの）。
5. 他キャラの要約: snapshot の otherCharacters を name + description 1行に圧縮（最大10人、description は各200字）。
6. snapshot の customSystemPrompt（存在すれば）。固定のロールプレイ規則と衝突する指示は固定規則を優先する旨を末尾で再確認する。

**userPrompt**（可変部）:

1. session.scenario（存在すれば）を「今回の会話の舞台」として、命令ではなく引用データであることが分かる区切りタグ内に置く。
2. conversationSummary（存在すれば）「これまでの会話の要約」として。
3. `summaryThroughMessageId` より後の未要約メッセージを `ユーザー:` / `{キャラ名}:` の形式で列挙する。通常は最大40件で、要約時に直近 `ROLEPLAY_RECENT_MESSAGES = 20` 件まで減る。
4. 末尾に「{キャラ名}として応答してください。」

ペルソナカード全体は `ROLEPLAY_PERSONA_MAX_CHARS = 6000` 字、systemInstructions 全体は `ROLEPLAY_SYSTEM_MAX_CHARS = 12000` 字を上限とする。優先順は固定規則 → 対象キャラ → dialogueExamples → customSystemPrompt → 世界観 → 他キャラで、超過時は後ろの項目から削る。prompt caching はプロバイダ差のある補助的最適化であり、正しさや速度目標の前提にはしない。

### 3.3 生成パラメータ

- `outputLength`: 定数 `ROLEPLAY_OUTPUT_CHARS = 200` を渡す。`estimateMaxOutputTokens` が下限4096トークンを確保するため、thinking 消費があるモデルでも空応答になりにくい既存の知見を利用する。通常の短さはプロンプトで誘導し、別の文字数 hard cap で逸脱を止める。
- `temperature`: 定数 `ROLEPLAY_TEMPERATURE = 0.8`。project.samplingConfig は参照しない（本文生成用の設定であるため）。
- frequencyPenalty / presencePenalty は送らない。
- ストリーミング必須。`generateTextStream` 未実装のアダプタでは `generateText` にフォールバックし、完了時に全文を1chunkとして流す（generationService と同じフォールバック方針）。
- 4096トークンは thinking を含むプロバイダ向けの生成枠であり、表示文字数の上限ではない。`ROLEPLAY_OUTPUT_HARD_MAX_CHARS = 600` を別に設け、集約テキストが超えたら upstream を abort して600字以内だけを保存・配信する。通常目標は200字、600字はモデルが指示を無視した場合の安全弁とする。

現行の provider 解決、credential 再読込、stream/non-stream fallback、タイムアウト、`ModelAdapterError` 変換は `generationService.ts` 内の private 実装であり、そのまま呼び出せない。Phase 1 でこれらを副作用の薄い共通 `modelGenerationService`（名称は実装時調整可）へ抽出し、本編生成・roleplay 応答・会話要約のすべてが同じ経路を使う。roleplaySessionService にアダプタ分岐を複製しない。

### 3.4 セッションのライフサイクル

新規 `src/server/services/roleplaySessionService.ts`。

- **作成**: roleplay 型プロジェクトと characterId の存在を検証し、scenario を正規化して `contextSnapshot` とセッションを保存する。対象キャラに `greeting` があれば、それを最初の `character` メッセージとして**LLMを呼ばずに**入れる。greeting が無ければ空のメッセージ列で開始する。
- **送信開始**: sessionId 単位の mutex 内で status / revision / 末尾を検査し、同一sessionIdの生成 in-flight フラグを立てる。末尾が user の場合は `pending_response` として新規送信を拒否し、再生成だけを許可する。user メッセージを保存して revision を1進めてからストリームを開始する。
- **送信コミット**: 生成テキストは未保存の暫定値として配信する。生成完了時に mutex 内でセッションを再読込し、期待 revision と in-flight 所有権を再検査して character メッセージを追記・revision を1進めて保存する。**保存成功がコミット点で、その後に `done` を送る**。保存前のエラー・中断では character メッセージを保存せず、user メッセージは残す。600字の hard cap 到達は正常終了として扱い、配信済み範囲と同じ本文を保存する。
- **再生成**: 末尾 character の直前が user の場合だけ再生成できる。旧 character メッセージは生成中も保存したまま、プロンプト構築時だけ除外し、新しい応答の保存成功時に原子的に置換する。末尾 user は送信失敗からの再試行としてそのまま応答を生成する。失敗・停止時は旧 character を失わない。初回 greeting のように直前 user がない character は再生成不可。
- **アーカイブ**: revision を検査して `status: 'archived'` にするソフト削除。一覧から隠し、archived セッションは閲覧のみ許可して送信・再生成を409で拒否する。

`safeWriteJson` だけでは read-modify-write の競合を防げないため、全変更操作は setupSessionService と同型の sessionId 単位 mutex を通す。生成中は別タブからの send / regenerate / archive を `generation_in_progress` で409にする。プロセス停止で in-memory フラグが消えても、保存済み末尾 user から regenerate できるため復旧可能である。

revision 遷移を固定する。messages-stream の要求 revision を `R` とすると、開始時に `R` を検査し、user 保存後は `R+1`、character コミット時の期待値は `R+1`、成功後は `R+2` とする。regenerate は開始時に保存変更をせず、要求 `R` を検査して旧応答を暫定除外し、コミット時の期待値は `R`、成功後は `R+1` とする。失敗時は messages-stream だけが user 保存分の `R+1` を維持し、regenerate は `R` のままにする。

### 3.5 会話要約による圧縮

- 会話ログはUI・バックアップ用に全件保持し、要約しても削除しない。`summaryThroughMessageId` をカーソルにし、「カーソルより後にある未要約メッセージ」の件数で発火を判定する。
- 発火条件: 未要約メッセージが `ROLEPLAY_SUMMARY_THRESHOLD = 40` 件を超えたとき。直近 `ROLEPLAY_RECENT_MESSAGES = 20` 件を未要約のまま残し、それ以前だけを conversationSummary へ畳む。`TURNS` ではなく実際の単位に合わせて `MESSAGES` と呼ぶ。
- 要約ジョブ開始時に、既存 summary、既存カーソル、畳む対象、対象末尾 messageId をスナップショットする。要約プロンプトには「関係の変化、呼び方の変化、交わした約束、明かされた事実」を残すよう指示する。
- 通常は**応答コミット後にサーバー側で非同期実行**し、sessionId 単位の summary in-flight フラグで二重生成を抑止する。フラグは in-memory でよく、再起動後も未要約件数という永続条件が残るため次回応答後に再試行できる。
- 要約生成中は mutex を保持しない。完了時だけ mutex 内で最新セッションを再読込し、summaryThroughMessageId が開始時カーソルと一致する場合に限り、最新 messages を保ったまま summary / カーソル / summaryUpdatedAt だけをマージ保存する。派生データ更新では会話の updatedAt とユーザー向け revision を進めず、一覧順や次回送信を不意に変えない。
- 次の送信開始時に未要約メッセージが40件を超え、対応する非同期要約がまだ完了していない場合は、履歴の中抜けを避けるため要約完了を待つか同じスナップショットから同期要約してから応答生成する。要約に失敗し、全未要約メッセージがプロンプト予算に収まらない場合は `summary_failed` をretryable errorとして返し、履歴を黙って欠落させない。
- 境界順序は「user 保存 → 未要約件数と文字数を再判定 → 必要なら要約 → 応答生成」とする。保存前が40件ちょうどでも、user 保存後の41件で要約が発火するため、生成プロンプトが上限を越えない。
- 件数だけでなく `ROLEPLAY_VARIABLE_PROMPT_MAX_CHARS = 24000`、`ROLEPLAY_SUMMARY_MAX_CHARS = 6000`、`ROLEPLAY_RECENT_MESSAGES_MAX_CHARS = 16000` を設ける。要約時は新しいものから最大20件かつ16000字以内を残し、それより古い未要約メッセージを畳む。conversationSummary は既存要約を再圧縮して6000字以内にする。scenario・区切り文を含む可変部全体が24000字を超える場合も件数に関係なく要約し、なお収まらなければ `summary_failed` として生成しない。

### 3.6 API

新規 `src/server/routes/roleplay.ts`。

```
POST   /api/projects/:id/roleplay/sessions
body: { characterId: string, scenario?: string }
→ 201 { session: RoleplaySessionView }

GET    /api/projects/:id/roleplay/sessions
→ 200 { sessions: RoleplaySessionSummary[] }   // active のみ、updatedAt 降順

GET    /api/projects/:id/roleplay/sessions/:sessionId
→ 200 { session: RoleplaySessionView }

POST   /api/projects/:id/roleplay/sessions/:sessionId/messages-stream
body: { message: string, revision: number }
→ SSE: event: chunk { text } / event: done { session: RoleplaySessionView } / event: error { error, code, retryable, revision? }

POST   /api/projects/:id/roleplay/sessions/:sessionId/regenerate-stream
body: { revision: number }
→ SSE: 同上

DELETE /api/projects/:id/roleplay/sessions/:sessionId
body: { revision: number }
→ 200 { session: RoleplaySessionView }   // アーカイブ後の最新revisionを返す
```

- route は `app.ts` の `/api` 配下へ登録する。SSE の実装は `generate-stream`（routes/generate.ts）と同じ形で、client close → AbortController → 生成中断とする。ただし切断と保存が競合した場合は保存済みセッションが正であり、再接続後のGETで確定状態を復元する。
- route の開始トランザクションは「入力・project/session・revision検査 → in-flight取得 → user保存（messages-streamのみ）」までとし、すべて成功してからSSEヘッダーを送る。ここまでの失敗は通常の JSON エラー、ヘッダー送信後の失敗は error event とする。user メッセージ保存後のerrorには最新 revision を含め、クライアントは error / stop 後にGETで再同期してから再試行する。
- revision 不一致は409と最新 revision を返す。生成完了時にも期待 revision を再検査し、不一致なら生成結果を保存せず `revision_conflict` とする。
- project が存在し `projectType === 'roleplay'` であること、session がURLの project に属すること、character が作成時に存在することを検証する。project/session不存在またはURLとの不一致は404、novel project は409 `project_type_mismatch`、ID書式不正は400とする。
- message は trim 後1〜2000字、scenario は trim 後0〜1000字。空message、不正ID、型不正は400。GETは archived も取得可能、一覧は active のみとする。

### 3.7 エラー時の体験

- chunk は「生成中」の暫定吹き出しにだけ表示する。`done` 受信時に返却 session の保存済み character メッセージへ置き換える。error / 切断では暫定吹き出しを破棄し、GETで最新 session と revision を再同期してから、エラーバナー＋「もう一度」ボタンを出す。「もう一度」は regenerate-stream を叩く。
- regenerate 中は旧 character 吹き出しを残し、その下または同位置に暫定表示する。失敗時は旧応答へ戻し、成功した `done` のみで置換する。
- APIキー未設定・タイムアウトなどのエラーコードは generationService の GenerateError 流儀に合わせ、retryable を UI の文言に反映する。

### 4.1 画面構成

新規 `src/client/components/RoleplayWorkspace.tsx`。ロールプレイ型プロジェクトの主画面。

```
┌────────────┬──────────────────────────────┐

│                │ └──────────────────────────┘ │
│                │ [入力欄............................] [送信] │
└────────────┴──────────────────────────────┘
```

- **新規会話モーダル**: キャラ選択（characters.json の全キャラ。1人しかいなければ選択済み）＋ シナリオ選択（project.scenarioSeeds をチップで提示 + 自由入力欄。空でも開始可）。

- **再生成**: 最後の character 発言の直前が user 発言の場合だけ「↻」ボタンを出す。初回 greeting には表示しない。
- **ヘッダー**: キャラ名、シナリオ、設定画面への導線（既存の WorkSettingsTab / TechSettingsTab へ）。WorkSettingsTab のキャラ編集に greeting / dialogueExamples を追加し、共通正規化と同じ上限をUIにも表示する。TechSettingsTab は roleplay で使わない本文字数・streamingEnabled・samplingConfig を非表示または無効化し、モデル選択だけを有効にする。
- **競合復旧**: 409では自動再送せず、最新sessionをGETして表示を更新する。入力中テキストは保持し、「別画面で会話が更新されました」と案内する。

## 7. 前提と割り切り

- **応答の短さ・速さはプロンプト規則と出力上限とモデル選択の合わせ技であり、保証ではない。** 遅いモデルを選べば遅い。推奨モデルの案内で誘導する。
- **secrets の秘匿はプロンプト指示ベースであり、モデルが漏らす可能性はゼロにできない。** 厳密なネタバレ制御（storyState の knownBy 参照など）は行わない。絶対に会話に出したくない情報は、そもそも設定に書かないのがユーザー側の運用となる。
- **キャラの知識はセッション開始時点の最新設定で固定。** contextSnapshot 作成後の設定変更は既存セッションへ反映しない。「第n話時点のキャラ」との会話や、既存セッションへの明示的な設定再読込は将来課題。
- **会話ログは本編・storyState に一切影響しない。** ロールプレイはサンドボックスであり、正史への反映は Phase 2 の抽出機能を通じてユーザーが明示的に行う。
- **要約圧縮で細部は失われる。** 要約プロンプトで関係変化・約束・呼び方を優先して残すが、完全ではない。長期の一貫性より会話のテンポを優先する設計判断。
- **モデル設定はプロジェクト単位の既存設定を使う。** ロールプレイ専用のモデル設定は増やさない（設定画面の複雑化と「同名別物」問題を避ける）。
- **Phase 1 は現在と同じ単一Nodeプロセス運用を前提とする。** sessionId 単位の in-memory mutex / in-flight フラグは Electron・ローカルサーバー・LAN配信の単一プロセス内で有効である。将来サーバーを複数プロセス化する場合は、ファイルロックまたは共有ストレージ上のCASへ置き換える。
- `RoleplaySession` は新規機能のため `schemaVersion: 1` から開始する。読み込み時には optional な summary 派生フィールドを正規化し、未知の schemaVersion は書き換えず明示エラーにする。将来の破壊的変更時に移行関数を追加する。
