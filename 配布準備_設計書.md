# 配布準備 設計書

## 0. 背景と目的

Yumeweaving を作者以外の環境に配布できる状態にするための設計書。これ単体で実装作業へ渡せることを目的とする。

配布形態のロードマップ:

- **第1段階(本書の範囲)**: zip + Node.js 20+ 前提の配布。技術に明るい相手向け。
- **第2段階**: Electron によるデスクトップアプリ化(別紙『Electron化_設計書.md』)。
- **最終目標**: スマホ単体で完結するアプリ(データ同期なし)。本書の範囲外だが、判断に影響する場合は各項目に注記する。

Electron化との関係: **D1(パス除去)と D7 の bat 改修は Electron 配布では不要になる**が、第1段階の配布と開発環境の健全化のため実施する。**D2(データ保存先)・D3(ライセンス)・D4(利用者ガイド)・D5(LAN認証)は Electron 化後もそのまま生きる**ため、先に実施する価値が高い。

対象コード:

- `start-yumeweaving.bat` / `start-yumeweaving-lan.bat`
- `create-shortcut.js` / `create-shortcut.vbs`
- `src/server/config.ts`(DATA_DIR 解決)
- `src/server/index.ts`(listen 失敗処理、LAN認証ミドルウェア)
- `package.json`(scripts / license)
- `scripts/`(パッケージングスクリプト新設)
- `README.md` / `docs/`(利用者ガイド新設)

前提となる現状仕様(変更しないもの):

- サーバーは Express、既定ポート 3001、既定バインド `127.0.0.1`。LAN モードは `YUMEWEAVING_HOST=0.0.0.0`。
- `YUMEWEAVING_DATA_DIR` は `path.resolve(PROJECT_ROOT, value)` で解決される(絶対パスはそのまま、相対パスはリポジトリ基準)。
- APIキーは `{DATA_DIR}/config/credentials.json` に平文保存。UI(技術設定タブ)からも `POST /api/models/credentials` 経由で保存できる。
- vitest はテスト用一時ディレクトリを自前で設定済み(c9edb92)。

## D1. 個人パスのハードコード除去

**問題**: 以下のファイルに `C:\Users\Yuhei\...` が直書きされており、他環境ではそのまま動かない。

- `start-yumeweaving.bat` — `cd /d` 先と `YUMEWEAVING_DATA_DIR`
- `start-yumeweaving-lan.bat` — 同上
- `create-shortcut.js` / `create-shortcut.vbs` — ショートカットの作成先・リンク先・作業ディレクトリ

**方針**: bat は自分自身の場所(`%~dp0`)基準に書き換える。`YUMEWEAVING_DATA_DIR` の明示設定は**削除せず、`%USERPROFILE%\Documents\Yumeweaving` へ書き換えて恒久的に維持する**。

- NOTE(削除しない理由): bat は `npm run dev` を呼ぶため、env 行を消すと D2 の開発既定(`data`)が効いて実データを見なくなる。「開発ツールチェーン経由でも実データで起動する」のが bat の役割であり、env 明示はその意図の表現として正しい。

**実装**:

1. 両 bat の `cd /d "C:\Users\Yuhei\Desktop\Yumeweaving"` を `cd /d "%~dp0"` に変更。
   - NOTE: `%~dp0` は UNC パス(共有フォルダから直接実行)では `cd /d` が失敗する。ガイド(D4)に「zip はローカルに展開してから実行」と明記して対処とする。
2. 両 bat の `set "YUMEWEAVING_DATA_DIR=..."` を `set "YUMEWEAVING_DATA_DIR=%USERPROFILE%\Documents\Yumeweaving"` に変更(D2 の既定値と同値。二重指定に見えるが上記 NOTE のとおり意図的)。
3. `create-shortcut.js` / `.vbs`: デスクトップパスは `WshShell.SpecialFolders("Desktop")`(vbs)/ PowerShell 側は `[Environment]::GetFolderPath('Desktop')` で取得し、リンク先はスクリプト自身の場所から解決する。
4. 修正後、`git grep -i yuhei -- ':(exclude)*.md'` がヒット0件であることを確認する(設計書類にはパス例として残るため md は除外)。

**受け入れ条件**:

- リポジトリを任意のフォルダ(例: `D:\apps\Yumeweaving`)に置いても bat から起動できる。
- `git grep -i yuhei -- ':(exclude)*.md'` が何も返さない。
- bat 経由の起動が実データ(`Documents\Yumeweaving`)を見る(D2 と両立)。

## D2. データ保存先の既定値変更

**問題**: `YUMEWEAVING_DATA_DIR` 未設定時の既定がリポジトリ内 `data\` のため、配布先では (a) 開発用フィクスチャと実データが混在し、(b) アプリ更新(フォルダ差し替え)で作品データが消えるリスクがある。作者環境では bat の環境変数で回避しているが、配布先ユーザーは環境変数を設定しない。

**方針**: 「未設定なら本番想定」に反転させる。既定を `%USERPROFILE%\Documents\Yumeweaving` とし、開発時の `data` 指定は**「未設定時のみ補完する」ラッパースクリプト**で行う(`scripts/start-lan.mjs:16` の `process.env.X = process.env.X ?? default` と同じ、リポジトリ既存のパターン)。

- Documents 直下を選ぶ理由: 小説という**ユーザー自身の創作物**であり、隠しフォルダ(`%APPDATA%`)より可視・バックアップ容易な場所が適切。作者の既存データ場所とも一致する。
- cross-env(無条件上書き)を使わない理由: 上書き方式は (a) `playwright.config.ts:21` が `npm run dev` 経由で渡す `.tmp/e2e-data` を潰して e2e の隔離を壊し、(b) bat が設定する `Documents\Yumeweaving` も潰して作者の日常起動が実データを見なくなる。**外から渡された env は常に尊重し、未設定時だけ開発既定を補う。**

**実装**:

1. `src/server/config.ts`(実施済み・未コミット):
   ```ts
   import { homedir } from 'node:os';
   export const DATA_DIR = process.env.YUMEWEAVING_DATA_DIR
     ? path.resolve(PROJECT_ROOT, process.env.YUMEWEAVING_DATA_DIR)
     : path.resolve(homedir(), 'Documents', 'Yumeweaving');
   ```
   NOTE: OneDrive のフォルダリダイレクト環境では `homedir()\Documents` が実体と異なる場合がある。まずこの素朴な解決で出し、問題報告があれば `Documents` 取得を精緻化する(過剰実装を避ける)。
2. `scripts/dev-server.mjs` 新設:
   ```js
   // 開発サーバー用エントリ。未設定時のみ開発フィクスチャ(リポジトリ内 data/)を指す。
   // bat(実データ)や playwright(.tmp/e2e-data)から渡される env は尊重する。
   process.env.YUMEWEAVING_DATA_DIR = process.env.YUMEWEAVING_DATA_DIR ?? 'data';
   ```
   その後 `tsx watch src/server/index.ts` を `spawn`(`stdio: 'inherit'`、終了コード伝搬)で起動する。
3. `package.json`: `"dev:server": "node scripts/dev-server.mjs"` に変更。cross-env は導入しない。
4. README の「データ保存場所」を更新。
5. 起動ログに解決後の DATA_DIR を1行出力する(実施済み・未コミット)。
6. **コミットの原子性**: 手順1(既定値反転)と手順2〜3(dev の data 補完)は**必ず同一コミット**で入れる。片方だけの状態では素の `npm run dev` が実データ(`Documents\Yumeweaving`)へ書き込む。本書執筆時点の作業ツリーはまさにこの片肺状態(config.ts のみ変更済み)なので、ラッパーが揃うまでコミットしないこと。

**受け入れ条件**:

- env 未設定で起動すると `Documents\Yumeweaving` にデータが作成される。
- 素の `npm run dev`(env 未設定)は従来どおりリポジトリ内 `data\` を使う。
- playwright の e2e が `.tmp/e2e-data` を使い続ける(env 継承が生きている)。vitest も実データディレクトリに触れない。
- bat 経由の起動(env = `%USERPROFILE%\Documents\Yumeweaving`)が実データを見る。

## D4. 利用者向けガイドの新設

**問題**: 現 README は開発者向け(vite / playwright 等)。配布先ユーザーが必要とする情報(APIキー取得、課金注意、バックアップ)がない。

**方針**: README は開発者向けのまま維持し、`docs/利用ガイド.md` を新設して zip に同梱する。以下の構成とする。

**実装** — `docs/利用ガイド.md` の章立て:

1. **Yumeweaving とは** — 3行程度。
2. **必要なもの** — Windows 10/11、Node.js 20+(インストーラURLと手順)、いずれかの LLM APIキー。
3. **インストールと起動** — zip を**ローカルフォルダに展開してから**(共有フォルダ直上では動かない: D1 NOTE)配布用 bat(`Yumeweaving起動.bat`)をダブルクリック(初回は依存取得で数分待つ旨。D7 の bat が対応)。
4. **APIキーの取得と登録** — Gemini(Google AI Studio)/ DeepSeek / OpenAI それぞれの取得ページURLと手順。アプリ内「技術設定」タブから登録すること。**太字で課金注意**: 「生成はあなた自身のAPIキーで行われ、利用量に応じて各社から課金されます。各社の料金ページを確認してください」。
5. **データの保存場所とバックアップ** — `Documents\Yumeweaving` フォルダをコピーすればバックアップ完了であること。APIキーが `config\credentials.json` に**平文で**保存されるため、フォルダを人と共有する際は `config` を除くこと。
6. **スマホから使う(任意)** — LAN モードの起動、ファイアウォール許可(プライベートのみ)、**接続用URL(トークン付き)の扱い**(D5)。「自宅など信頼できる Wi-Fi でのみ使うこと」。
7. **トラブルシューティング** — ポート3001衝突(D7 の改善後メッセージ)、ファイアウォールで拒否してしまった場合の再設定、起動しない場合のログの場所。

**受け入れ条件**: ガイドだけを読んだ第三者が、口頭サポートなしで「インストール → APIキー登録 → 1話生成」まで到達できる(D6 の受け入れテストで実証)。

## D5. LAN モードの認証

**問題**: LAN モード(`0.0.0.0` バインド)は無認証のため、同一ネットワークの任意の端末が作品の閲覧・改変に加え、**保存済みAPIキーでの生成実行(=作者への課金)** が可能。自宅専用の現状は許容範囲だが、配布物の機能としては危険。

**方針**: 事前共有トークン + Cookie 方式。実装が小さく、スマホ側はURLを1回開くだけで済む。

- トークンは LAN モード初回起動時に生成し `{DATA_DIR}/config/lan-token.json` に保存(以降再利用)。
- 起動ログに `http://<LAN_IP>:3001/?token=<token>` を表示。スマホはこれを開く。
- サーバーはクエリの token を検証し、合致すれば HttpOnly Cookie を発行してクエリなしURLへリダイレクト。
- 以降の全リクエスト(静的配信・API とも)は Cookie を検証。
- **ループバック(127.0.0.1 / ::1)からのアクセスは無条件で許可**(PC 本体の利用と localhost バインド時の挙動を変えないため)。

**実装**:

1. `src/server/services/lanAuthService.ts` 新設:
   - `ensureLanToken(): Promise<string>` — `crypto.randomBytes(24)` を base64url 化して保存/読込。
   - `verifyToken(value: string): boolean` — **両辺を sha256 ダイジェストにしてから** `crypto.timingSafeEqual` で比較する(`timingSafeEqual` は長さ違いの入力で throw するため、直接比較しない)。
2. `src/server/index.ts` に認証ミドルウェアを追加。有効化条件は「**バインドアドレスがループバックでない場合**」とする(`HOST !== '127.0.0.1'` という文字列比較では `::1` 等を取りこぼす):
   - `req.socket.remoteAddress` がループバック → 通す。判定は `127.0.0.1` / `::1` / `::ffff:127.0.0.1`(IPv4射影)をすべてカバーするヘルパー `isLoopbackAddress()` で行う。
   - Cookie `yw_lan_auth` が有効 → 通す。
   - `req.query.token` が有効 → Cookie 発行(`HttpOnly; SameSite=Lax; Max-Age=90日`)し、クエリを除いたURLへ302。
   - いずれも不成立 → 401。**`/api/*` へは JSON(`{ error: ... }`)**を返し(クライアントのエラー表示経路を壊さない)、それ以外(静的ページ)へは簡素な日本語HTML(「起動ログに表示されたURLから開いてください」)。
   - NOTE: ルーター登録より**前**、`express.json` より前に配置する。
3. Cookie パースは依存追加を避け、`req.headers.cookie` の素朴な分解で行う(値は base64url 英数のみ)。
4. 起動ログ(`listLanUrls` 周辺)をトークン付きURLの表示に変更。
5. `YUMEWEAVING_LAN_TOKEN` env で固定値を上書き可能にする(テスト・詰まった時の逃げ道)。
6. **Service Worker の追随修正**: `sw.js` の `networkFirst` は `res.ok` を確認せず `cache.put` するため、401 レスポンスがキャッシュされうる(`cacheFirst` 側にしか ok チェックがない)。`networkFirst` にも `res.ok` チェックを追加する。実害は限定的(LAN の平文 http は secure context 外で SW 自体が登録されない)だが、認証導入と同時に直す。

**受け入れ条件**:

- LAN モードで、トークンなしアクセスは静的ページ・`/api/*` とも 401。
- トークン付きURLを一度開けば、以降はクエリなしで利用できる(Cookie)。
- localhost バインド(通常モード)の挙動が一切変わらない。PC 本体から `127.0.0.1:3001` を開く動線も無認証のまま。
- ユニットテスト: token 検証、ループバック素通り、Cookie 発行。

## D6. クリーン環境での受け入れテスト

**問題**: 作者環境で暗黙に満たされている前提(Chrome/Edge の存在、Node の PATH、ファイアウォール設定済み等)が、配布先で最初に壊れる。コード変更ではなく検証項目として定義する。

**方針**: 開発ツールの入っていない Windows(別PC、または新規 Windows ユーザーアカウントでも近似可)で、**配布 zip と利用ガイドだけ**を使って以下を通す。

**チェックリスト**:

1. Node.js 20 LTS をガイドの手順でインストールできる。
2. zip 展開 → bat ダブルクリックで起動し、ブラウザ/アプリウィンドウが開く(初回の依存取得込み)。
3. 技術設定タブから APIキーを登録し、検証(validate)が通る。
4. 新規作品を作成し、1話生成できる。
5. アプリ終了 → 再起動でデータが残っている。`Documents\Yumeweaving` に生成物がある。
6. LAN モード: ファイアウォールダイアログ → スマホからトークンURLで接続 → 閲覧・生成できる。トークンなしでは 401。
7. ポート3001を故意に塞いだ状態で起動し、D7 のエラーメッセージが表示される。
8. アンインストール相当(フォルダ削除)をしてもユーザーデータが消えないことを確認。
9. 配布 zip を **Explorer 以外の解凍ツール(7-Zip 等)でも展開**し、フォルダ階層と日本語ファイル名が壊れないことを確認(D7 のアーカイブ形式の検証)。

**受け入れ条件**: 全項目をガイド記載の手順のみで通過。詰まった箇所はガイド(D4)へフィードバックして再試行。

## D7. 配布物の整理とパッケージング

**問題**: 現状は「リポジトリごと渡す」以外の手段がなく、ログ・テスト成果物・設計書群が混ざる。またポート衝突時の案内が bat 内の PowerShell コマンド頼みで不親切。

**方針**: `npm run package` 一発で配布 zip を生成する。zip には `node_modules` を含めず、初回起動時に bat が `npm ci --omit=dev` を実行する(zip サイズと Windows のパス長問題を回避)。**開発用 bat と配布用 bat は分離する** — 「dist があれば本番モード」のような兼用判定は、LAN 用にビルド済みの作者環境で日常起動が古い dist 配信に化ける(HMR なし・src 変更が反映されない)一方、配布環境では dev 側に到達しないため、統一の利益がない。

**実装**:

1. `scripts/package.mjs` 新設:
   - `npm run build` 実行(または実行済みを前提に検証)。
   - ステージングフォルダへ以下をコピー: `dist/`、`presets/`、利用ガイド、`LICENSE`、`package.json`、`package-lock.json`、`open-app-window.js`、`scripts/start-lan.mjs`。
   - **配布用 bat(`Yumeweaving起動.bat` / `Yumeweavingスマホ共有.bat`)はステージング時に package.mjs が生成する**(リポジトリ内の開発用 bat はコピーしない)。内容は手順2。
   - **含めないもの**: `data/`(フィクスチャ)、`src/`、`tests/`、`*.log`、`playwright-report/`、`test-results/`、設計書群 `*.md`(リポジトリ直下)。
   - アーカイブは Windows 10+ 標準の **`tar.exe -a -cf`** で生成する(`yumeweaving-v{version}.zip`)。PowerShell 5.1 の `Compress-Archive` は区切り文字がバックスラッシュになり Explorer 以外の解凍ツールで階層が壊れる既知問題+日本語ファイル名の文字化けリスクがあるため使わない。日本語名の破損が D6-9 の検証で観測された場合は、zip 内ファイル名を ASCII(`UserGuide.md` 等)へ寄せるフォールバックを取る。
2. 配布用 bat の内容(開発用 `start-yumeweaving.bat` は現状の役割のまま維持):
   - `node -v` が失敗したら「Node.js 20+ をインストールしてください」+ ガイド参照を表示して終了。
   - `node_modules` がなければ `npm ci --omit=dev` を実行(初回メッセージ付き)。
   - 起動順(**zip には devDeps がなく `concurrently` が使えない**ため npm scripts の dev 系は流用不可):
     ```bat
     start "" /b node open-app-window.js http://localhost:3001
     node dist\server\index.js
     ```
     `open-app-window.js` は `waitForServer` を持つ(`open-app-window.js:29`)ため、先にバックグラウンドで起動しておけばサーバーの listen を待ってからウィンドウを開く。サーバーはフォアグラウンドで走らせ、bat の窓がサーバーの生存と一致するようにする。
3. ポート衝突の一次対応をサーバー側へ移す。`src/server/index.ts` の `app.listen` に `error` ハンドラを付け、`EADDRINUSE` のとき日本語で「ポート3001が使用中です。既に Yumeweaving が起動していないか確認してください」+ 対処コマンドを表示して `process.exit(1)`。bat 側の案内文は残す。
4. `package.json` の `version` を配布のたびに上げる運用を README に明記(配布先の不具合報告と突き合わせるため)。

**受け入れ条件**:

- `npm run package` で zip が生成され、中に `src/`・`data/`・ログ・開発用 bat が含まれない。
- zip を展開した素の状態から配布用 bat だけで起動できる(D6 で実証)。
- 作者環境の `start-yumeweaving.bat`(開発用)の挙動が変わらない(HMR が効き続ける)。
- ポート衝突時に日本語メッセージが出て即終了する(ハングしない)。
