# 運用機能拡張 G1 レビュー修正指示

コードレビュー(8観点並列 + 検証)の確定所見に基づく修正指示。本書単体で実装に渡せることを目的とする。§1 は必須(重大度順)、§2 は任意。各項目に場所・問題・修正方法・受け入れ条件を記載する。

前提: 対象は G1 データディレクトリ変更の実装(コミット未取り込み・作業ツリー状態)。修正はそのまま同じ作業ツリーに重ねてよい。

## §1. 必須修正(重大度順)

### F1. シーン採用後の背景タスクが writeScope を継承してロックをすり抜ける【サイレントなデータ損失】

**場所**: [src/server/services/generationService.ts:674](src/server/services/generationService.ts:674) の `void refreshStoryStateAfterAcceptance(projectId).catch(...)`。土台は [src/server/services/dataDirLock.ts:20](src/server/services/dataDirLock.ts:20) の再入エスケープ `if (writeScope.getStore()) return fn();`。

**問題**: `acceptGenerationUnlocked` は `withProjectWriteLock` → `withDataDirWrite` の中で実行されるため AsyncLocalStorage の `writeScope=true` が有効。その中で fire-and-forget される `refreshStoryStateAfterAcceptance` は Node の継続チェーンを通じて `writeScope=true` を継承したままLLM を待つ。外側 `accept` は即 return して `activeWrites` を0に戻すが、背景タスクの後続 storage 書き込みは「ネスト書き込み」判定で `activeWrites` を触らず・`locked` チェックもすり抜ける。

**帰結**: ユーザーがシーン採用直後にデータディレクトリ移動を実行 → `waitForActiveWrites()` は 0 と見て snapshot コピー → `pendingCleanup=旧DATA_DIR` を書いて再起動 → 次回起動で `cleanupPendingDataDir` が旧側を `fs.rm(recursive:true)`。**背景タスクが旧側に書いた storyState 更新はコピーされず、再帰削除で永久に消える**。

既存テスト `tests/unit/generationServiceLock.test.ts:86` の「does not hold the project write lock while background story state refresh waits on the model」はこの経路の存在を裏付けている。

**修正**(以下2案のどちらでもよいが、案Aを推奨):

**案A(推奨・確実)**: 背景タスクを writeScope の外で起動する。`acceptGenerationUnlocked` の内部で fire-and-forget するのではなく、外側 `acceptGeneration` の `withProjectWriteLock` 完了後にキックする。または `writeScope.run(false, () => refreshStoryStateAfterAcceptance(...))` で明示的にスコープから抜ける(AsyncLocalStorage の run は override 可能)。

```ts
// generationService.ts の該当箇所
// 現在: writeScope 内で fire-and-forget
void refreshStoryStateAfterAcceptance(projectId).catch(...);
// 修正: writeScope から抜けてから起動
writeScope.run(false, () => {
  void refreshStoryStateAfterAcceptance(projectId).catch(...);
});
```
これで背景タスクの後続書き込みは通常の `withDataDirWrite` 経由となり、`activeWrites` を正しく増減する。

**案B(補強のみ)**: 案A に加え、`refreshStoryStateAfterAcceptance` 自身の入口で `withDataDirWrite` を明示的に呼ぶ(現状は暗黙にラップされる前提で書かれている)。案Aで十分だが、防御深化として入れてよい。

**受け入れ条件**:

- ユニットテスト追加: 「acceptGeneration 直後に背景タスクが未完了の状態で `withDataDirLock` を要求すると、`waitForActiveWrites` が背景タスクの完了を待ってから進む」。既存 `generationServiceLock.test.ts:86` のテストは維持したうえで、この新規テストが通ること。
- 手順: (1) 背景タスクを人工的に遅延させるモデルスタブを差し込む → (2) accept を実行 → (3) 即座に `withDataDirLock` を呼ぶ → (4) ロック取得が背景タスクの完了後になることを assert。

### F2. cleanup 前の差分検証がなく F1 と組み合わさるとデータ損失を助長

**場所**: [src/electron/main.ts:112](src/electron/main.ts:112) の `cleanupPendingDataDir`。

**問題**: 「新側 `projects/` が isDirectory なら OK」だけで旧側を `fs.rm(recursive:true, force:true)`。verifyCopiedData 以降に旧側へ書かれた差分(F1 の背景書き込みや、他の post-copy race)を検出せず消す。

**修正**: cleanup の直前に manifest 再検証を走らせる。`dataDirMoveService` の `verifyCopiedData` が既に持っている「旧側の全ファイル relative path + size を新側に照合」を、cleanup 経路からも呼び出す。差分があれば、①差分ファイルを新側へコピーで追いつく(推奨)か、②pendingCleanup をクリアせず warn を出してユーザーに手動対応を促す(保守的)。F1 と組み合わせて F2 も実装する場合、案①が「保険」として最も強い。

```ts
// main.ts cleanupPendingDataDir 内
const diff = await verifyManifestForCleanup(pendingCleanup, dataDir);
if (diff.missingInNew.length > 0) {
  await copyMissingFiles(pendingCleanup, dataDir, diff.missingInNew);
}
await fs.rm(pendingCleanup, { recursive: true, force: true });
```

`verifyManifestForCleanup` は `dataDirMoveService` から export し、`main.ts` と共有する。

**受け入れ条件**:

- ユニットテスト追加: 旧側に「新側に無いファイル」を意図的に置いた状態で `cleanupPendingDataDir` を呼ぶと、新側に補填コピーされてから削除される。
- F1 と組み合わせた結合テスト(可能なら): 背景タスクを遅延させて accept → 移動 → 再起動、で storyState の書き込みが失われないことを確認。

### F3. 非 Electron 起動経路(LAN モード)で移動 UI がサーバーを殺し新側を孤児化させる

**場所**: [src/server/index.ts](src/server/index.ts)(全体)、[src/client/App.tsx](src/client/App.tsx) の AppSettingsPanel マウント条件。

**問題**: `src/server/index.ts` は `startServer` に `onRestartRequest` を渡さず、`app-settings.json` も読まない(Electron の bootstrap を通らないため)。しかし `App.tsx` は Electron 判定なしで AppSettingsPanel を表示するので、ブラウザ経由の LAN モードユーザーが移動を実行できてしまう。実行すると:

1. `applyDataDirMove` は成功(コピー+検証+writeAppSettings)
2. `requestRestart` は `onRestartRequest` 未設定なので system.ts の fallback `process.exit(0)` → サーバー死
3. `scripts/start-lan.mjs` を手動で再実行しても、`index.ts` は app-settings を読まないので `YUMEWEAVING_DATA_DIR` は旧既定値のまま → 新側のコピー済みデータは孤児化

**修正**(以下2案のどちらでもよい。**LAN モード自体の位置付け次第**):

- **案A(推奨・単純)**: LAN 起動時は移動 UI を無効化する。クライアントが Electron 環境かどうかを判定して AppSettingsPanel を分岐。判定方法: 起動時に `GET /api/system/version` などのレスポンスに `runtime: 'electron' | 'server'` を含める(サーバー側で `process.versions.electron` を確認して返す)。LAN モードでは AppSettingsPanel を開くとバナー「LAN モードではデータ保存先を変更できません。Electron 版から実行してください」を表示。
- **案B**: `src/server/index.ts` に `main.ts` の `prepareAppSettingsBeforeServer` 相当を移植し、`onRestartRequest` は `process.on('SIGUSR2', ...)` などで supervisor(scripts/start-lan.mjs)に伝達 → スクリプト側で再 exec。ただし LAN モードはユーザーが手動で再起動する運用なので、この作り込みは過剰と判断する。

**推奨は案A**。

**受け入れ条件**:

- `npm run start:lan` で起動したブラウザから AppSettingsPanel を開くと、移動 UI が無効化され案内メッセージが出る。
- Electron 版では従来どおり動作する。

### F4. 生成中の移動要求で数分無応答になる

**場所**: [src/server/services/generationService.ts:232](src/server/services/generationService.ts:232) の `withProjectWriteLock`、[src/server/services/dataDirLock.ts:44](src/server/services/dataDirLock.ts:44) の `waitForActiveWrites`、[src/client/components/AppSettingsPanel.tsx:74](src/client/components/AppSettingsPanel.tsx:74) の handleApply 表示。

**問題**: withProjectWriteLock がストリーミング LLM 呼び出し全体(最大 idle 120秒 × 反復)を withDataDirWrite で包む → `waitForActiveWrites` に timeout がなく、UI 側にも進捗表示なし。生成中に移動を要求すると数分無応答。

**修正**: 3つの層で対応する。

1. **サーバー側 timeout**: `waitForActiveWrites` に timeout(推奨 30秒)を導入。timeout で `DataDirBusyError`(F7 の 503 経路に相乗り)を投げる。
2. **/apply の事前チェック**: `applyDataDirMove` の入口で「進行中の write があれば 409 で早期拒否」する軽量チェックを入れ、ユーザーに「生成の完了を待ってから再度お試しください」と返す(いきなりロック待ちに入らない)。
3. **クライアント表示**: 上記 409/503 を retryable として扱い、AppSettingsPanel に「生成中のため移動を開始できません。生成完了後に再度お試しください」を表示。

**受け入れ条件**:

- ユニットテスト: `waitForActiveWrites` が timeout で reject する。
- 手動確認: 生成を開始 → apply を実行 → 「生成中のため…」の案内が即座に出る(ハングしない)。

### F5. 作品ゼロの環境で pendingCleanup が永久に処理されない

**場所**: [src/electron/main.ts:112](src/electron/main.ts:112) の `fs.stat(<newDataDir>/projects)`。

**問題**: `PROJECTS_DIR` は起動時に ensureDir されず、`createProject` で遅延生成される([src/server/services/storageService.ts:141](src/server/services/storageService.ts:141))。作品ゼロで移動を行うと、新側 `projects/` が存在しないため cleanup 判定が ENOENT で捕捉されて no-op → `pendingCleanup` が永久残り、AppSettingsPanel に「旧データが残っています」バナーが出続ける。

**修正**: **F2 で verify-manifest 方式に切り替える場合、この項目は自動的に解消される**(空環境でも manifest が empty → 差分ゼロで削除に進む)。F2 を採用しない場合は、以下のいずれかで対処:

- **代替A**: cleanup 判定を `fs.stat(newDataDir)` の isDirectory(projects/ ではなく親フォルダ)+ `dataDir` の値が現在の env と一致することの2条件に変える。
- **代替B**: bootstrap または server.ts の起動処理で `ensureDir(PROJECTS_DIR)` を呼ぶ(config.ts から export 済み)。

**推奨は F2 との統合**(F2 の manifest 検証をそのまま流用)。

**受け入れ条件**:

- 作品ゼロで移動 → 再起動 → cleanup が正常完了し、バナーが消える。
- 既存テスト全通過。

### F6. rollback が verify 済みのコピーも消してしまう

**場所**: [src/server/services/dataDirMoveService.ts:80](src/server/services/dataDirMoveService.ts:80) の `applyDataDirMove` catch 節。

**問題**: `fs.cp` と `verifyCopiedData` が成功した後に `writeAppSettings` が失敗すると(AV ロック・AppData 一時アクセス拒否・ディスクフル)、catch が rollbackNewDataDir を呼び、**数分かけてコピー成功済みの新側を再帰削除**する。ユーザーは全再送を強いられる。

**修正**: rollback をステージ分けする。

```ts
try {
  await fs.cp(...);
  await verifyCopiedData(...);
} catch (err) {
  await rollbackNewDataDir(newDataDir, oldDataDir);  // ここまではロールバック
  throw ...;
}
try {
  await writeAppSettings({ dataDir: newDataDir, pendingCleanup: oldDataDir });
} catch (err) {
  // 新側は残す。ユーザーに「保存に失敗しました。もう一度お試しください」を返す
  throw new DataDirMoveError(
    'settings-write-failed',
    '新しい場所へのコピーは成功しましたが、設定の保存に失敗しました。もう一度お試しください。'
  );
}
```

再試行時は `applyDataDirMove` の入口で「新側が既に存在し verify を通る」場合は writeAppSettings だけを再実行する経路を追加(コピー再実行を避ける)。

**受け入れ条件**:

- ユニットテスト追加: writeAppSettings のみをスタブで失敗させると、新側フォルダは残り、リトライで writeAppSettings だけが再走される。
- コピー段階の失敗では従来どおり新側が rollback される。

### F7. write 系ルートが DataDirLockedError を素通しして 500 に化ける

**場所**: [src/server/routes/generate.ts:140](src/server/routes/generate.ts:140) 以降の accept/reject/revert/unaccept/navigate、および `state.ts` / `memories.ts` / `expressions.ts` / `refine.ts` / `projects.ts` / `setupSessions.ts` のライト系ルート。app.ts([src/server/app.ts](src/server/app.ts):81 付近)には custom middleware なし。

**問題**: 移動中に上記ルートが叩かれると `DataDirLockedError` が投げられ、各ルートは `catch(err){next(err)}` するだけ。app.ts の generic middleware が 500 で返す(err.message 自体は日本語で保存されるが、`retryable` フラグと 503 ステータスが失われる)。generationService だけが個別に `GenerateError('data_dir_moving', retryable: true)` に remap している。

**修正**: app.ts に一括 middleware を追加する(**最深の解**)。ルート個別対応より、失敗箇所が構造的にゼロになる。

```ts
// src/server/app.ts、ルーター登録より後、既存の generic middleware より前
app.use((err, _req, res, next) => {
  if (err instanceof DataDirLockedError) {
    return res.status(503).json({
      error: err.message,
      code: 'data_dir_moving',
      retryable: true,
    });
  }
  next(err);
});
```

`DataDirLockedError` は `src/server/services/dataDirLock.ts` から export 済み前提。generationService の `GenerateError('data_dir_moving', ...)` への remap は**削除して**この middleware に一本化する(重複を避ける)。

**受け入れ条件**:

- 移動中に PUT /api/projects/:id/state / PUT /api/projects/:id/memories / POST /api/projects/:id/generation/:id/accept 等を叩くと、いずれも 503 で `{ error, code:'data_dir_moving', retryable: true }` が返る。
- クライアント側([src/client/clientApi.ts:378](src/client/clientApi.ts:378) 付近)の retryable ハンドリング(「少し待って再試行できます」)が発火する。
- ユニットテスト追加: 移動中の各ルートが 503 を返すことを確認する統合テスト1本(mock で `withDataDirLock` を握った状態を作る)。

### F8. AppSettingsPanel の state 二重管理(message + error + applying)で矛盾表示

**場所**: [src/client/components/AppSettingsPanel.tsx:74-81](src/client/components/AppSettingsPanel.tsx:74) の handleApply、17-18 行の state 定義。

**問題**:

- 失敗時: catch で setError だけ実行し message が残る → 「コピーと検証を実行しています。アプリを終了しないでください。」と「移動に失敗しました: …」が同時表示。
- 成功時: setApplying(false) が呼ばれない → 「移動が完了しました」テキストの隣で「移動中…」ラベルのスピナーが回り続ける(再起動が走るまで数百 ms 〜数秒)。

**修正**: 3つの state を単一の `status: { kind: 'idle' | 'previewing' | 'applying' | 'success' | 'error', text: string }` に集約する。**遷移箇所は必ず setStatus 1回で完結**するのでどちらかが取り残されない。

```tsx
type Status =
  | { kind: 'idle' }
  | { kind: 'previewing' }
  | { kind: 'applying'; text: string }
  | { kind: 'success'; text: string }
  | { kind: 'error'; text: string };

const [status, setStatus] = useState<Status>({ kind: 'idle' });

// handleApply 内:
setStatus({ kind: 'applying', text: 'コピーと検証を実行しています。アプリを終了しないでください。' });
try {
  await api.applyDataDirMove(...);
  setStatus({ kind: 'success', text: '移動が完了しました。アプリを再起動しています。' });
} catch (err) {
  setStatus({ kind: 'error', text: `移動に失敗しました: ${...}` });
}

// レンダリング: status.kind === 'applying' でスピナー、それ以外は静的テキスト
```

`applying` フラグの参照(ボタン disabled 等)は `status.kind === 'applying' || status.kind === 'previewing'` に置換。

**受け入れ条件**:

- 失敗時に「実行しています」と「失敗しました」が同時に見えない。
- 成功時にスピナーが即停止(または F9 の「再起動待ち」表示に切り替わる)。

### F9. クライアントが再起動を観測していない → 固まる可能性

**場所**: [src/client/components/AppSettingsPanel.tsx:76](src/client/components/AppSettingsPanel.tsx:76) の success メッセージ以降。

**問題**: apply 成功後 message を出すだけで、再起動が実際に走ったかを観測しない。F3 の非 Electron 経路や、Electron でも `app.relaunch()` が失敗するケース(署名エラー・アップデート競合)で、UI が永久にスピナーで固まる。

**修正**: success 遷移後にヘルスチェックのポーリングと timeout を持たせる。

```tsx
// success 遷移後
const timeoutId = setTimeout(() => {
  setStatus({
    kind: 'error',
    text: '自動再起動が完了しませんでした。アプリを一度閉じて開き直してください。',
  });
}, 15000);

// GET /api/system/version を1秒間隔でポーリング → 到達不能になったら再起動確認
const interval = setInterval(async () => {
  try {
    await fetch('/api/system/version', { cache: 'no-store' });
  } catch {
    clearInterval(interval);
    clearTimeout(timeoutId);
    // サーバーが落ちた = 再起動が始まった。ページリロードを試みる
    window.location.reload();
  }
}, 1000);
```

`handleShutdown` が既に類似のフォールバックパターンを持っているので参考にする(「サーバーを停止しました。このウィンドウを閉じてください。」)。

**受け入れ条件**:

- 非 Electron 経路(F3 の案A で無効化されるが、フェールセーフとして): apply → 15 秒後にエラー案内が出る。
- Electron 経路: apply → サーバー close 検知 → ページリロードで新プロセスに接続。

### F10. データディレクトリロックが safeWrite プリミティブに埋まっていない(altitude)

**場所**: [src/server/utils/safeWrite.ts](src/server/utils/safeWrite.ts) の `safeWriteFile` / `safeWriteJson` / `ensureDir`(未変更)。ロックは `storageService` / `credentialService` / `lanAuthService` / `shortcutService` それぞれで `withDataDirWrite` を呼んで実装している。

**問題**: 現状は全ての書き込み経路が上記4サービスを通るため実害なし。しかし将来 (a) 新サービスが safeWrite プリミティブを直接使う、(b) 既存サービスに raw `fs.writeFile` / `fs.rm` が追加される、といった変更で**静かに移行を跨ぐ書き込み**が発生する。設計原則が「caller discipline」に依存している脆さ。

**修正**: `withDataDirWrite` を `safeWriteFile` / `ensureDir` / `safeWriteJson` の内部に埋め込む。

```ts
// src/server/utils/safeWrite.ts
import { withDataDirWrite } from '../services/dataDirLock.js';

export async function safeWriteFile(filePath: string, content: string): Promise<void> {
  return withDataDirWrite(async () => {
    // 既存の atomic rename ロジック
  });
}
// ensureDir / safeWriteJson も同様
```

サービス層の `withDataDirWrite` 呼び出しは基本的に**削除して二重ロックを避ける**(dataDirLock の再入エスケープが働くので二重でも動くが、意図を明確にするため一箇所に寄せる)。

**注意**: この修正は移動処理 `applyDataDirMove` 自身への影響に注意。`applyDataDirMove` はロック取得中に safeWriteJson を呼ぶ(pendingCleanup の書き込み)ため、再入エスケープに乗る必要がある。既存の再入ロジックが機能することを確認する。

**受け入れ条件**:

- 既存のロックテスト全通過(`tests/unit/dataDirLock.test.ts` / `generationServiceLock.test.ts`)。
- ユニットテスト追加: safeWriteFile を直接呼んだ場合でも、withDataDirLock 保持中は待たされる。
- 各サービスで冗長になった `withDataDirWrite` ラップを削除。
